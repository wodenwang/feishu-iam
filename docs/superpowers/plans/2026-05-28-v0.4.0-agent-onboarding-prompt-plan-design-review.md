# feishu-iam v0.4.0 Agent 接入初始化提示词设计评审

日期：2026-05-28

## 结论

通过，按 `v0.4.0` 新能力版本执行。该版本的主交付是提升第三方内部项目的 Agent 接入效率，让 Codex / Claude Code 可以用一段初始化提示词创建或维护第三方项目的 `AGENTS.md`、`CLAUDE.md`，并完成 IAM 接入 SOP。

## 设计评分

| 维度 | 评分 | 判断 |
|---|---:|---|
| 用户目标清晰度 | 9/10 | 目标用户明确：第三方项目 Agent 和应用管理员。 |
| 信息架构 | 8/10 | 复用应用详情页接入配置区，避免新增页面；需要保证普通 prompt 和 one-time prompt 区分清晰。 |
| 主路径效率 | 9/10 | 创建/轮换后直接复制完整 prompt，最短路径清晰。 |
| 安全边界表达 | 8/10 | 允许一次性明文 secret 是有意识的效率取舍；必须通过 warning 和审计不泄密约束降低风险。 |
| 状态覆盖 | 8/10 | 覆盖普通、创建一次性、轮换 appSecret、轮换 apiSecret、复制失败；后续实现需覆盖空 redirect URI。 |
| Ant Design 适配 | 9/10 | 使用现有 Card、Alert、Modal、Typography copyable、Button，不需要自定义复杂 UI。 |

## 关键设计决策

- D1：不新增独立“接入工作台”，避免把范围扩大到文档站或 SDK 平台。
- D2：普通应用详情页只生成占位符 prompt，避免暗示历史 secret 可再次查看。
- D3：创建/轮换后的一次性 prompt 可以包含明文 secret，服务内部项目效率。
- D4：一次性 prompt 必须在 UI 上和普通 prompt 明确区分，并用 warning 说明不要提交到 Git。
- D5：Pencil 原型本版跳过，原因是 UI 只是在既有详情页和 Modal 内增加复制动作，不改变页面结构、导航和表格模式。

## 需要实现时注意的体验点

- 普通 prompt 的标题应明确为“占位符版”，避免用户误以为包含 secret。
- 一次性 prompt 的按钮应靠近一次性 secret 结果，降低用户错过的概率。
- 创建应用结果中的完整 prompt 应同时包含 `IAM_APP_SECRET` 和 `IAM_API_SECRET`。
- 单独轮换时只填入当次新 secret，另一个 secret 使用占位符，避免伪造系统不知道的明文。
- prompt 内容应是给 Agent 的可执行任务，不要变成普通 README 摘要。

## 进入工程评审的调整

- 在工程评审中明确是否需要后端生成 prompt。结论倾向前端生成，因为所有明文 secret 已在前端一次性结果中存在，后端无需保存或重新拼装。
- 审计 kind 需要扩展，便于区分普通 prompt 与 one-time prompt。
- 测试必须直接断言 audit metadata 不含 secret 明文。
