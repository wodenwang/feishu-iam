# v0.1.16 同步运营 Runtime 前端切片设计评审

## 结论

通过。当前切片是在既有 `/sync` 飞书同步中心上接入后端 Runtime 状态、权限预检和 scheduled/system actor 可见性，不改变信息架构，不新增页面，不扩展 `/directory`，因此不需要重新制作 Pencil 原型。

## 依据

- 既有 Pencil 原型已覆盖 `/sync` 的状态卡、同步记录表、失败详情、权限不足和运行中状态。
- v0.1.15 已完成 `/sync` Runtime 接入与设计审查，确认该页可承载同步状态、同步记录和失败详情。
- 本切片只新增运维诊断控件和状态字段，符合 Ant Design `Card` / `Statistic` / `Table` / `Drawer` / `Alert` 模式。

## 评分

| 维度 | 评分 | 说明 |
|---|---:|---|
| 信息架构 | 9/10 | 保持单页同步中心，不新增导航层级；状态、操作、记录仍在同一工作流中。 |
| 操作效率 | 8/10 | 预检按钮放在同步任务工具栏，和手动同步、刷新并列；失败详情继续用 Drawer。 |
| 状态表达 | 8/10 | 使用健康状态 Tag、目录投影数量、最近成功/失败时间和预检阶段列表，覆盖日常排障入口。 |
| 权限边界 | 9/10 | `sync:view` 可读，`sync:run` 才能手动同步、重试或预检；不向应用管理员开放全局同步运营能力。 |
| 领域边界 | 10/10 | 不伪造飞书用户；scheduled run 以 `operatorType=system` 展示为系统任务，不写入虚假的 `operatorFeishuUserId`。 |
| 可实现性 | 9/10 | 复用现有 `iamService`、TanStack Query hooks、mock store 和页面测试，不引入新 UI 框架。 |

## 页面方案

1. `同步状态摘要`
   - 展示后端 `GET /api/sync/status` 的 `healthStatus`、`healthReasons`、`isRunning`。
   - 展示最近成功同步时间、最近失败同步时间、目录用户数量和部门数量。
   - 当状态接口失败时显示可重试 `Alert`，不伪造健康结果。

2. `同步任务`
   - 保持同步历史表、详情 Drawer、手动同步、重试和刷新。
   - 新增 `运行预检` 按钮，调用 `POST /api/sync/preflight`。
   - `sync:view` 但缺少 `sync:run` 时，手动同步、重试和预检均禁用并展示原因。
   - `operatorType=system` 或空 `operatorFeishuUserId` 的记录显示为 `系统任务`。

3. `预检结果 Drawer`
   - 展示整体结果、检查时间、请求批次数、Request ID、错误码和消息。
   - 阶段列表展示 token、departments、users 的 passed/failed 状态。
   - 预检不创建 sync run，不修改目录投影，前端文案避免暗示已经同步。

## 不做

- 不新增 `/directory` 展示、编辑或导出能力。
- 不新增前端路由、导航或页面级重设计。
- 不新增 release metadata。
- 不展示或保存飞书凭证、tenant token、原始通讯录 payload、导出用户列表或同步快照。

## 设计门禁

Pencil 不重做。该切片属于既有 Admin Console 模式内的状态和操作增补，使用 Ant Design 标准组件即可完成；实现后需通过页面单测、构建和浏览器验证再进入后续 QA/review。
