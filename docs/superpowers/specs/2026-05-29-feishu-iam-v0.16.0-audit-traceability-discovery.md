# Feishu IAM v0.16.0 生产追踪与接入排障 Discovery

状态：DRAFT。已从 `office-hours` 的单一“审计排障”方向调整为 `superpowers:brainstorming` 后确认的 C 方案：生产追踪与接入排障交集版。已完成非交互 `plan-design-review` 初审、Pencil 原型、原型设计复审和非交互 `plan-eng-review`。当前结论是可以进入 `Superpowers writing-plans`，工程约束以 `docs/superpowers/reviews/2026-05-29-feishu-iam-v0.16.0-audit-traceability-eng-review.md` 为准。

## 1. 背景

`v0.15.2` 已发布并完成 `192.168.2.112:~/feishu-iam` 停机升级验证。最近三个版本分别收口组织树、权限可解释性和组织用户选择体验。

当前 GitLab opened issues：

| Issue | 标题 | 标签 | 与 `v0.16.0` 的关系 |
|---|---|---|---|
| `#27` | 未登录提示页需要按 shadcn 风格重做为独立单页 | 优化 | 建议纳入，并作为统一问题提示页的第一屏样板。`v0.16.0` 必须先兑现原 issue 的未登录页验收，再把同一信息架构扩展到登录失败、无权限、会话过期和接入错误场景。 |

历史规划曾把“审计、排障、接入可追溯”顺延到后续版本。上一轮 `office-hours` 先把它收敛为“生产可追溯性与审计排障增强”，但该边界仍偏窄：它强调后台审计页，却没有充分覆盖第三方系统接入时最常见的登录、换 token、userinfo、权限查询、client 配置和回调地址问题。

因此 `v0.16.0` 推荐定义为：生产追踪与接入排障版本。它不是单纯审计页，也不是单纯开发者文档，而是围绕真实问题闭环：第三方系统登录、换 token、userinfo 或权限查询异常时，管理员能从 `request id`、应用、client、用户和时间窗口查到原因与下一步。

## 2. 候选方案对比

| 方案 | 主题 | 是否适合作为 `v0.16.0` | 结论 |
|---|---|---:|---|
| A | 生产追踪与审计排障 | 高 | 有价值，但如果只做审计页，接入异常主路径仍不够完整。 |
| B | 第三方接入开发者体验 | 高 | 有价值，但如果只做文档/接入包，生产问题仍查不清。 |
| C | 生产追踪与接入排障交集版 | 最高 | 推荐。用生产追踪能力直接服务第三方接入排障。 |
| D | OpenAPI 与机器可读接入契约 | 中 | 适合作为后续或 C 的子能力，不适合单独压过排障主线。 |
| E | OAuth / Token 生命周期增强 | 中低 | 容易扩大到 refresh token 或完整 OIDC，本版本暂不碰。 |
| F | 平台底层能力升级 | 低 | 平台 token scope、缓存、性能优化需要明确生产痛点后再做。 |

## 3. 目标用户

第一目标用户：

- 平台管理员：需要在真实生产问题发生时判断是配置变更、OAuth 流程、client、用户状态、权限授权还是飞书同步导致异常。
- 应用管理员：需要围绕自己负责的应用，解释某个用户为什么登录失败、权限不对、菜单按钮不可见或接口被拒绝。
- 审计查看员：需要只读追溯高风险变更、登录失败、安全事件和同步记录，不执行写操作。
- 终端用户：遇到未登录、登录失败、无权限、接入异常或会话失效时，需要看懂当前状态，并快速复制 `request id` 和问题信息反馈给管理员或接入开发者。

第二目标用户：

- 同步管理员：需要定位飞书同步失败、字段缺失、用户不可登录和同步 run 与接入异常之间的关系。
- 第三方系统接入开发者：通常不直接进入 Feishu IAM 后台，但会把 `request id`、client、回调地址、用户和错误码交给管理员排障。
- 后续 Agent：需要从接入文档和后台排障结果里得到稳定、可复制的诊断路径。

## 4. 核心问题

当前系统已经具备审计日志、安全事件、飞书同步记录、登录与 Token 记录、真实第三方接入 demo 和权限解释能力。但这些能力分散在不同页面和文档里，还不能稳定回答接入生产问题。

`v0.16.0` 要回答四类问题：

1. 某个第三方应用登录、换 token、userinfo 或权限查询失败时，失败发生在哪一段。
2. 某个 `request id` 背后有哪些审计日志、安全事件、同步记录、OAuth 事件和应用配置上下文。
3. 某个应用、client、回调地址、角色或用户最近发生过哪些相关变更，是否影响接入。
4. 管理员下一步应该查什么：应用状态、client 状态、回调地址、用户状态、同步状态、权限组/权限点，还是第三方系统自身。

## 5. 现状判断

已有基础：

- `apps/api/src/admin/admin-query.service.ts` 已提供审计日志和安全事件查询，并对敏感字段做递归脱敏。
- `apps/admin-web/src/features/records/RecordQueryView.tsx` 已有 `审计日志 / 安全事件 / 同步记录 / 登录与 Token 记录` 四个 Tab。
- Prisma 已有 `audit_logs`、`security_events`、`feishu_sync_runs`、`oauth_login_states`、`oauth_access_tokens` 等表。
- `v0.12.0` 已通过 `feishu-iam-sso-demo` 验证真实第三方系统可完成授权码、token、userinfo 和权限查询闭环。
- `v0.13.0` 已把飞书同步页升级为排障优先的运维控制台，包含同步历史详情、失败阶段和 request id。
- `v0.15.1` 已补角色权限可解释性，角色详情可查看最终有效权限点。

主要缺口：

- 缺少围绕 `request id`、应用、client 和用户聚合的接入排障时间线。
- 操作审计和登录 / Token 记录之间缺少“这次接入失败为什么发生”的诊断叙事。
- 应用详情、角色详情、飞书同步 run、OAuth 错误和安全事件之间缺少稳定跳转关系。
- 审计详情中的 `before/after` 虽已脱敏，但不够面向管理员解释变更影响。
- 接入文档、统一问题提示页和后台排障入口没有形成闭环：终端用户或开发者看到错误后，不知道应该把哪些字段交给管理员，管理员也没有一个统一入口消费这些字段。

## 6. 推荐版本边界

推荐 `v0.16.0` 最小切片：接入追踪工作台。

范围锁定在 `系统管理 / 操作审计`、统一问题提示页、应用详情、角色详情、飞书同步 run 详情和接入文档索引。不新增权限模型，不改变 SSO 协议，不做大范围 DDL，不把它做成 BI 报表。

版本切片分两层：

- P0 必须交付：`#27` 未登录提示页独立单页、统一问题提示页组件规范、`request id` 独立复制、一键复制问题信息、操作审计里的追踪查询入口、接入文档字段收集指南。
- P1 可随实现评估交付：应用详情和角色详情上下文跳转、审计 `before/after` 字段级解释、OAuth 接入排障摘要面板。如果工程评审发现事件数据不足，P1 可以缩小为“只展示已能可靠关联的数据和下一步建议”。

### 6.0 opened issue `#27`：统一问题提示页

- 将 `#27` 纳入 `v0.16.0` 的前置体验切片，而不是单独开 `v0.15.3`。
- 第一验收对象仍是原 issue 描述的“未登录提示页”：它必须是独立单页、符合 shadcn/ui 风格、突出飞书登录、弱化技术信息、覆盖桌面和窄屏。
- 统一问题提示页是该 issue 的设计抽象，不反向扩大 `#27` 的必做验收。除未登录页外，登录失败、无后台权限、会话过期和 OAuth 接入失败可以复用同一组件与文案合同；具体落地范围由后续 engineering review 按现有路由和错误处理能力锁定。
- 主操作随场景变化：未登录场景突出“飞书登录”；无权限场景突出“返回工作台 / 退出登录”；接入失败场景突出“返回应用 / 重试 / 联系管理员”。
- 不改变管理员 session 校验机制、飞书登录 OAuth 后端流程、权限判定或破窗登录边界。

统一布局：

- 页面采用独立单页结构，背景使用后台系统一致的浅色底色，不做营销页 hero。
- 主体使用单张 shadcn/ui 风格 `Card` 承载，不嵌套多层卡片；桌面宽度控制在可读范围，390px 窄屏保持完整可用。
- 顶部为 Feishu IAM 品牌区：产品名、简短系统定位和状态图标。
- 中部为问题摘要区：清晰标题、终端用户可理解的说明、当前状态标签和主操作按钮。
- 下部为排障信息区：错误码、`request id`、发生时间、当前路径、应用或 client 上下文。技术信息使用低视觉权重，但必须可见、可复制。
- 底部为反馈引导区：提示终端用户把复制的信息发给内部管理员或接入开发者，不要求用户理解 OAuth 或权限模型。

复制能力：

- `request id` 必须有独立复制按钮，按钮文案明确为“复制 request id”。
- 问题提示页必须提供“一键复制问题信息”，复制内容使用稳定多行文本，包含：
  - `request id`
  - 错误码
  - 页面路径或回调路径
  - 应用 `app_key` 或 client id，如果当前上下文可得
  - 用户标识，如果已知且允许展示
  - 发生时间
  - 简短问题说明
- 复制成功后使用轻量 toast 或内联状态反馈；复制失败时展示可手动选择的文本块。
- 复制内容不得包含 access token、client secret、developer api token、cookie、authorization header、堆栈或原始敏感 payload。

反馈引导：

- 终端用户文案应说明“把 request id 和问题信息发给管理员/技术支持”，避免只展示框架错误。
- 如果是第三方接入链路错误，应提示用户同时提供第三方系统名称、当前操作、发生时间和截图。
- 如果是管理员后台未登录，应弱化 `ADMIN_SESSION_REQUIRED`，主路径仍是飞书登录。
- 如果是权限不足，应说明当前账号没有对应后台权限，并引导联系平台管理员或应用管理员。
- 如果是 OAuth 回调、token、userinfo 或权限查询错误，应提示接入开发者携带复制的问题信息到 Feishu IAM 后台追踪视角排查。

视觉和响应式要求：

- shadcn/ui 组件优先使用 `Card`、`Alert`、`Button`、`Badge`、`Separator`、`Tooltip` 和复制按钮。
- `request id`、URL、client id、错误码等长文本必须支持截断、换行或复制，不得撑破布局。
- 主按钮尺寸稳定，不因“飞书登录”等中文文案换行变形。
- 桌面和 390px 窄屏下无异常空白、遮挡、横向溢出或内容挤压。
- 页面应与 Feishu IAM 管理后台视觉体系一致，克制、专业、信息密度适中。

页面变体合同：

| 场景 | 主标题 | 主操作 | 辅助操作 | 必显排障字段 |
|---|---|---|---|---|
| 管理后台未登录 | 需要登录 Feishu IAM 管理后台 | 飞书登录 | 复制 request id | 错误码、request id、发生时间、当前路径 |
| 会话过期 | 登录状态已失效 | 重新登录 | 返回登录入口、复制问题信息 | 错误码、request id、发生时间、当前路径 |
| 无后台权限 | 当前账号没有访问权限 | 返回工作台或退出登录 | 复制问题信息 | 错误码、request id、当前账号标识、发生时间 |
| 登录失败 | 登录没有完成 | 重新登录 | 复制问题信息 | 错误码、request id、发生时间、失败阶段 |
| OAuth 接入失败 | 第三方应用登录异常 | 返回应用或重试 | 复制问题信息 | 错误码、request id、app_key 或 client id、回调路径、发生时间 |

桌面布局合同：

1. 首屏中心区域只出现一张主 `Card`，最大宽度建议控制在 `560px` 到 `640px`，上下留白稳定。
2. Card 内部顺序固定为品牌区、问题摘要、主操作、排障信息、反馈引导。不要把技术字段放在标题说明第一行。
3. `request id` 行必须有字段名、可读值、复制按钮和复制成功状态；长值可以中间截断，但复制内容必须完整。
4. “一键复制问题信息”是次级按钮或轻量 outline 按钮，不与主操作抢视觉层级。

390px 窄屏合同：

1. Card 宽度跟随 viewport，左右安全边距不小于 `16px`。
2. 主按钮和复制按钮不得因中文文案换行导致高度异常；必要时主操作独占一行，复制操作下沉为次级整行按钮。
3. `request id`、URL、client id 等长文本必须使用可换行文本块或截断加复制，不允许横向滚动撑破页面。
4. 排障信息超过四行时允许折叠，但 `request id`、错误码和发生时间必须默认可见。

可访问性合同：

- 所有复制按钮必须有明确 `aria-label`，例如“复制 request id”。
- 复制成功反馈必须通过 toast 或 `aria-live` 文案传达，不只依赖颜色变化。
- 键盘 Tab 顺序为主操作、复制 request id、一键复制问题信息、辅助操作。
- 主操作和复制按钮触控目标不小于 `44px` 高度。

一键复制问题信息模板：

```text
Feishu IAM 问题反馈
问题：<用户可理解的问题摘要>
错误码：<ERROR_CODE>
request id：<REQUEST_ID>
发生时间：<ISO 或本地时间>
页面路径：<PATH_OR_CALLBACK_URL>
应用：<APP_KEY_OR_EMPTY>
client：<CLIENT_ID_OR_EMPTY>
用户：<USER_IDENTIFIER_OR_EMPTY>
下一步：请把以上信息发送给 Feishu IAM 管理员或接入开发者排查。
```

模板中的空字段不展示占位符；复制内容不得包含 access token、client secret、developer api token、cookie、authorization header、堆栈或原始敏感 payload。

该切片与接入追踪主线的关系：统一终端可见问题页、错误码和 `request id` 展示方式，让终端用户能高效反馈问题，也让管理员后续能用同一个 `request id` 进入追踪视角。

### 6.1 追踪查询入口

- 在 `系统管理 / 操作审计` 增加“追踪”视角，支持按 `request id` 查询。
- 支持辅助筛选：`applicationId` / `app_key`、`clientId`、`feishuUserId`、时间窗口、结果状态。
- 查询结果按时间线展示审计日志、安全事件、同步记录、OAuth 登录 / Token / userinfo / 权限事件。
- 无完整关联时，也要展示“查到了什么 / 没查到什么 / 下一步建议查什么”。
- 保留原四类 Tab，不用追踪视角替代原始查询。

追踪视角推荐放在 `操作审计` 页面内，作为原四类 Tab 前的独立“追踪”Tab。理由是用户带着 `request id` 进入时不应先选择事件类型；查不到完整链路后，再引导进入审计日志、安全事件、同步记录或登录与 Token 记录原始 Tab。

追踪视角状态矩阵：

| 状态 | 页面要表达什么 | 主操作 |
|---|---|---|
| 初始态 | 输入 `request id`、应用、client、用户或时间窗口开始排查 | 查询 |
| 加载中 | 保持筛选条件可见，时间线区域显示骨架屏 | 无 |
| 完整命中 | 时间线按阶段展示事件，并给出诊断摘要 | 查看事件详情 |
| 部分命中 | 明确说明已查到哪些事件、缺哪些事件，避免误判“系统没问题” | 继续查看原始记录 |
| 无结果 | 说明可能原因：时间窗口不对、request id 错误、事件未记录或权限不足 | 调整筛选 |
| 权限不足 | 说明当前角色能查的范围，不泄露不可见资源存在性 | 联系平台管理员 |
| 脱敏字段 | 展示字段已脱敏的原因和可继续排查的字段 | 复制可见信息 |

时间线事件卡片最少包含：事件类型、阶段、结果、时间、资源、request id 和“查看详情”。不同事件类型使用 `Badge` 和图标区分，但不要使用复杂图表。

### 6.2 接入排障诊断摘要

- 对 OAuth 登录、换 token、userinfo、权限查询失败展示稳定诊断摘要。
- 摘要至少包含：阶段、错误码、应用、client、用户标识、request id、时间、可能原因、下一步。
- 常见原因优先覆盖：应用停用、client 停用、回调地址不匹配、授权码失效、access token 过期或撤销、用户不可登录、权限为空或角色授权缺失。
- 不展示明文 access token、client secret、developer api token、cookie 或 authorization header。

### 6.3 资源上下文跳转

- 应用详情提供“查看接入追踪”入口，默认带上应用上下文。
- 应用详情的 client / 回调地址 / 开发信息区域可以跳到相关审计或追踪视角。
- 角色详情提供“查看授权相关审计”入口，默认带上角色或应用上下文。
- 飞书同步 run 详情、安全事件详情、审计详情都可以进入追踪视角。
- URL 保留查询条件，返回时保留列表、Tab 和详情上下文。

### 6.4 审计差异解释

- 审计详情对 `before/after` 做字段级差异摘要，突出旧值、新值和影响范围。
- 对应用状态、client 状态、回调地址、角色主体绑定、权限组绑定、权限点绑定等高价值变更给出中文解释。
- 长 JSON 保留复制、折叠和脱敏展示，不能撑破页面或抽屉。
- 脱敏规则复用现有后端脱敏策略，前端不得自行展示敏感原文。

### 6.5 接入文档闭环

- README 或接入验收文档补一段“出现登录 / token / userinfo / 权限查询问题时，应该收集哪些字段”。
- 字段至少包括：`request id`、应用 `app_key`、client id、回调地址、用户标识、错误码、发生时间。
- 文档应明确区分终端用户反馈字段和接入开发者排障字段：终端用户只需复制问题信息；接入开发者再补充 client、回调地址、接口路径和请求时间。
- 文档链接回后台追踪入口，不把排障说明散落在多个页面。

### 6.6 Pencil 原型最低画板

进入工程计划前必须先补 Pencil 原型，至少包含以下画板：

1. 统一问题提示页：桌面未登录态、390px 未登录态、OAuth 接入失败态。
2. 操作审计追踪 Tab：初始态、完整命中、部分命中、无结果或权限不足。
3. 追踪详情：诊断摘要、时间线、事件详情抽屉或详情面板。
4. 应用详情来源跳转：从应用详情进入追踪视角，并能返回原应用详情 Tab。
5. 长文本和脱敏样例：长 `request id`、长回调 URL、脱敏字段、复制成功反馈。

Pencil 原型不追求像素级实现，但必须把层级、状态、响应式和复制路径画清楚。没有这些画板，不应进入 `plan-eng-review`。

## 9. 成功标准

`v0.16.0` 成功不看新增页面数量，而看真实接入问题能否被定位：

- 拿到一个第三方系统报错的 `request id` 后，管理员能在后台看到相关审计、安全事件、同步或 OAuth 事件。
- 未登录访问管理后台时，`#27` 要求的独立登录提示页可用，飞书登录入口突出，错误码和 `request id` 保持可见但低权重。
- 统一问题提示页能让终端用户一键复制 `request id` 和结构化问题信息，并明确知道应反馈给管理员或接入开发者。
- 从某个应用详情能进入该应用相关的接入追踪，并返回原详情上下文。
- 登录、换 token、userinfo 或权限查询失败能看到阶段、错误码、应用、client、用户和下一步建议。
- 审计详情能解释关键字段变化，不暴露敏感值。
- 接入文档明确告诉开发者遇到问题时收集哪些字段给管理员。
- `platform_admin`、`audit_viewer`、`application_admin`、`sync_admin` 权限边界不回退。
- 桌面和 390px 视口下，筛选区、时间线、详情抽屉、长 JSON 和诊断摘要无横向溢出。

## 11. 后续 `plan-design-review` 需要挑战的问题

- “追踪”视角应该是操作审计里的 Tab、顶部模式切换，还是独立入口。
- 统一问题提示页是否只覆盖管理后台，还是同时覆盖 OAuth 错误页和第三方回调失败页；如果覆盖多场景，哪些字段必须一致。
- `request id` 和“一键复制问题信息”在视觉层级上如何做到既容易找到，又不压过主操作。
- 时间线如何表达不同事件类型，避免把审计、安全、同步、OAuth 事件混成不可读列表。
- 接入诊断摘要应该放在追踪详情、事件详情，还是单独的诊断面板。
- 无结果、部分结果、权限不足、敏感字段脱敏、长 JSON、移动端窄屏的状态如何设计。
- 从应用详情、角色详情、同步 run 详情跳转到追踪页后，如何保留来源上下文和返回路径。
- 审计查看员、应用管理员、同步管理员看到的信息是否应有不同空态和解释文案。
- 接入文档中的字段收集指南如何与后台入口互相链接，避免文档和产品两套口径。
- Pencil 原型是否证明了 390px 下主操作、复制 request id、复制问题信息和长文本同时存在时仍可用。

## 12. 后续 `plan-eng-review` 需要挑战的问题

- 现有 `audit_logs`、`security_events`、`feishu_sync_runs`、`oauth_login_states` 和 `oauth_access_tokens` 是否足以支撑接入追踪。
- 是否需要新增只读聚合 endpoint，还是前端调用现有列表接口后组合；性能和权限边界如何控制。
- `request_id` 在 OAuth authorize、token、userinfo、权限查询、admin、feishu 路径是否稳定传递；哪些路径还缺失。
- 安全事件是否已覆盖 userinfo 和权限查询成功/失败；如果没有，本版本是否只做失败事件。
- `before/after` 差异摘要在后端生成还是前端生成；脱敏规则如何复用 `AdminQueryService`，避免泄露 secret。
- 大数据量下按 `request id`、`applicationId`、`clientId`、`feishuUserId`、`resourceType/resourceId` 查询是否已有索引支撑。
- 应用管理员、审计查看员、同步管理员的查询边界是否与现有 `AdminPermissionService` 一致。
- 测试策略如何覆盖权限边界、敏感字段脱敏、深链 URL、响应式、生产未登录态和真实 demo 接入错误样例。

## 13. `plan-eng-review` 锁定结论

工程评审结论：可以进入 `Superpowers writing-plans`，但实施计划必须按以下约束收口：

- 必须新增后端只读追踪聚合接口，不让前端组合旧列表接口后自行裁剪权限和脱敏。
- `request_id` 不需要强行追加到所有 OAuth 持久表；追踪事实以 `audit_logs`、`security_events` 和 `feishu_sync_runs` 为主，`oauth_login_states`、`oauth_access_tokens` 只作为上下文补充。
- 必须补齐 token exchange、userinfo、权限查询路径的最小安全事件元数据，重点覆盖失败和关键成功点，不做每次权限查询全量审计。
- 必须新增 `security_events.request_id`、`security_events.application_id + created_at`、`security_events.client_id + created_at`、`security_events.feishu_user_id + created_at` 和 `feishu_sync_runs.request_id` 小型索引迁移。
- 权限边界必须在后端执行：`platform_admin` / `audit_viewer` 可全局只读，`application_admin` 只能看授权应用，`sync_admin` 只能看同步相关上下文。
- 脱敏必须后端统一执行，不把 secret、token、cookie、authorization、client secret、tokenHash、stateHash、raw_payload 暴露给浏览器。
- 统一问题提示页先覆盖未登录、会话过期、无权限和 OAuth 接入失败；不改变管理员 session、OAuth 协议或破窗登录边界。
