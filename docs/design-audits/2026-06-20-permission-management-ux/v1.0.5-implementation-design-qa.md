# Feishu IAM v1.0.5 权限管理实现设计 QA

## 结论

`v1.0.5 - 权限管理角色配置工作台` 当前实现通过 step10 设计还原 QA，可以停止在 step10，不进入 step11。

本次 QA 重点对照：

- `DESIGN.md` 的 shadcn/ui + tweakcn 后台系统基线。
- `design/v1.0.5-permission-management-sop.md`。
- `design/prototypes/v1.0.5-permission-management/README.md`。
- Playwright 实机截图与浏览器报告：`output/playwright/v1.0.5-permission-workbench/`。

总体判断：

- 权限管理首屏已回到角色列表。
- 角色配置工作台已是独立页面，不是抽屉。
- `组织与用户` 与 `应用权限` 已拆成独立工作区，避免单页过重。
- 组织和用户通过同一个 `OrgUserSelector` 选择器勾选。
- 应用权限工作区保留应用切换、权限组绑定、权限点预览和权限点对比。
- 应用详情中的角色管理 Tab / 入口 / 操作已移除，并替换为权限资产查看和比对。
- PC 端后台信息密度、表格优先、图标操作、筛选区和 Tabs 结构符合当前 `DESIGN.md` 方向。

## shadcn / DESIGN.md 对齐

| 检查项 | 结果 |
|---|---|
| 后台系统定位 | 页面以筛选、表格、Tabs、详情工作区为主，没有营销页或 hero 化表达 |
| 信息密度 | 列表和工作台保持中等密度，未出现大面积低密度卡片堆叠 |
| 操作列 | 角色配置、编辑、启停使用稳定尺寸 icon button group |
| 复杂配置承载 | 权限配置没有使用窄抽屉，符合 `DESIGN.md` 对高认知负载流程优先独立页的要求 |
| 技术标识符 | 角色 key、app key、权限点 key 等均有中文字段上下文或表格列名 |
| 应用详情边界 | 应用详情聚焦应用自身配置和权限资产，不再承载角色管理 |

## 接受的偏差

| 偏差 | 原因 | 影响 |
|---|---|---|
| 原型中 `角色基础信息编辑` 可独立页面承载；当前实现主要通过列表和工作台内对话框/表单承载新增、编辑和状态操作 | 本版本核心是重构授权流程与角色独立化；现有交互已覆盖角色基础信息维护，不阻塞权限管理主旅程 | 可接受，后续如要做角色元数据治理增强，可再单独抽成详情页 |
| 原型提到删除角色；当前实现不强化常规硬删除 | 项目安全约定倾向软删除或禁用，权限资源删除需要更强审计和影响确认 | 可接受，当前通过启用/停用覆盖常规运营场景 |
| 工作台包含 `总览 / 基础信息` 等辅助 Tab | 用于承载角色上下文和低风险元数据，不改变 `组织与用户`、`应用权限` 分离原则 | 可接受 |
| 权限点对比依赖权限组返回权限点明细 | 当前前后端契约和测试数据覆盖该路径；若生产权限组明细量很大，后续可服务端分页或专用 compare API | 可接受，已在工程评审中列为性能关注点 |

## 风险

- 生产数据量明显增大后，权限点对比表可能需要后端分页、缓存或专用 compare endpoint。
- 应用管理员不再能在应用详情里直接维护角色，需要通过权限管理入口完成角色授权；当前版本已用权限边界和页面 IA 做切换，但发布说明需要强调。
- 旧 URL 深链兼容依赖前端路由映射和角色查询回填，发布前 step11 仍需做更完整系统 QA。
