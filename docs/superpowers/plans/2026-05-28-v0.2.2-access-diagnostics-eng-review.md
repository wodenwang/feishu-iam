# feishu-iam v0.2.2 应用接入诊断与可观测性补强工程评审

日期：2026-05-28

## 结论

通过，按窄切片执行。`v0.2.2` 作为 `v0.2.0` 应用接入生产化和 `v0.2.1` 应用接入闭环补缺之后的诊断补强版本，不开启新的协议产品化、目录治理、同步增量或部署产品化范围。

本版本目标是让平台管理员、应用管理员和第三方系统开发者在接入失败时能快速判断问题归因：redirect URI、secret、权限注册、角色授权、登录态、目录用户投影或应用 scope。

## 版本边界

### In Scope

- 应用详情新增 `接入诊断` 入口，展示配置完整性、最近失败、可操作修复建议和关键 requestId。
- 后端新增脱敏诊断 API：`GET /api/applications/:id/diagnostics`。
- 后端新增脱敏诊断包复制审计：`POST /api/applications/:id/diagnostics/copy`。
- 前端支持复制 Markdown 诊断包，内容包含 appKey、端点、active redirect URI、权限注册摘要、角色授权摘要和最近失败 requestId。
- 新增或扩展诊断验收脚本，验证正常、warning、failure 和脱敏输出。
- README 与 `docs/integration/thirdparty-demo.md` 增加接入失败排查矩阵。
- 版本元数据更新为 `0.2.2` / `v0.2.2`，补充 QA、design review、pre-landing review 和 ship 证据目录。

### Out Of Scope

- OIDC discovery、JWKS、PKCE、refresh token、复杂 consent。
- SDK、CLI、Helm、Terraform 或安装升级产品化。
- 飞书 webhook、event sync、incremental sync 或同步告警。
- `/directory` 编辑、导入、导出或目录治理。
- 多租户 SaaS IdP。
- username/password、本地 root 或独立账号体系。
- 真实 secret、token、authorization code、HMAC signature、cookie 的任何展示、日志或复制。

## 工程决策

### D1：诊断 API 与应用详情同权

`GET /api/applications/:id/diagnostics` 使用现有 `requireAdminActor` 和 `requireApplicationScope`。平台管理员可看所有应用，应用管理员只可看自己 scope 内应用。诊断 API 不引入新的权限码。

### D2：诊断只返回脱敏状态与可复核线索

允许返回：

- `appKey`
- IAM base URL 和公开 endpoint path
- active / disabled redirect URI 计数与 URI 本身
- `appSecret` / `apiSecret` 最近轮换时间和 issued/missing 状态
- 权限组、权限点、角色、授权对象计数
- 最近 OAuth、Application API、permission query、secret/config 相关审计的 action、result、requestId、createdAt
- 明确的 warning / failed reason 和下一步建议

禁止返回：

- `appSecret`
- `apiSecret`
- bearer token
- authorization code
- HMAC signature
- cookie
- 飞书 token
- secret hash

### D3：健康分级保持简单

诊断状态使用三档：

- `healthy`：具备 active redirect URI、issued secret、至少 1 位应用管理员、存在权限注册，且最近失败为空。
- `warning`：配置可继续使用但存在风险，例如没有角色授权、存在 disabled redirect URI、权限组为空、最近有失败审计。
- `failed`：核心接入不可用，例如没有 active redirect URI、secret 状态缺失、应用被停用。

每条发现包含：

- `severity`: `info` | `warning` | `critical`
- `code`: 稳定机器可读 code
- `title`: 中文短标题
- `description`: 具体原因
- `nextAction`: 明确下一步
- `relatedRequestId`: 可选 requestId

### D4：复制诊断包只复制前端生成内容，后端只记审计

后端 `POST /api/applications/:id/diagnostics/copy` 只写 `application.diagnostics.copy` 审计，不接收或保存诊断包全文。

前端根据 `ApplicationDiagnostics` 生成 Markdown。生成逻辑放在 `src/pages/Applications/diagnostics.ts`，避免塞进 `Detail.tsx`。

### D5：不新建复杂可观测性系统

本版本不引入指标库、日志检索系统、trace 系统或后台任务。诊断数据来自已有 runtime 表和 `audit_logs`，只做产品化聚合。

## 风险与控制

| 风险 | 控制 |
|---|---|
| 诊断包意外泄露 secret | 后端不返回 secret；前端生成函数有单测断言敏感模式不存在；脚本扫描输出。 |
| 诊断范围膨胀成监控平台 | 只聚合已有表和审计，避免新增日志/trace/metrics 系统。 |
| 应用管理员越权看其他应用诊断 | 复用 `requireApplicationScope`，增加后端测试。 |
| 诊断建议误导真实飞书配置 | 文档明确外部飞书后台仍需人工配置 callback URL、权限和 IP 白名单。 |
| UI 变成低密度说明页 | 使用 Ant Design Tabs、Descriptions、Alert、Table、Timeline，不做营销式页面。 |
