# Feishu IAM v0.8.0 后台前端重构方向与首轮收口

日期：2026-05-21
状态：已确认采用方向 A，首轮前端实现和验证已完成

## 1. 版本目标

`v0.8.0` 的目标是对 Feishu IAM 后台管理系统前端进行完整重构，解决 `v0.7.0` 后台 UI/UX 效果不满意的问题。

一句话边界：

`v0.8.0 = 保留现有业务模型和业务流程，推倒重做后台前端表达与交互体验`。

本轮已完成原型确认，用户已选择 `方向 A：清单与详情抽屉型`。首轮实现已按该方向替换后台主要模块的前端表达；业务模型、业务流程和后端契约保持不变。

## 2. 已确认方向

用户确认采用 `方向 A：清单与详情抽屉型` 作为 `v0.8.0` 应用管理页重构方向。

该方向的实现基线：

- 应用管理首页采用传统后台清单结构，默认只展示应用清单。
- 顶部栏左侧承载品牌，中间区域留白，右上方是当前登录人的控制区。
- 当前登录人控制区当前只提供用户姓名、管理员类型和 `退出登录` 操作；未来如有账号偏好、安全设置等登录人相关能力，可继续收拢到右上方区域。
- 顶部栏不承载可点击的模块页签、当前模块标签或环境状态标签。
- 一级模块导航只保留在左侧菜单，避免顶部栏和左侧菜单功能重复。
- 主区域保留应用清单、搜索、筛选、分页和创建入口。
- 点击清单行内 `详情` 后，从右侧打开应用详情抽屉。
- 应用详情抽屉使用二级 tab：`基础信息`、`环境与回调`、`Client`、`操作记录`。
- 关闭应用详情抽屉后回到原清单，并保留列表筛选、分页、排序和必要的滚动位置。
- 从其他模块跳转或 URL 携带 `app_key` 等应用上下文时，可以默认打开对应应用的详情抽屉。
- 创建、编辑、停用、启用、回调管理、client 管理、secret 查看和轮换都走明确弹窗或抽屉流程。
- 高风险动作必须有二次确认、短时展示、审计日志和安全事件。

该方向暂不采用：

- `方向 B：应用档案型` 作为首页主结构。
- `方向 C：环境矩阵型` 作为首页主结构。

但 `方向 B` 和 `方向 C` 中可复用的局部表达可以在后续详细实施计划中作为子页或组件参考，例如在 `环境与回调` tab 中局部使用环境分组。

## 3. 固定业务边界

本轮原型必须保留以下既有业务模型和业务流程：

- `应用管理` 是应用概念总入口，吸收所有应用级 SSO 接入配置。
- 应用详情继续承载基础信息、环境、redirect URI、client、secret 查看与轮换、操作记录。
- `权限管理` 仍按 IAM 角色分配权限，不按人直接分配权限。
- 权限组和权限点仍由第三方开发者通过 API 管理，后台 UI 不做创建、编辑、删除。
- 应用管理员只能管理自己被授权的应用；平台管理员可管理全部应用。
- `client_secret` 查看和轮换必须二次确认、短时展示、写审计日志和安全事件。
- dev 环境可允许 localhost 和 RFC1918 私有 IPv4 HTTP 回调；test、prod 环境必须使用 HTTPS 精确匹配。

## 5. 原型方向记录

### 5.1 方向 A：清单与详情抽屉型

定位：传统后台最稳妥的应用管理页。该方向已被确认为 `v0.8.0` 应用管理页重构方向，并在评审后收敛为“清单 -> 点击详情 -> 右侧抽屉打开明细”的交互范式。

页面结构：

- 顶部为 Feishu IAM 全局栏和用户区，不作为模块导航页签。
- 左侧为稳定一级菜单。
- 内容区上方为应用管理页头、主操作和筛选区。
- 下方为应用清单表格，默认不展示任何应用详情。
- 点击清单行内 `详情` 后，右侧打开应用详情抽屉。
- 应用详情抽屉使用二级 tab：`基础信息`、`环境与回调`、`Client`、`操作记录`。

关键交互：

- 搜索、筛选、分页和行内 `详情` 操作并行，打开和关闭详情抽屉时不丢失列表上下文。
- 详情抽屉不改变当前路由主页面；如 URL 携带应用上下文，可在进入页面时自动打开对应应用详情抽屉。
- 创建应用使用弹窗或抽屉；编辑基础信息、管理回调、停用应用从详情抽屉触发。
- 查看或轮换 secret 必须进入确认弹窗，确认后短时展示。

优势：

- 最接近 Ant Design / ProComponents 的中后台范式。
- 批量检索、多应用连续操作、平台管理员日常运营效率最高。
- 重构风险较低，容易拆成 `ApplicationList`、`ApplicationDetail`、`ClientPanel` 等模块。

风险：

- 视觉方向较保守。
- 如果详情内容继续增长，需要严格控制 tab 层级和抽屉宽度，避免详情抽屉变成长表单。

### 5.2 方向 B：应用档案型

定位：突出单个应用的完整接入状态。

页面结构：

- 左侧为应用选择列表。
- 中间为当前应用档案，集中展示应用基础信息、接入环境、client、secret 状态和操作记录入口。
- 右侧为风险提示和快捷操作区。

关键交互：

- 选择应用后，中间展示该应用完整档案。
- 每个档案章节都有明确入口：基础信息编辑、环境与回调管理、Client 管理、操作记录查询。
- 右侧风险区根据当前应用裁剪操作，例如 secret 轮换、prod 回调复核、停用应用。
- 应用管理员进入后更像处理“我的应用”，而不是面对平台级列表。

优势：

- “应用管理吸收 SSO 配置”的概念最清楚。
- 对应用负责人更友好，单个应用的接入状态、风险和下一步操作更直观。
- 适合把操作记录、安全事件和权限入口作为应用上下文的一部分暴露。

风险：

- 批量运营效率弱于方向 A。
- 应用数量较多时，左侧应用选择需要强搜索和状态筛选，否则会变重。

### 5.3 方向 C：环境矩阵型

定位：突出 dev、test、prod 环境治理和接入安全。

页面结构：

- 内容区以当前应用为上下文。
- 左侧为应用总览。
- 右侧按 `dev`、`test`、`prod` 三列展示环境配置。
- 每列包含回调地址、client、secret 状态和环境级操作。
- 下方提供新增回调、查看 secret、轮换 secret、跳转权限等操作说明。

关键交互：

- 所有回调地址和 client 操作都必须带环境上下文。
- 新增回调先选择环境，再按环境规则校验 URL。
- prod 环境的高风险操作更醒目，例如 secret 查看、client 停用、正式回调修改。
- 从应用矩阵跳转到权限管理时保留应用上下文，不在应用管理页直接维护 IAM 角色。

优势：

- 环境边界最清楚，能减少 dev/test/prod 配置混淆。
- 对 SSO 接入治理、安全复核和正式环境变更更友好。
- 适合后续扩展接入验收状态、环境级审计筛选。

风险：

- 对普通应用列表 CRUD 不如方向 A 直接。
- 如果未来环境不止 dev/test/prod，矩阵列需要可横向滚动或改为分段视图。

## 6. 首轮实现状态

首轮实现已经落到 `apps/admin-web`，覆盖以下模块：

- `应用管理`：默认展示应用清单、筛选、分页和创建入口；点击应用行内详情后打开 `应用详情` 右侧抽屉；抽屉内保留 `基础信息`、`环境与回调`、`Client`、`操作记录`。
- `权限管理`：默认展示应用选择、IAM 角色清单和创建角色入口；点击 IAM 角色详情后打开 `角色详情` 右侧抽屉；抽屉承载角色编辑、启停、绑定权限组和绑定成员。
- `管理员授权`：默认展示管理员授权清单、搜索、状态筛选和新增入口；点击 `新增管理员` 或行内 `详情` 后打开右侧抽屉，保留新增、编辑、应用范围、启停和历史角色只读规则。
- `记录查询`：默认展示审计日志或安全事件清单和筛选；点击行内 `详情` 后打开右侧抽屉展示单条记录上下文；审计日志详情中的 `before/after` 会对 `secret`、`token`、`password`、`cookie`、`authorization` 等敏感 key 做展示脱敏。
- `系统设置`：采用左侧设置项清单、右侧详情工作区；保留 `飞书同步`、`系统运行`、`版本信息` 的原有操作和 tab 语义，不强行套用行级详情抽屉。
- `顶部栏`：移除无价值的当前模块/生产后台上下文，只保留当前登录人控制区和 `退出登录`。

首轮验证结果：

- `pnpm check`：通过，覆盖全仓库 typecheck、lint 和测试。
- `pnpm build`：通过，覆盖 API Nest 构建和 admin-web Vite 构建。
- 浏览器自检：使用 Vite 本地管理端和拦截 API 数据，验证桌面和 390px 窄屏下五个主要模块无页面级横向溢出、无 console 错误、无非预期失败请求。
- 当前限制：本轮浏览器自检未使用真实 `http://localhost:3000/` 管理员 session，因为本机 API 当前要求真实管理员登录；后续生产形态验收需要补真实 session 路径。

## 7. 后续实施原则

后续继续推进 `v0.8.0` 前端重构时，默认按以下原则展开：

- 以 `方向 A：清单与详情抽屉型` 作为应用管理页主结构。
- 应用管理页优先服务平台管理员的批量运营和多应用连续操作。
- 同时保证应用管理员只看到和管理自己被授权的应用。
- 保留唐群座椅深蓝、青绿、白色和制造业技术风格作为视觉基线。
- 继续优先使用 Ant Design / ProComponents 的中后台组件体系。
- 继续保持 `清单 -> 点击详情 -> 右侧抽屉` 作为主要资源管理范式。
- 后续如果引入 Ant Design / ProComponents，应优先做等价迁移，不改变本轮确认的业务流程和页面职责。

## 8. 本轮未做

- 不调整后端 API、数据库迁移或业务模型。
- 不新增权限组、权限点 UI CRUD。
- 不发布镜像、不升级远端环境。
- 不做真实管理员 session 下的 `http://localhost:3000/` 生产形态浏览器验收。
