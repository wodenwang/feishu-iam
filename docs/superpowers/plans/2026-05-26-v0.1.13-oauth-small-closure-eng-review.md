# v0.1.13 OAuth 小收口工程评审

## 结论

通过。`v0.1.13` 只修复 `v0.1.12` 留下的 OAuth 小缺口：未登录 authorize request 恢复、过期 OAuth 数据清理、关键审计补齐。

本版本不进入 `v0.2.0` 的完整第三方应用配置产品化，不做 redirect URI 管理 UI，不做应用管理员权限边界，不做 Sync runtime。

## 范围锁定

In scope:

- 未登录用户访问 `GET /api/oauth/authorize` 时创建 pending OAuth request。
- pending request 使用服务端数据库表保存元数据，浏览器只保存 HttpOnly 随机 token cookie。
- IAM 登录成功后，在平台已初始化的前提下恢复 pending authorize request。
- 未初始化 IAM 时优先进入 `/initialize`，不得绕过首次平台管理员绑定。
- 清理过期 authorization code、OAuth bearer session、pending request。
- 审计 pending create、resume success、resume failure、cleanup summary。
- 后端测试覆盖 pending 恢复、初始化优先级、过期清理、审计。
- Demo 文档和版本元数据更新到 `v0.1.13`。

Out of scope:

- redirect URI 管理 UI。
- OIDC discovery、JWKS、PKCE、refresh token、复杂 consent 页面。
- Sync runtime、tenant token、通讯录全量同步。
- `/directory` 能力扩展。
- username/password 登录。

## 架构决策

### 1. Pending request 使用服务端表 + HttpOnly cookie

采用 `application_oauth_pending_requests` 表保存 `client_id`、`redirect_uri`、`state`、过期时间和消费状态。浏览器 cookie 只保存 `pendingToken`，数据库只存 token hash。

原因：

- 与现有 `iam_oauth_state`、`iam_session`、authorization code hash、bearer token hash 的安全模型一致。
- 避免把完整 authorize request 塞进 URL 或前端可读 cookie。
- 方便过期、消费和审计。

### 2. 初始化优先级高于 OAuth 恢复

登录成功后先判断 IAM 是否已经初始化。如果 `platform_admins` 为空，必须进入 `/initialize`，即使浏览器带着 pending OAuth request。

原因：

- IAM 未完成首次平台管理员绑定时，第三方授权不应该成立。
- 这符合飞书用户必须绑定超级管理员的项目硬边界。

### 3. 已初始化后 pending OAuth 优先于 Admin Console redirect

如果平台已初始化，且存在有效 pending request，则恢复第三方 OAuth。即使当前飞书用户不是 platform admin，也允许恢复第三方 OAuth。

原因：

- 第三方应用授权面向普通飞书用户，不应要求用户有 Admin Console 权限。
- 当前 authorize endpoint 已经只要求 `request.actor`，没有要求 `isPlatformAdmin`。

### 4. Cleanup 先做 service + opportunistic 调用

本版不引入后台调度器。清理逻辑放在 OAuth 模块 service 中，在 OAuth route 入口 opportunistic 触发；只有清理到数据时才写 `oauth.cleanup` 审计。

原因：

- 足以控制 demo/runtime 测试造成的数据增长。
- 不引入额外 worker、cron、部署复杂度。
- 后续运维产品化版本可以把同一个 service 接到调度器或 admin endpoint。

## 风险与控制

| 风险 | 控制 |
|---|---|
| pending token 被重放 | token 只存 hash，pending request 消费后写 `consumed_at`，后续恢复失败。 |
| 未初始化时绕过平台管理员绑定 | 登录成功 redirect 决策先检查 `platform_admins`，未初始化直接 `/initialize`。 |
| pending request 过期后仍恢复 | 恢复时事务内检查 `expires_at > now()`，过期写 failure audit 并清 cookie。 |
| 普通飞书用户误进 Admin Console | 只有 pending request 成功时跳第三方 callback；无 pending 时仍走 `/403`。 |
| cleanup 误删有效 session | cleanup 只删除 `expires_at <= now()` 的 code/session/pending rows。 |
| 审计过度噪音 | `oauth.cleanup` 仅在实际清理到数据时写入。 |

## 测试策略

必须覆盖：

- 未登录 authorize 创建 pending request 并跳 `/login?status=loginRequired`。
- 平台已初始化时，真实飞书 callback 后恢复 pending OAuth 并跳回 demo callback。
- 平台未初始化时，真实飞书 callback 仍进入 `/initialize`。
- pending request 过期后不能恢复，并写 resume failure audit。
- cleanup 删除过期 authorization code、OAuth session、pending request，且幂等。
- 现有已登录 OAuth happy path、invalid redirect、wrong secret、cross-app bearer token 测试继续通过。
