# 团队通用 Codex Agent 工作指南模板

本文档从 Feishu IAM 仓库的 `AGENTS.md`、设计文档和会话归档规则中提炼团队通用约定。后续新项目可以复制“模板正文”作为仓库根目录 `AGENTS.md` 的基础版本，再替换项目变量和业务边界。

## 提炼出的通用要求

### 1. 入口文档优先

- 每个项目根目录放置 `AGENTS.md`，作为 Codex 和其他 Agent 进入仓库后的首要上下文。
- `AGENTS.md` 只写稳定约定、当前阶段、权威设计来源和执行边界，不承载临时讨论。
- 如果项目状态、版本边界或权威设计变化，必须同步更新 `AGENTS.md`，避免后续 Agent 被过期上下文误导。

### 2. 中文优先

- 面向用户、开发者、运维和 Agent 的说明统一使用中文。
- 代码标识符、API 字段、数据库字段、配置项、错误码可以使用英文。
- Git commit message 可以使用英文，但文档正文必须使用中文。
- 引用外部英文资料时，必须用中文总结结论。

### 3. 会话归档

- 凡是产生项目决策、实施计划、代码变更、文档变更、排障结论或重要上下文的独立 Codex 会话，都必须归档。
- 归档目录统一为 `docs/codex-sessions/`。
- 推荐命名规则为 `YYYY-MM-DD-HHMM-简短主题.md`。
- 归档内容至少包含会话目标、用户关键要求、关键约束、设计决策、修改文件、关键命令与验证结果、未完成事项和下一步建议。
- 归档必须使用中文，不记录明文密钥、token、cookie、密码或其他敏感凭证。
- 只有极轻微的单行修改可以不归档，但最终回复中要说明原因。

### 4. 插件、技能和 harness 使用边界

- 默认按普通 Codex 工作流执行：阅读上下文、修改文件、运行验证、总结结果。
- 除非用户主动指定插件、技能或 harness 工作流，否则不要默认启用 harness 框架。
- 如果用户明确要求使用某个插件或技能，才按该插件或技能的规则执行，并在开始时说明使用原因。
- 项目文档中不要把某个插件或技能写成所有任务的默认前置条件，除非这是项目明确要求。
- 如果用户显式指定使用 Superpowers 或其他 harness，其过程文件、计划文件、检查记录和缓存目录应按该工具的工作流进入仓库，例如 `.superpowers/`。
- 如果用户没有显式指定使用 harness，不要为了常规任务自行创建 `.superpowers/` 等 harness 过程目录。

### 5. 设计来源和阶段边界

- `AGENTS.md` 必须列出当前权威设计文档、实施计划或产品决策来源。
- 实施前先阅读权威设计来源，不要绕开设计文档自行改变核心架构。
- 文档中要明确当前阶段、优先事项、纳入范围和排除范围。
- 对多阶段项目，写清楚当前版本不做什么，防止 Agent 在实现中扩范围。

### 6. 修改前保护用户工作

- 修改前先执行 `git status --short`，确认工作区状态。
- 发现已有修改时，默认视为用户或其他流程的有效修改，不要回退、覆盖或重写。
- 不要执行破坏性 Git 命令，例如 `git reset --hard`、`git checkout --`、强制推送等，除非用户明确要求。
- 优先使用 `rg` 查找文件和文本。
- 手工编辑文件时优先使用 `apply_patch`。

### 7. 验证和收尾

- 非平凡变更完成后运行相关检查，例如测试、类型检查、lint、构建或项目约定的健康检查。
- 如果无法运行检查，必须在最终回复中说明原因和剩余风险。
- 文档变更完成后，检查是否存在非预期模板变量、敏感信息或中英文混杂的说明性正文。
- 最终回复要说明修改内容、验证结果和后续建议。

### 8. 安全和敏感信息

- 不在文档、会话归档、测试数据或日志中写入明文密钥、token、cookie、密码。
- 涉及 secret、token、client credential、平台凭证时，只记录配置项名称、用途和安全处理原则。
- 失败信息面向用户时要稳定、可排查，但不暴露堆栈、密钥、内部 token 或框架默认错误。

### 9. 面向 Agent 的文档质量

- 关键接入文档应同时方便人类和 Agent 阅读。
- 对接口、命令、流程和验收要求，优先提供可复制命令、curl 示例、错误码、流程图和 checklist。
- 新增能力时同步更新 README、设计文档、接入文档、变更日志和会话归档中受影响的部分。

````markdown
## 语言要求

- 本项目所有文档必须使用中文。
- 面向用户、开发者、运维和 Agent 的说明都使用中文。
- 代码标识符、API 字段、数据库字段、配置项、错误码可以使用英文。
- Git commit message 可以使用英文，但文档正文必须使用中文。
- 如果引用外部英文资料，需要用中文总结其结论。

## 项目定位

《用 3 到 7 条说明项目是什么、服务谁、解决什么问题、第一版成功标准是什么。》

## 设计来源

当前设计、实施和产品边界的权威来源是：

- 《docs/.../项目设计文档.md》
- 《docs/.../当前版本实施计划.md》

实施前必须先阅读上述文档。除非用户明确要求修改设计，否则不要绕开设计文档自行改变核心架构。

## 当前阶段

## 安全要求

- 不要在代码、文档、测试数据、日志或会话归档中写入明文密钥、token、cookie、密码或其他敏感凭证。
- 涉及 secret、token、client credential、平台凭证时，只记录配置项名称、用途和安全处理原则。
- 面向用户的错误信息必须稳定、可理解、可排查，不显示堆栈、密钥、token 或框架默认错误。
- 所有高风险写操作必须有审计、日志或可追溯记录；如果当前阶段尚未实现，需要在设计或任务中明确补齐计划。

## 插件、技能和 harness 使用边界

- 默认按普通 Codex 工作流执行：阅读上下文、修改文件、运行验证、总结结果。
- 除非用户主动指定插件、技能或 harness 工作流，否则不要默认使用 harness 框架。
- 如果用户明确要求使用某个插件或技能，先说明使用原因，再按对应规则执行。
- 不要把某个插件或技能写成所有任务的默认前置条件，除非这是本项目明确要求。
- 如果用户显式指定使用 Superpowers 或其他 harness，其过程文件、计划文件、检查记录和缓存目录应按该工具的工作流进入仓库，例如 `.superpowers/`。
- 如果用户没有显式指定使用 harness，不要为了常规任务自行创建 `.superpowers/` 等 harness 过程目录。

## 工作流程要求

- 修改前先执行 `git status --short`，避免覆盖用户已有修改。
- 优先使用 `rg` 查找文件和文本。
- 手工编辑文件时优先使用 `apply_patch`。
- 不要执行破坏性 Git 命令，例如 `git reset --hard`、`git checkout --`、强制推送等，除非用户明确要求。
- 发现已有修改时，默认视为用户或其他流程的有效修改，不要回退或覆盖。
- 对非平凡变更，完成后运行相关检查或说明无法运行的原因。
- 项目文档变更完成后，检查是否存在非预期模板变量、敏感信息或中英文混杂的说明性正文。
- 生成或修改提交前，确认构建产物、本地密钥和非任务相关临时文件不会进入仓库；如果本次任务显式使用 harness，则保留其要求入库的过程文件。

## Codex 会话归档要求

每个独立 Codex 会话结束前，如果本次会话产生了项目决策、实施计划、代码变更、文档变更、排障结论或重要上下文，必须在项目内归档一份会话摘要，便于未来追溯。

归档位置：

```text
docs/codex-sessions/
```

命名规则：

```text
YYYY-MM-DD-HHMM-简短主题.md
```

归档内容至少包括：

- 会话目标。
- 用户原始关键要求摘要。
- 本次会话使用或形成的关键提示词/约束。
- 重要设计决策和原因。
- 修改过的文件。
- 执行过的关键命令和验证结果。
- 未完成事项和下一步建议。

注意事项：

- 会话归档必须使用中文。
- 不要记录明文密钥、token、cookie、密码或其他敏感凭证。
- 如果用户要求不归档某段敏感内容，以用户要求为准。
- 如果会话只处理非常轻微的单行修改，可以在最终回复中说明无需归档；其他情况默认归档。

## 验收和交付

- 代码变更优先提供测试、类型检查、lint、构建或可复现的手工验证证据。
- 文档变更需要确认链接、命令、路径、版本边界和模板变量语义清晰。
- 最终回复必须说明修改内容、验证结果和剩余风险。
````

## 推荐落地方式

1. 在团队级模板仓库保存本文档，并约定新项目初始化时复制“模板正文”为根目录 `AGENTS.md`。
2. 每个项目只允许在“项目定位、设计来源、当前阶段、技术栈、架构决策、业务边界”中写项目特定内容。
3. “语言要求、会话归档、插件技能边界、工作流程、安全要求、验收交付”作为团队统一条款，默认不删改。
4. 项目每完成一个版本或重要里程碑，由 Agent 同步更新 `AGENTS.md` 的当前阶段和优先事项。
