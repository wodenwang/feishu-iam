# Feishu IAM v0.11.x 后台生产操作闭环路线规格

日期：2026-05-26
状态：已形成版本建议，待进入单版本实施计划

## 1. 背景

`v0.10.0` 到 `v0.10.4` 已完成管理后台 shadcn/tweakcn/Tailwind 运行时迁移、全后台 UI/UX 收口和若干 GitLab issue 级布局修复。当前新的 GitLab issue 不再只是局部样式问题，而是集中暴露出后台信息架构、应用接入生产操作、角色授权配置、飞书同步诊断和真实数据长尾体验仍需继续收口。

因此后续版本不建议继续作为 `v0.10.x` 纯补丁线推进。推荐进入 `v0.11.x` 后台生产操作闭环系列，并在完成后台主流程后进入 `v0.12.0` 真实第三方接入验收版本。

## 2. 输入来源

本路线规格基于以下来源整理：

- `docs/superpowers/specs/2026-05-15-feishu-iam-design.md`：第一版成功标准和核心架构边界。
- `docs/superpowers/specs/2026-05-20-feishu-iam-v0.7.0-admin-ia-design.md`：后台模块化信息架构和角色中心权限边界。
- `docs/superpowers/specs/2026-05-22-feishu-iam-v0.8.1-application-onboarding-design.md`：应用接入包、应用去环境化、应用级回调地址和接入提示词边界。
- `docs/superpowers/specs/2026-05-24-feishu-iam-v0.10.0-admin-web-runtime-rebuild.md`：shadcn/tweakcn/Tailwind 运行时事实源和设计规范边界。
- `docs/superpowers/plans/2026-05-25-feishu-iam-v0.10.1-s6-css-release.md`：`v0.10.2` 或更后版本候选事项。
- GitLab issue `#5` 到 `#11`。
- 当前 `DESIGN.md` 中已补充的导航、复杂详情、表格列宽、身份字段和角色组织树交互约束。

## 4. 版本切分原则

1. 先修后台壳层、导航层级、通用展示规则，再重做具体业务页面。
2. 飞书同步失败属于生产可信性问题，应在角色组织树之前处理，因为角色成员选择依赖可用的飞书镜像数据。
3. 应用管理承载应用本身和应用下角色元数据生命周期；权限管理只负责角色授权绑定。
4. 复杂配置不再塞进窄抽屉。角色详情、组织树选择、权限组绑定等高认知负载流程必须使用 Tab 化详情页、独立详情页或默认填满的 `DetailSheet`。
5. 后续版本继续排除完整 OIDC、SAML、ABAC、资源级权限、飞书角色同步、飞书用户组同步、HTTPS、反向代理和高可用，除非用户单独开版本。

## 5. 推荐版本路线

### 5.1 `v0.11.0`：系统管理 IA 与通用体验基线

版本目标：

把后台信息架构、通用表格展示和身份字段表达统一起来，为后续应用管理、飞书同步和角色授权重构提供稳定壳层。

纳入范围：

- 处理 `#8`：后台一级导航调整为 `工作台`、`应用管理`、`权限管理`、`系统管理`。
- `系统管理` 下沉为二级菜单：`飞书同步`、`管理员授权`、`操作审计`、`系统信息`。
- 原 `记录查询` 对外展示名称统一改为 `操作审计`。
- 原系统设置中的飞书同步能力拆出为 `飞书同步` 二级菜单，剩余部署、版本、运行状态等内容收敛为 `系统信息`。
- 处理 `#10`：管理员列表、详情、授权记录、操作审计等涉及管理员身份的位置统一展示 `飞书 user_id: xxxxxx` 或等价清晰标签。
- 处理 `#11`：建立通用列表列宽策略、详情底部留白和底部关闭按钮规则。
- 保留旧 URL 的兼容跳转或重定向，避免已有深链直接失效。
- 更新 `AGENTS.md` 当前阶段，避免后续 Agent 继续误判项目停留在 `v0.8.1`。

不纳入范围：

- 不重做应用管理完整操作闭环。
- 不重做角色详情和组织树选择。
- 不处理飞书同步失败根因，除非发现是导航拆分造成的前端入口回归。

验收标准：

- 左侧导航和移动端导航均能展示系统管理二级菜单。
- 面包屑、页面标题、URL 和可见文案中的 `操作审计`、`飞书同步`、`系统信息` 命名一致。
- 常见桌面宽度下主要列表不默认出现横向滚动条。
- 详情底部最后一项不被固定操作区遮挡。
- 管理员相关界面不再裸露无法识别含义的 user id。
- Browser 自检覆盖桌面和窄屏导航、二级菜单展开、刷新后路由保持、console 和 Network。

### 5.2 `v0.11.1`：飞书同步可信诊断版本

版本目标：

处理 `#5`，把 112 环境飞书同步失败从现场现象变成可诊断、可追踪、可恢复的系统能力。

纳入范围：

- 核对 112 当前 `/version`、镜像版本、API 容器日志和 `feishu_sync_runs`。
- 识别飞书客户端错误码、失败阶段、run id、request id 和可操作诊断建议。
- 在 `系统管理 / 飞书同步` 页面展示同步状态、最近同步结果、失败阶段和稳定中文错误。
- 区分配置缺失、飞书权限不足、网络访问失败、tenant token 获取失败、字段权限不足和代码兼容问题。
- 真实触发一次 112 飞书同步并完成验收。

不纳入范围：

- 不新增飞书角色同步或飞书用户组同步。
- 不改变飞书作为唯一身份主数据来源的架构。
- 不在 issue、文档、日志或会话归档中记录明文凭证。

验收标准：

- 112 触发真实飞书同步成功，或失败时页面和日志能明确定位失败阶段与排查方向。
- 同步历史显示合理的部门、用户、用户部门关系数量。
- 失败路径展示稳定中文错误，不暴露 secret、token、cookie、堆栈或框架默认错误。
- 后端测试覆盖至少一个飞书同步失败诊断路径。

### 5.3 `v0.11.2`：应用管理生产闭环版本

版本目标：

处理 `#6`，并承接 `#9` 中“角色元数据归应用管理”的边界，把应用接入包从创建时闭环推进到长期维护闭环。

纳入范围：

- 应用详情重组为清晰分组：基础信息、回调地址、OAuth 凭证、开发者 API、接入提示词、应用角色、操作记录、危险操作。
- 支持编辑应用名称、描述和负责人等基础信息。
- 支持维护应用级回调地址，包含字段级校验、保存中状态、成功反馈和失败提示。
- 支持启用和停用应用，并展示审计含义和状态变化。
- 接入提示词提供明确复制入口、用途说明、复制成功反馈和安全提示。
- 应用详情内增加应用下角色清单。
- 支持新增角色、编辑角色名称、角色 key、描述和状态等角色元数据。
- 权限管理不再提供新增角色或编辑角色基础信息入口。

不纳入范围：

- 不把权限点和权限组 CRUD 暴露为后台主流程。
- 不在本版本实现角色组织树授权详情。
- 不改变应用去环境化模型，不重新引入 dev/test/prod 环境概念。

验收标准：

- 112 环境应用管理能展示应用状态。
- 能编辑应用基础信息并刷新保持。
- 能编辑回调地址，非法 URL 有字段级错误提示。
- 能启用和停用应用，状态展示和审计记录同步变化。
- 能复制接入提示词，复制成功有明确反馈。
- 能在应用详情中管理角色元数据。
- Browser 自检覆盖应用清单、详情、表单、复制、启停、常见桌面宽度和窄屏。

### 5.4 `v0.11.3`：权限管理角色授权工作区

版本目标：

处理 `#7` 和 `#9` 的权限管理侧，把权限管理收敛为角色授权绑定工作区。

纳入范围：

- 权限管理首页改为角色列表，支持按应用筛选和搜索角色。
- 打开角色进入角色详情，至少包含 `组织与用户绑定`、`权限组绑定` 两个 Tab。
- 复杂详情使用独立详情页或默认填满的 `DetailSheet`，不得使用窄抽屉承载组织树和多列表选择。
- 组织与用户绑定采用飞书组织树交互：
  - 左侧为待选区，展示组织树和组织下用户。
  - 勾选用户表示绑定用户主体。
  - 勾选组织表示绑定组织主体。
  - 右侧为已选区，清楚区分已选用户和已选组织。
  - 已选内容支持移除、搜索和按类型过滤。
- 权限组绑定放在同一角色详情的另一个 Tab 中。
- 保存前展示本次变更摘要，包括新增用户、移除用户、新增组织、移除组织和权限组变化。

不纳入范围：

- 不在权限管理中重复实现角色创建或角色基础编辑。
- 不实现飞书角色同步或飞书用户组同步。
- 不实现资源级权限、ABAC、deny 规则或数据范围权限。

验收标准：

- 权限管理首页不出现新增角色或编辑角色基础信息主入口。
- 能按应用筛选角色并打开角色详情。
- 角色详情内能在同一工作区切换组织与用户绑定、权限组绑定。
- 组织树左待选、右已选交互可用，用户和组织主体区分清楚。
- 保存前能看到变更摘要，保存后刷新仍保留。
- Browser 自检覆盖角色筛选、角色详情、组织树选择、权限组绑定、刷新后路由保持、console 和 Network。

### 5.5 `v0.12.0`：真实第三方接入验收与运营增强

版本目标：

回到第一版成功标准，用一个真实内部 Web 系统完成完整接入验收，验证 Feishu IAM 不只是后台可配置，而是真正能被第三方系统稳定消费。

纳入范围：

- 选择一个真实内部 Web 系统或可代表真实系统的 demo。
- 完成 Feishu IAM 应用创建、回调地址、OAuth 凭证、开发者 API 凭证和接入提示词交付。
- 验证授权码登录、`/oauth/token`、`/oauth/userinfo`、`/api/v1/apps/{app_key}/me/permissions`。
- 第三方系统根据权限控制菜单、按钮和后端接口。
- 管理员在 Feishu IAM 修改角色授权后，第三方系统能感知权限变化。
- 补齐接入文档、curl 示例、错误码、Agent 接入提示词和验收 checklist。
- 纳入 `v0.10.2` 曾候选但未单独发布的事项：真实生产登录态数据长尾 QA、权限管理后端分页、更多工作台风险来源、飞书镜像质量增强和运营聚合 API。

不纳入范围：

- 不实现完整 OIDC Discovery、JWKS、ID Token、SAML 或默认 refresh token。
- 不实现 HTTPS、反向代理、高可用或滚动升级。
- 不实现 ABAC、资源级权限、deny 规则或数据范围权限。

验收标准：

- 一个真实第三方系统完成完整登录和权限查询闭环。
- 第三方系统不保存飞书密钥，只对接 Feishu IAM。
- Feishu IAM 审计日志可追溯管理员授权变化。
- 接入文档能被人类开发者和 Agent 直接执行。
- 生产登录态浏览器 QA 覆盖核心后台模块和第三方接入主链路。

## 6. 可选热修版本

如果 `#5` 已经阻断当前生产使用，可以先插入 `v0.10.5`：

```text
v0.10.5 = 只修 112 飞书同步失败，不重组 IA，不重做应用管理或权限管理。
```

该热修只适合处理明确生产阻断，不应顺手合并 `#6` 到 `#11`。

如果 `#5` 不阻断当前使用，则推荐直接从 `v0.11.0` 开始，按 `v0.11.0 -> v0.11.1 -> v0.11.2 -> v0.11.3 -> v0.12.0` 推进。

## 7. 长期排除项

以下事项继续保持排除，除非用户单独开版本并确认范围：

- 完整 OIDC Discovery、JWKS、ID Token、SAML。
- 默认 refresh token。
- HTTPS、域名、反向代理、高可用、滚动升级。
- ABAC、资源级权限、deny 规则、数据范围权限。
- 飞书角色同步、飞书用户组同步。
- 多租户 SaaS 化。
- 常规硬删除应用、管理员、角色、权限组或权限点。

## 8. 实施顺序建议

1. 每个版本先写独立实施计划，不直接从本路线规格进入代码修改。
2. 涉及复杂交互的 `v0.11.2` 和 `v0.11.3`，实现前先确认页面蓝图、数据契约和 Browser 自检路径。
3. 每个版本完成后都必须同步 README、CHANGELOG、会话归档、版本号、镜像发布信息和远端验收证据。
4. 后台 UI 改动完成后必须按 `AGENTS.md` 使用 Browser 做真实自检，覆盖布局、交互、console、Network 和响应式。
5. 涉及 112 真实环境诊断时，只记录配置项名称、错误码、run id、request id 和安全处理原则，不记录明文凭证。
