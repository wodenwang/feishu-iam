## 项目定位

Feishu IAM 是企业内部身份与权限管理中台。第一版目标是让一个真实内部 Web 系统完成完整接入：

1. 在 Feishu IAM 创建应用。
2. 配置 dev、test、prod 环境回调地址和 client。
3. 通过 Feishu IAM 跳转飞书登录。
4. 第三方系统用授权码换取 Feishu IAM token。
5. 第三方系统获取用户信息和权限组/权限点清单。
6. 第三方系统根据权限控制菜单、按钮和后端接口。
7. 管理员能在 Feishu IAM 修改角色授权，并通过审计日志追溯变化。

## 核心架构决策

- 飞书是身份和组织主数据的唯一来源。
- Feishu IAM 只配置一个企业级飞书自建应用，用于飞书登录和通讯录同步。
- 第三方应用不绑定独立飞书应用，不保存飞书 `app_id` 或 `app_secret`。
- Feishu IAM 是统一 SSO Provider。
- 第三方应用只对接 Feishu IAM。
- Feishu IAM 返回权限组和权限点清单，不做第三方业务资源级判断。
- 第三方应用自行决定菜单、按钮、接口和业务行为。
- 第一版实现 OIDC 兼容的简化授权码流程，不实现完整 OIDC 协议。

## 飞书身份约定

- 飞书镜像字段尽量与飞书 OpenAPI 字段名保持一致。
- 用户业务主键优先使用飞书 `user_id`。
- 必须保存 `open_id` 和 `union_id`，并在适合范围内建立唯一约束。
- 镜像表需要保留 `raw_payload jsonb`，用于保存飞书原始响应。
- 禁用、离职、主动退出或未激活的飞书用户不可登录。
- 飞书部门、用户组或角色删除后，相关绑定标记为 orphaned，不立即硬删除。

## 权限模型约定

- 应用拥有全局唯一 `app_key`。
- 权限数据按应用隔离。
- 权限点 key 必须以 `${app_key}.` 开头。
- 推荐权限点 key 校验规则：

```text
^${app_key}\.[a-z0-9][a-z0-9._-]{0,127}$
```

- 管理后台和开放 API 必须使用同一套后端校验逻辑。
- 禁止跨应用绑定权限组或权限点。
- 第一版不做 deny 规则、资源级权限、数据范围权限或 ABAC 表达式。

## SSO 与接入文档约定

- 第三方应用通过 Feishu IAM 的 `/oauth/authorize` 发起登录。
- Feishu IAM 在需要时跳转飞书完成登录。
- Feishu IAM 回调第三方应用并发放自己的授权码。
- 第三方应用后端通过 `/oauth/token` 换取 Feishu IAM access token。
- 第三方应用通过 `/oauth/userinfo` 获取用户信息。
- 第三方应用通过 `/api/v1/apps/{app_key}/me/permissions` 获取权限组和权限点。
- 登录失败必须展示 Feishu IAM 统一错误页，不能显示裸 JSON、堆栈或框架默认错误。
- 接入文档必须包含详细流程图、curl 示例、错误码、验收 checklist 和 Agent 接入指南。

## 安全与审计约定

- 所有 Web/API 写操作必须写入审计日志。
- 审计日志至少包含操作者、来源、目标资源、操作类型、before/after diff、结果、IP、User-Agent、时间和 request id。
- `client_secret`、平台 API token、飞书 `app_secret` 只能保存哈希或加密密文。
- 明文 secret 只允许在创建或轮换时展示一次。
- 回调地址必须精确匹配，不允许通配符回调。
- `dev` 环境可以允许 `localhost` 和 RFC1918 私有 IPv4 HTTP 回调，用于内网验收；`test`、`prod` 环境仍必须使用 HTTPS。
- 删除操作默认禁用或软删除，第一版后台不提供常规硬删除。

## 前端 Admin UI 开发规范

后续涉及管理后台页面、前端交互、CRUD 资源管理或响应式布局时，真实设计规范以仓库根目录 `DESIGN.md` 为准。不得再使用已过时的 `feishu-iam-admin-ui` 技能作为默认前置工作流；如需使用其他插件或技能，必须由用户明确指定，并说明使用原因。

`v0.10.0` 之后的管理后台前端工作默认按 `DESIGN.md` 的 shadcn/ui + tweakcn + Tailwind 规范执行。UI/UX 策划不需要拘泥旧实现交互；Pencil 原型只作为流程和状态参考，不作为像素级还原目标；Pencil 原型本身也必须严格遵守 shadcn/tweakcn 约束，不能自由发挥成与实现体系冲突的视觉稿。

### 技术和组件边界

- Feishu IAM 管理后台组件体系以 `DESIGN.md` 的当前版本决策为准；`v0.10.0` 起采用 shadcn/ui + tweakcn + Tailwind 作为前端设计和实现基线。
- 不要把旧 Ant Design / ProComponents 或过时技能说明作为当前默认基线；如后续再次替换 UI 组件体系，必须先说明原因、迁移影响和与 `DESIGN.md` 的关系。
- 管理后台必须是传统后台管理系统，而不是营销页、门户页或展示型 dashboard。
- 页面布局应保持左侧导航、顶部用户区、内容区、表格/表单工作区的稳定结构。
- UI 风格遵循唐群座椅深蓝、青绿、白色和制造业技术风格，整体克制、专业、信息密度适中。
- 避免装饰性 hero、大面积渐变、漂浮卡片、过度圆角、低密度大卡片堆叠和与后台系统无关的视觉噪声。

### CRUD 页面最低交付标准

每个资源管理页面至少应具备：

- 列表表格，包含加载、空状态、错误状态和正常数据状态。
- 查询筛选区，包含查询、重置和必要的字段校验。
- 分页能力，优先使用后端返回的总数和分页参数。
- 新增流程，包含表单校验、提交中状态、成功反馈和失败反馈。
- 编辑流程，能够加载当前值，区分可编辑字段和不可变字段。
- 禁用、启用、软删除或其他状态变更流程；除非设计明确要求，不提供常规硬删除。
- 详情入口，用于承载表格中不适合直接展示的长文本、回调地址、权限点、审计信息等。
- 操作列宽度稳定，操作顺序一致，危险操作必须有确认。
- 权限不足时，按钮应禁用、隐藏或展示稳定可理解的提示，不能依赖前端假权限绕过后端校验。

### 布局和交互要求

- 后续功能模块默认采用“清单 -> 点击详情 -> 右侧抽屉打开明细”的主范式。页面默认只展示清单、筛选、分页和主操作；点击行内 `详情` 后再打开右侧抽屉。
- 详情抽屉关闭后应回到原清单状态，保留当前筛选条件、分页、排序和必要的滚动位置。
- 从其他模块跳转或 URL 携带资源上下文时，可以默认打开对应资源的详情抽屉，但不能让普通入口默认占用清单空间展示某条资源详情。
- 表格、筛选区、表单弹窗、抽屉、分页、侧边栏和顶部栏必须在常见桌面宽度下稳定展示。
- 长 `app_key`、权限点 key、URL、回调地址、描述文本等必须有换行、截断、tooltip 或复制能力，不能撑破布局。
- 弹窗和抽屉必须有明确标题、主次按钮、关闭行为、提交中状态和错误展示。
- 表单错误信息要贴近字段展示；接口错误要用稳定中文说明，不暴露堆栈、secret、token、cookie 或框架默认错误。
- 响应式布局至少保证窄屏下导航、筛选区、表格横向滚动、弹窗内容和操作按钮可用。

### 开发流程要求

- 前端实现前先梳理当前页面的数据契约：字段、筛选项、分页参数、创建/编辑表单、行操作、权限和审计影响。
- 优先复用已有组件、样式变量、数据请求封装、错误处理和路由结构，避免每个页面重复造一套表格和表单。
- 如果同类 CRUD 页面达到两个以上，再考虑抽取通用资源页模式；不要为了单页提前做过度抽象。
- 代码实现后运行相关检查，通常为 `pnpm check` 或更窄范围的前端类型检查、lint、构建命令。

### Browser 自检硬约束

涉及管理后台页面、前端交互或响应式布局的开发完成后，必须使用 `@Browser` 打开本地页面进行真实浏览器自检，默认入口为 `http://localhost:3000/`，并在最终交付中说明检查结果。

自检至少覆盖：

- 页面是否存在明显布局错位、内容溢出、元素遮挡或异常空白。
- 表格、筛选区、表单弹窗或抽屉、分页、侧边栏和响应式布局是否符合后台管理系统使用习惯。
- 新增、编辑、禁用/启用、筛选、重置、分页等基础 CRUD 交互是否可用。
- 浏览器 console 是否存在错误，Network 是否存在非预期失败请求。
- 如果 UI 不符合后台管理系统标准，必须继续修复并重复自检，直到通过后再交付。

## Codex 会话归档要求

每个独立 Codex 会话结束前，如果本次会话产生了项目决策、实施计划、代码变更、文档变更、排障结论或重要上下文，必须在项目内归档一份会话摘要，便于未来追溯。

归档位置：

```text
docs/codex-sessions/
```

命名规则：

```text
YYYY-MM-DD-HHMM-简短主题.md
```

归档内容至少包括：

- 会话目标。
- 用户原始关键要求摘要。
- 本次会话使用或形成的关键提示词/约束。
- 重要设计决策和原因。
- 修改过的文件。
- 执行过的关键命令和验证结果。
- 未完成事项和下一步建议。

注意事项：

- 会话归档必须使用中文。
- 不要记录明文密钥、token、cookie、密码或其他敏感凭证。
- 如果用户要求不归档某段敏感内容，以用户要求为准。
- 如果会话只处理非常轻微的单行修改，可以在最终回复中说明无需归档；其他情况默认归档。
