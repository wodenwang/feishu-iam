## 信息架构

`应用管理` 继续采用“清单 -> 点击详情 -> 右侧抽屉打开明细”的主范式。清单只展示可扫描字段：应用、`app_key`、状态、负责人、接入完整度、更新时间和详情入口。长回调地址、提示词、secret、角色描述和审计信息全部收拢到详情。

应用详情按以下分区组织：

1. 概览：状态、负责人、接入完整度和关键入口。
2. 基础信息：名称、描述、负责人，支持编辑和字段级错误。
3. 回调地址：应用级精确 URL 列表，支持新增和禁用。
4. 接入包：OAuth credential、developer credential、安全版 Codex 接入提示词和复制反馈。
5. 应用角色：角色元数据清单、新增角色、编辑角色名称 / key / 描述 / 状态。
6. 操作记录：跳转到操作审计，不在本原型展开。
7. 危险操作：启用 / 停用应用和影响确认。

## 关键交互决策

- 回调地址维护优先采用“新增新地址 + 禁用旧地址”，不在原型中设计原地编辑 URL，避免审计语义不清。
- 安全版接入提示词不包含明文 secret、developer token 或 cookie，提供明确复制按钮和复制成功反馈；`v0.11.2` 只要求安全版提示词复制闭环。若本版不实现 secret 轮换 API，完整提示词按钮必须置灰并标注“后续版本”。
- 应用状态展示不只依赖状态 badge，清单增加接入完整度和停用影响摘要。停用应用不删除配置；后续 `/oauth/authorize`、`/oauth/token`、`/oauth/userinfo`、权限清单和 developer API 请求返回稳定 `application_disabled` 错误。已有 access token 不物理删除，但后续使用被拒绝。
- 停用应用时保留 OAuth client、developer credential、角色元数据和既有授权绑定；停用期间配置只读，重新启用后恢复。启用 / 停用动作必须写入操作审计，记录 before / after、操作者、IP、User-Agent 和 request id。
- 应用角色元数据属于应用管理。权限管理只负责后续授权绑定工作区；本原型只保留后续授权入口标注 `v0.11.3`，不绘制组织树、用户 / 部门绑定或权限组绑定交互。
- 角色元数据必须覆盖空状态、无权限只读状态、保存失败状态和接口失败重试状态，避免工程实现只覆盖 happy path。
- 窄屏下详情默认全屏，底部保留关闭 / 保存按钮；长 URL 和提示词只在详情内复制。

## 明确不纳入

- 不画 GitLab issue `#7` 的飞书组织树、用户 / 部门绑定、权限组绑定或角色授权工作区。
- 不画 GitLab issue `#13` 的飞书同步运维控制台。
- 不扩展完整 OIDC、ABAC、资源级权限、HTTPS、反向代理、高可用或滚动升级。
- 不把权限点和权限组 CRUD 暴露为后台主流程。

## 进入工程计划前的检查点

- 确认回调地址本版只做新增 / 禁用，还是必须支持编辑；当前原型推荐新增 / 禁用。
- 确认安全版提示词复制 API 或前端组装路径；“轮换 secret 生成完整提示词”不作为 `v0.11.2` 必须能力，除非工程评审确认已有后端 API。
- 确认应用角色新增时 `role key` 是否可编辑，编辑已有角色时 `role key` 是否不可变；当前原型展示新增时可填写，编辑时需在工程计划中锁定。
- 工程计划必须落地停用应用语义：新 OAuth 请求拒绝，已有 access token 后续使用拒绝，developer API 拒绝，角色元数据和授权绑定保留只读，审计写入 `application.disable` / `application.enable`。
- 确认 Browser 自检路径覆盖桌面、390px 窄屏、清单、详情、编辑、回调、新增角色、复制提示词和启停确认。
