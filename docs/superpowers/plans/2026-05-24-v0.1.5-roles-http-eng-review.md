## 范围

`v0.1.5` 只把已有 Roles 后端能力接入 Admin Console HTTP mode：

- `GET /api/roles`
- `POST /api/roles`
- `PATCH /api/roles/:id`
- `PUT /api/roles/:id/authorization`
- 从 runtime 派生 permission tree
- 前端错误态和 requestId 反馈

明确不做：

- 扩大 `/directory` 只读浏览边界。
- Sync runtime。
- Dashboard runtime summary。
- Application Detail / Onboarding HTTP runtime。
- 真实飞书 OAuth。
- 新的批量角色 API。
- 部署、tag、release。

## 架构决策

1. 后端先增强 `GET /api/roles` 返回前端需要的 projection：应用名、权限数量、绑定数量、当前授权 keys、部门 IDs、用户 IDs。
2. 前端 `httpApi` 增加 role mapper 和 HTTP methods，保持 UI 只依赖 `iamService`。
3. `listIamPermissionTree` 从 runtime permission groups / points 聚合，不新增后端路由；复用 `GET /api/roles` 返回的当前应用角色授权不足以构造完整权限树，所以后端增加只读 `GET /api/roles/permission-tree`。
4. `UpsertRoleInput.applicationId` 在前端仍使用 application id；HTTP method 内映射到对应 application 的 `appKey` 后提交给后端。
5. 批量停用在 HTTP mode 逐个调用 `PATCH /api/roles/:id`，避免为本切片增加新的后端批量接口。

## 数据流

```text
RolesPage
  -> useRoles / useCreateRole / useUpdateRole / useDisableRoles / useUpdateRoleAuthorization
  -> iamService
  -> httpApi
  -> Fastify roleRoutes
  -> PostgreSQL roles / permission_points / role_*_bindings / applications
```

## 风险与处理

| 风险 | 处理 |
|---|---|
| 角色列表 projection 不足导致 UI 显示空授权 | 后端查询聚合 permission keys、departmentIds、userIds 和 count。 |
| 前端以 applicationId 提交，后端需要 appKey | `httpApi` 缓存或查询 applications，提交时转换。 |
| permission tree 没有 runtime 来源 | 增加只读 `GET /api/roles/permission-tree`，只暴露 code/name/status。 |
| HTTP 错误被泛化 | 复用 `IamHttpError`，页面 message 和 Alert 展示 requestId。 |
| `/directory` 边界被扩大 | 只消费现有 departments/users 只读接口，不修改 directory routes。 |
