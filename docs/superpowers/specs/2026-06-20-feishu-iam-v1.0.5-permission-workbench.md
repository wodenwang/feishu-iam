# Feishu IAM v1.0.5 权限管理角色配置工作台设计说明

## 当前状态

本设计说明已按最新方向修订为 v1.0.5 定稿。用户已确认当前 Product Design 静态原型作为后续工程设计输入。

用户已明确：角色是独立资源，不再附属于应用；`权限管理` 首屏是角色列表；角色列表支持查询、启用 / 停用和批量操作；点击 `配置` 进入角色配置页；配置页拆成 `组织与用户` 和 `应用权限` 两个一级工作区，避免单个界面过重；应用管理移除 `角色管理`，聚焦密钥、回调、Codex 提示词和应用权限资产查看。

最新补充约束：

- `组织与用户` 不拆成两个独立选择器，必须在同一个组织树选择器中勾选组织节点和用户节点。
- `应用权限` 中的权限点对比能力必须保留，作为理解权限组差异的核心区域。
- `角色配置页` 必须是独立页面，不能做成抽屉、Sheet 或弹窗；用户通过 `返回角色列表` 回到角色列表。

## 版本目标

`v1.0.5` 的主要目标是优化权限管理流程，把角色配置和授权相关操作统一收敛到一个 PC 端工作台中。

核心改进点：

- 绑定组织 / 用户和绑定应用权限拆成两个角色配置工作区完成。
- 管理员不再在多个割裂 Tab 中来回切换。
- 每个工作区都有独立保存摘要和保存动作。
- 移动端只做兜底可用，不作为主要体验优化范围。
- 角色列表支持启用 / 停用和批量启用 / 停用。
- 角色可以绑定多个应用；权限组绑定必须按应用上下文处理。
- 应用管理不再提供角色管理。
- 本版本必须移除原 `应用管理` 中的角色管理部分，角色新增、编辑、删除、启停和配置统一归入 `权限管理`。

## 已确认模型

- 角色是独立 IAM 资源，不再附属于应用。
- 权限管理拥有角色列表、角色状态启停和批量启停。
- 角色与应用之间是多对多关系。
- 组织 / 用户绑定在角色级处理，回答“谁拥有这个角色”。
- 权限组仍然归属应用，角色绑定权限组时必须先确定应用上下文。
- 权限点是只读结果，按当前应用展示。
- 权限计算按应用执行：当前应用 -> 找到绑定该应用的角色 -> 匹配拥有该角色的主体 -> 汇总该应用下绑定的权限组和权限点。
- 应用管理移除角色管理，保留应用权限组和权限查看、查询、比对能力。
- 应用详情页不再展示 `角色管理` Tab 或角色相关操作入口；如需引导配置角色，只提供跳转 `权限管理` 的链接。

## 用户进入权限管理后的主流程

1. 用户点击左侧导航 `权限管理`。
2. 页面进入 `权限管理角色列表`。
3. 用户使用应用、名称、编号、状态等查询条件定位角色。
4. 用户可在列表中启用 / 停用角色，或勾选多个角色做批量操作。
5. 用户点击目标角色行的 `配置`。
6. 页面进入 `角色配置工作台`。
7. 用户进入 `组织与用户` 工作区，通过同一个组织树选择器维护拥有该角色的组织和用户。
8. 用户进入 `应用权限` 工作区，切换应用、绑定该应用下的权限组，并查看权限点只读预览和比对结果。
9. 用户在对应工作区的保存检查栏核对变更。
10. 用户分别保存 `组织与用户` 或 `应用权限` 配置。

## 工作台信息架构

### 顶部上下文

- 面包屑：`后台 / 权限管理 / 角色配置工作台`
- 标题：`角色配置工作台`
- 当前角色名称、角色 key、角色状态。
- 角色编号。
- 已关联应用数量。
- 当前正在配置的应用。
- 返回角色列表。
- 当前登录管理员。

顶部必须压缩，不再重复展示两个“角色详情”大卡片。

### 角色配置页工作区

配置页使用顶部 Tabs 拆分重任务：

1. `组织与用户`
   - 添加授权对象。
   - 使用同一个组织树选择器展示组织和用户。
   - 组织节点和用户节点都可勾选。
   - 搜索组织、用户、部门路径或 user_id。
   - 展示已选组织 / 用户。
   - 移除已选主体。
   - 保存按钮：`保存组织与用户`。

2. `应用权限`
   - 切换应用。
   - 添加应用。
   - 添加权限组。
   - 权限组选择器必须限定在当前应用内。
   - 展示已绑定权限组。
   - 查看权限组内权限点。
   - 只读展示当前应用下的最终权限点。
   - 支持权限组和权限点查询、比对；权限点比对是主工作区能力，必须保留。
   - 保存按钮：`保存应用权限`。

3. `变更记录`
   - 查看组织 / 用户、应用、权限组绑定变更。
   - 支持跳转操作审计。

### 右侧保存检查栏

每个工作区右侧栏只承担当前工作区的结果核对和保存动作：

1. `当前授权对象`
   - 已选组织数。
   - 已选用户数。
   - 主要已选对象。

2. `最终权限`
   - 已绑定权限组数。
   - 最终权限点数。
   - 授权范围。

3. `变更与保存`
   - 新增 / 移除组织。
   - 新增 / 移除用户。
   - 变更涉及的应用。
   - 只展示当前工作区相关变更。
   - 保存按钮。

## 关键规则

- 选择组织表示绑定组织主体，不展开保存为全部用户。
- 选择用户表示绑定用户主体，可以与组织主体并存。
- 组织和用户必须在同一个选择器中勾选，不拆成两个添加入口。
- 权限组是可绑定对象，但必须带应用上下文。
- 权限点是只读结果，不在权限管理中编辑，且按当前应用展示。
- 权限点对比必须保留在 `应用权限` 主界面，不能降级为隐藏弹窗。
- 角色是独立资源，不再归应用管理维护。
- 角色配置页必须是独立页面，不做抽屉式详情。
- 角色启用 / 停用在权限管理角色列表中完成。
- 保存失败必须保留草稿。
- 应用停用或角色停用时进入只读模式。
- 所有写操作必须写入操作审计。

## 本版本不做

- 不新增 DDL，除非后续工程评审证明现有接口无法支撑统一草稿。
- 不实现权限点 CRUD。
- 不实现完整用户全局权限查询。
- 不实现 ABAC、资源级权限、deny 规则或数据范围权限。
- 不改变管理员 session 和管理员权限校验机制。
- 不做移动端体验精修，只做不溢出、不遮挡、基础链路可用。

## 工程评审重点

当前 Product Design 方案已锁定。下一步应进入 `gstack /plan-eng-review`，工程评审需要重点确认：

1. 是否新增 `role_application_bindings` 或等价模型。
2. 主体绑定继续挂在角色上时，现有权限计算如何按应用汇总。
3. 权限组绑定如何从角色维度迁移为角色-应用维度。
4. 是否需要新增聚合接口返回角色、关联应用、每个应用下权限组、最终权限点和主体摘要。
5. 旧 `tab=subjects` 和 `tab=groups` 深链如何兼容到新工作台的应用上下文。
6. 右侧保存检查栏如何按应用计算 diff。
7. 保存失败时如何保留草稿并展示 request id。
8. 现有测试如何覆盖角色-应用多对多、授权对象、授权内容、只读状态、保存失败和权限点只读预览。
9. 应用管理移除角色管理后的路由兼容、Tab 清理、按钮清理和回归测试。

## 完成标准

- `权限管理` 首屏为角色列表。
- 角色列表支持应用、名称、编号、状态查询。
- 角色列表支持启用 / 停用和批量操作。
- 角色列表行操作明确为 `配置`。
- 角色详情默认进入 PC 端配置工作台。
- 角色配置通过独立页面打开，并提供 `返回角色列表`。
- 角色工作台能够展示和管理角色关联应用。
- 组织 / 用户绑定和按应用权限组绑定拆成两个工作区完成。
- `组织与用户` 工作区使用同一个组织树选择器勾选组织和用户。
- 权限组绑定有明确应用上下文，不允许跨应用误绑。
- 权限点对比保留为 `应用权限` 主工作区的一部分。
- 应用管理不再提供角色管理。
- 应用管理应用详情中原 `角色管理` Tab / 入口 / 操作按钮已移除。
- 应用管理保留密钥、回调、Codex 提示词、权限组 / 权限查看查询比对。
- 当前工作区右侧保存检查栏始终展示对应摘要：`组织与用户` 展示主体摘要，`应用权限` 展示当前应用、权限组和最终权限摘要。
- 保存按钮按工作区区分为 `保存组织与用户` 和 `保存应用权限`。
- 旧深链不会失效，至少能映射到新工作台对应区域。
- 自动化检查通过。
- 浏览器检查覆盖 PC 端主要路径。
