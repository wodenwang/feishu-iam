# Feishu IAM 权限管理 SOP

## 当前设计口径

本 SOP 已按最新方向调整为 v1.0.5 定稿。用户已确认当前 Product Design 静态原型作为后续工程设计输入。

用户确认的核心方向：

- 角色是独立资源，不再附属于应用。
- `权限管理` 首屏就是角色列表。
- 角色列表支持按应用、名称、编号等条件查询。
- 角色列表支持启用 / 停用，并支持批量操作。
- 点击 `配置` 进入角色配置页。
- 角色配置页拆成两个一级工作区：
  - `组织与用户`：绑定组织和用户。
  - `应用权限`：绑定应用、权限组并查看权限点。
- 角色配置页必须是独立页面，不做抽屉、Sheet 或弹窗；用户通过 `返回角色列表` 回到列表页。
- 组织和用户必须在同一个组织树选择器中勾选，不拆成两个独立选择器。
- 权限点对比是 `应用权限` 的核心能力，必须保留。
- 两类绑定不强行塞进一个界面，避免单页过重。
- 应用管理移除 `角色管理`。
- 本版本必须把原 `应用管理` 中的角色管理部分移除，包括应用详情中的角色管理 Tab / 入口 / 操作按钮；角色新增、编辑、删除、启停和配置统一回到 `权限管理`。
- 应用管理聚焦密钥刷新 / 设置、回调设置、Codex 提示词生成，以及应用权限组和权限的查看、查询、比对。

当前推荐产品模型：

- 组织 / 用户主体绑定在角色级生效，即“谁拥有这个角色”。
- 权限组 / 权限绑定在角色-应用上下文中生效，即“这个角色在某个应用中拥有哪些权限组和权限点”。
- 权限计算按应用执行：当前应用 -> 找到绑定该应用的角色 -> 匹配拥有该角色的主体 -> 汇总该应用下绑定的权限组和权限点。

## 适用范围

本文描述 Feishu IAM 管理后台中 `权限管理` 模块的标准使用流程。重点是 PC 端后台配置体验，面向平台管理员和应用权限管理员。

权限管理的目标是围绕独立 IAM 角色完成管理和授权配置：管理员先在角色列表中找到角色，再进入配置页，分别配置“谁拥有这个角色”和“该角色在每个应用下带来哪些权限”。

它不负责创建应用、刷新应用密钥、设置应用回调或生成应用接入提示词；这些能力留在 `应用管理`。

## 权限管理的边界

### 本模块负责

- 查看 IAM 角色授权状态。
- 查看角色基础信息。
- 启用 / 停用角色。
- 批量启用 / 停用角色。
- 查看或维护角色关联的应用。
- 进入某个角色的授权工作台。
- 绑定或移除飞书组织主体。
- 绑定或移除飞书用户主体。
- 在应用上下文中绑定或移除权限组。
- 查看该角色在当前应用下最终拥有的权限点。
- 保存授权变更并触发操作审计。

### 本模块不负责

- 创建应用。
- 创建、编辑、停用权限点。
- 创建、编辑、停用权限组的基础信息。
- 刷新或设置应用密钥。
- 设置应用回调地址。
- 生成应用 Codex 接入提示词。
- 实现资源级权限、ABAC、deny 规则、数据范围权限。
- 同步飞书角色或飞书用户组。

这些能力应分别放在 `应用管理`、开发者 API、飞书同步或后续专门能力中。`应用管理` 不再承载 `角色管理`。

### 应用管理同步调整

本版本必须同步清理原 `应用管理` 中的角色管理能力：

- 应用详情页不再展示 `角色管理` Tab。
- 应用详情页不再提供角色新增、编辑、删除、启用、停用或授权配置入口。
- 应用详情页如需提示角色配置，应只提供跳转到 `权限管理` 的轻量链接。
- 应用管理保留 `详细资料`、`开发信息`、`回调与密钥`、`Codex 提示词`、`权限资产`、`危险操作` 等应用自身能力。
- `权限资产` 只允许查看、查询和比对权限组 / 权限点；不承担角色授权。

## 用户进入权限管理后首先看到什么

用户从左侧导航点击 `权限管理` 后，首先进入 `权限管理角色列表`。

PC 首屏应让用户能同时理解：

- 当前有哪些角色。
- 每个角色关联了哪些应用或关联应用数量。
- 每个角色绑定了多少组织 / 用户。
- 每个角色在各应用下绑定了多少权限组。

应用筛选是过滤条件，不再代表“进入某个应用下的角色管理”。

### 页面首屏结构

1. 顶部标题区
   - 标题：`权限管理`
   - 说明：查看 IAM 角色，维护角色状态，并进入工作台配置组织、用户、应用权限组和权限。
   - 当前筛选条件摘要，例如关联应用、角色状态、授权状态。

2. 筛选区
   - `应用` 下拉：过滤已绑定某应用的角色，例如 `Base Portal / base-portal`。
   - `角色查询`：搜索角色名称、角色 key 或描述。
   - `角色编号`：搜索角色编号或 role key。
   - `状态`：全部、启用、停用。
   - `授权状态`：有主体绑定、有权限组绑定、存在异常绑定。
   - `排序`：角色 key 升序、最近更新优先等。
   - `查询` 和 `重置`。

3. 角色列表
   - 角色名称。
   - 角色 key。
   - 角色编号。
   - 状态。
   - 启用 / 停用开关。
   - 关联应用数。
   - 已绑定组织数。
   - 已绑定用户数。
   - 权限组数，按应用汇总展示。
   - 最近更新时间。
   - 操作：`配置`。

4. 分页区
   - 总数。
   - 每页条数。
   - 上一页 / 下一页。

5. 批量操作区
   - 勾选角色。
   - 批量启用。
   - 批量停用。
   - 高风险批量停用必须二次确认，并展示影响角色数。

### 用户在列表页如何操作

1. 可先按关联应用过滤角色，也可直接搜索角色。
2. 如角色很多，用角色查询、应用或状态筛选缩小范围。
3. 可直接在列表中启用 / 停用角色。
4. 可勾选多个角色做批量启用 / 停用。
5. 在目标角色行点击 `配置`。
6. 系统进入该角色的 `角色配置页`。

如果用户没有选择应用，列表仍可展示全部角色或最近角色；应用只作为过滤条件。若某角色没有关联应用，行内提示 `未关联应用`，引导进入工作台补充应用绑定。

## 角色配置页 SOP

角色配置页是权限管理的主工作区。它必须拆分为两个一级工作区，避免把组织 / 用户、应用、权限组、权限点全部塞在一个重界面里。

- 当前配置的是哪个角色。
- 当前角色关联了哪些应用。
- 当前正在配置哪个应用下的权限组。
- 哪些组织 / 用户拥有这个角色。
- 这个角色在当前应用下最终带来哪些权限。

### 配置页首屏结构

1. 顶部上下文
   - 面包屑：`后台 / 权限管理 / 角色配置`
   - 标题：`角色配置`
   - 角色：`base-portal.admin`
   - 角色编号。
   - 关联应用：例如 `3 个`
   - 当前配置应用：`base-portal`
   - 状态：启用 / 停用
   - 返回：`返回角色列表`

2. 一级工作区 Tabs
   - `组织与用户`
   - `应用权限`
   - `变更记录`

3. 当前工作区主配置区
   - `组织与用户` 工作区只展示主体绑定。
   - `应用权限` 工作区只展示应用切换、权限组绑定、权限点预览和比对。

4. 当前工作区保存检查栏
   - 只展示当前工作区的变更摘要和保存按钮。

### 步骤 1：确认角色上下文

用户进入配置页后，第一眼需要确认：

- 角色名称和角色 key 是否正确。
- 角色编号是否正确。
- 角色状态是否启用。
- 角色关联了哪些应用。
- 当前正在配置哪个应用。
- 页面是否有只读提示。

如果当前应用停用或角色停用，页面进入只读模式：允许查看当前授权，但不能保存变更。

### 步骤 2：进入 `组织与用户` 工作区并配置授权对象

授权对象回答“谁拥有这个角色”。

用户可执行：

- 添加授权对象。
- 移除已选组织。
- 移除已选用户。
- 搜索组织或用户。
- 在同一个组织树中下钻组织并勾选组织或用户。

操作规则：

- `组织与用户` 工作区只提供一个统一入口：`添加授权对象`。
- 组织和用户在同一棵组织树 / 同一个选择器内展示。
- 组织节点和用户节点都可以勾选，勾选结果进入同一个已选对象列表。
- 不提供彼此割裂的 `添加组织`、`添加用户` 两套选择器。
- 选择组织表示绑定组织主体。
- 选择组织不等于自动绑定该组织下全部用户。
- 选择用户表示绑定用户主体。
- 组织主体和用户主体可同时存在。
- 重复选择同一主体不应重复保存。
- 已失效或未同步主体要保留展示，并提示风险。
- 主体绑定在角色级生效：拥有该角色的组织 / 用户，会根据该角色在不同应用下绑定的权限组获得对应应用权限。

推荐操作流：

1. 在 `授权对象` 区点击 `添加授权对象`。
2. 在统一组织树选择器中搜索或下钻组织。
3. 在同一个树中勾选目标组织节点或用户节点。
4. 确认选择器右侧的待保存对象。
5. 返回 `组织与用户` 工作区后，在列表和右侧 `当前授权对象` 中核对。
6. 如误选，点击对应行的移除按钮。
7. 点击 `保存组织与用户`。

### 步骤 3：进入 `应用权限` 工作区并配置授权内容

授权内容回答“这个角色能做什么”。

用户可执行：

- 添加权限组。
- 移除权限组。
- 切换应用。
- 搜索权限组。
- 搜索权限点。
- 比对权限组和权限点。
- 查看权限组内的权限点。
- 查看最终权限点预览。

操作规则：

- 权限组是可绑定对象，但必须属于当前配置应用。
- 权限点是只读结果，不在权限管理中直接编辑。
- 权限点来自权限组或后端返回的直接绑定结果。
- 停用权限组不应允许新绑定；如果已绑定，应显示为存量风险并允许移除。
- 切换当前配置应用时，权限组列表、已选权限组、最终权限点和权限组 diff 都必须随应用切换。

推荐操作流：

1. 在 `授权内容` 区点击 `添加权限组`。
2. 先确认当前配置应用。
3. 从当前应用的权限组列表中选择目标权限组。
4. 查看每个权限组的权限点数量和状态。
5. 权限组或权限点很多时，使用名称、key、状态、权限点数量、是否已绑定等条件查询。
6. 使用比对视图查看多个权限组覆盖的权限点差异。
7. 权限点对比必须在主工作区可见，不能隐藏成弹窗内的附属功能。
8. 展开或点击 `权限点` 查看只读权限点预览。
9. 在 `最终权限` 中确认当前应用下累计权限点数量和授权范围。
10. 点击 `保存应用权限`。

### 步骤 4：核对变更

用户在保存前必须看见变更摘要。

当前工作区的 `变更与保存` 至少展示：

- 新增组织数。
- 移除组织数。
- 新增用户数。
- 移除用户数。
- 在 `应用权限` 工作区中，新增权限组数、移除权限组数、变更涉及的应用和当前应用最终权限点变化。
- 保存后会写入操作审计。

如果没有变更，当前工作区保存按钮禁用。只读状态下，保存按钮不展示或禁用，并说明原因。

### 步骤 5：保存当前工作区配置

用户点击 `保存组织与用户` 或 `保存应用权限` 后：

1. 页面展示提交中状态。
2. 前端按当前工作区提交组织 / 用户变更或应用权限变更。
3. 后端写入操作审计。
4. 成功后刷新当前工作区数据和顶部授权摘要。
5. 右侧变更状态回到 `无未保存变更`。

如果保存失败：

- 不清空草稿。
- 显示稳定中文错误。
- 展示 request id。
- 允许用户重试。
- 不暴露堆栈、secret、token、cookie 或原始后端 detail。

## 用户日常场景与 SOP

### 场景 1：给某个部门开通应用权限

目标：让某个部门整体拥有某个角色。

操作：

1. 进入 `权限管理`。
2. 可用 `应用` 筛选缩小角色范围。
3. 搜索或定位目标角色。
4. 点击 `配置`。
5. 进入 `组织与用户` 工作区，在 `授权对象` 中添加组织。
6. 在组织树中选择目标部门。
7. 在右侧 `当前授权对象` 核对部门名称和路径。
8. 点击 `保存组织与用户`。
9. 切换到 `应用权限` 工作区。
10. 选择目标应用。
11. 确认该应用下已有正确权限组；如没有，补充绑定。
12. 点击 `保存应用权限`。

注意：

- 如果该部门下用户未来变化，是否自动影响权限取决于后端权限计算规则；UI 文案必须避免让管理员误以为“选择组织会立即展开保存所有用户”。

### 场景 2：给某个员工单独开通权限

目标：让某个具体飞书用户拥有某个角色。

操作：

1. 进入目标角色配置工作台。
2. 进入 `组织与用户` 工作区，在 `授权对象` 中搜索员工姓名或 user_id。
3. 选择用户主体。
4. 在右侧 `当前授权对象` 核对用户名称、主体类型和组织路径。
5. 点击 `保存组织与用户`。
6. 切换到 `应用权限` 工作区，确认目标应用下权限组正确。
7. 如有权限组变化，点击 `保存应用权限`。

适用：

- 临时授权。
- 跨部门授权。
- 少量特殊岗位授权。

### 场景 3：撤销某个用户或组织的角色

目标：移除不应继续拥有该角色的主体。

操作：

1. 进入目标角色配置工作台。
2. 进入 `组织与用户` 工作区。
3. 在 `当前授权对象` 或左侧已选列表中找到组织 / 用户。
4. 点击移除。
5. 在 `变更与保存` 中核对移除数量。
6. 点击 `保存组织与用户`。

注意：

- 移除组织不应误删单独绑定的用户。
- 移除用户不应影响其所在组织的授权。
- 保存前必须让管理员看清移除的是组织主体还是用户主体。

### 场景 4：调整某个角色能访问的菜单或功能

目标：通过权限组调整角色的授权内容。

操作：

1. 进入目标角色配置工作台。
2. 进入 `应用权限` 工作区。
3. 在应用切换区选择目标应用。
4. 在 `授权内容` 中查看当前应用已绑定的权限组。
5. 添加或移除权限组。
6. 展开权限组或查看权限点预览。
7. 权限组或权限点较多时，使用查询和比对能力确认覆盖范围。
8. 在右侧 `最终权限` 中核对当前应用累计权限点数量。
9. 点击 `保存应用权限`。

注意：

- 权限点不在本页面编辑。
- 如果权限点名称、key 或归属不正确，应转到开发者 API 或应用管理相关能力维护。

### 场景 5：排查某个用户为什么有权限

目标：确认用户是否因为某个角色、组织或权限组获得权限。

操作：

1. 进入 `权限管理`。
2. 可用目标应用过滤可能相关的角色。
3. 搜索可能相关的角色。
4. 进入角色配置工作台。
5. 进入 `组织与用户` 工作区，在 `授权对象` 搜索该用户。
6. 检查该用户是否直接绑定。
7. 检查该用户所在组织是否被绑定。
8. 切换到 `应用权限` 工作区。
9. 选择目标应用，查看 `授权内容` 和 `最终权限`，确认权限点来源。

后续建议：

- 如果系统已有追踪 / 审计模块，可以从工作台提供跳转到操作审计或 request id 追踪。
- 权限管理工作台只回答当前角色视角；如果要回答用户全局权限，应后续建设“按用户查权限”入口。

### 场景 6：排查某个用户为什么没有权限

目标：确认缺失发生在主体绑定、权限组绑定还是飞书同步。

操作：

1. 确认用户在飞书同步镜像中存在且可用。
2. 进入目标角色配置工作台。
3. 进入 `组织与用户` 工作区，在 `授权对象` 搜索用户。
4. 如果用户不在列表，检查是否应绑定其所在组织。
5. 如果主体已绑定，切换到 `应用权限` 工作区。
6. 检查 `授权内容` 是否包含对应权限组。
7. 如果权限组已绑定，检查最终权限点是否包含目标权限点。

常见结果：

- 用户未同步或已失效：转到飞书同步排查。
- 组织未绑定：在授权对象中补充组织。
- 用户需要例外授权：单独绑定用户。
- 权限组缺少权限点：转到权限组 / 开发者 API 维护。

### 场景 7：审批或复核一次权限变更

目标：保存前确认本次变更影响范围。

操作：

1. 管理员在 `组织与用户` 工作区完成主体调整。
2. 查看当前工作区 `变更与保存`。
3. 核对新增 / 移除组织和用户。
4. 点击 `保存组织与用户`。
5. 管理员切换到 `应用权限` 工作区完成权限组调整。
6. 核对变更涉及的应用、权限组和最终权限点摘要。
7. 点击 `保存应用权限`。
8. 保存后进入操作审计查看记录。

要求：

- 保存前摘要必须足够具体，不能只写“有未保存变更”。
- 高风险变更可要求二次确认，尤其是移除多个组织或绑定高权限组。

### 场景 8：应用或角色处于停用状态

目标：查看历史授权，但不允许继续变更。

页面表现：

- 顶部展示只读提示。
- `添加授权对象`、`添加权限组`、移除按钮和保存按钮禁用。
- 当前授权对象、权限组和最终权限点仍可查看。

解决方式：

- 如果应用停用，需要到应用管理恢复应用状态。
- 如果角色停用，可在权限管理角色列表或角色配置工作台恢复角色状态。

### 场景 9：发现已失效组织或用户

目标：处理本地镜像中已经 orphaned 的主体。

页面表现：

- 主体行展示 `已失效或未同步`。
- 仍显示原 ID 和可用的历史名称。
- 允许管理员移除该主体。

操作：

1. 核对是否确实不再需要。
2. 如为同步问题，先去飞书同步排查。
3. 如为历史残留，移除后保存。

### 场景 10：查看角色基础信息

目标：确认角色名称、key、编号、描述和状态。

操作：

1. 在权限管理角色列表查看基础信息和状态。
2. 在角色配置工作台顶部查看角色上下文。
3. 如需更完整信息，打开基础信息区域或帮助抽屉。
4. 角色启用 / 停用直接在权限管理列表或工作台处理，不再跳转应用管理。

## 页面状态 SOP

### 加载中

- 保留页面布局和区域标题。
- 角色列表显示骨架行。
- 工作台显示区域骨架，不跳动。

### 空状态

- 无筛选条件且无角色：提示暂无角色。
- 应用筛选无结果：提示当前应用未绑定任何角色，可清空筛选或进入角色配置工作台维护应用权限绑定。
- 角色无授权对象：提示添加组织或用户。
- 当前应用下角色无权限组：提示进入 `应用权限` 工作区，为该应用添加权限组。

### 错误状态

- 区域内展示错误，不整页崩溃。
- 保留已输入筛选和未保存草稿。
- 显示 request id 和重试动作。

### 无权限

- 列表页可显示无权限提示。
- 工作台进入只读或禁止进入。
- 不展示会误导用户的可点击保存动作。

### 只读状态

- 应用停用或角色停用时进入只读。
- 可查看授权对象、权限组和最终权限点。
- 不允许新增、移除或保存。

## 推荐实现切片

### 第一切片：PC 角色配置页骨架

- 权限管理入口改为角色列表。
- 角色列表支持应用、名称、编号、状态查询。
- 角色列表展示启用 / 停用开关和批量操作入口。
- 角色列表操作改为 `配置`。
- 角色详情默认进入角色配置工作台。
- 顶部上下文压缩。
- 配置页提供 `组织与用户`、`应用权限`、`变更记录` 三个一级工作区。
- `组织与用户` 工作区展示授权对象、主体选择器和当前工作区保存检查栏。
- `应用权限` 工作区展示应用切换、权限组绑定、权限点预览和当前工作区保存检查栏。
- 暂不实现应用管理重构。

### 第二切片：工作区草稿和保存

- 主体绑定和按应用权限组绑定分别维护未保存状态，顶部显示跨工作区未保存提醒。
- 当前工作区右侧显示对应变更摘要。
- 点击 `保存组织与用户` 或 `保存应用权限` 后提交当前工作区变更。
- 保存失败保留草稿。
- 兼容旧 `tab=subjects` 和 `tab=groups` 深链。

### 第三切片：排障和效率增强

- 增加 `仅看已选`、`仅看有变更`。
- 权限组展开查看权限点。
- 权限组和权限点查询、比对能力。
- 应用管理移除角色管理，保留应用权限组 / 权限查看、查询、比对。
- 加强 orphaned 主体、停用权限组和只读状态提示。
- 对接操作审计或追踪入口。

## SOP 验收标准

- 用户从 `权限管理` 进入后，能在 10 秒内定位应用和角色。
- 用户进入角色后，能清楚区分 `组织与用户` 和 `应用权限` 两个工作区。
- PC 首屏能同时看见当前工作区配置区和保存检查栏。
- 用户保存前能看清当前工作区新增和移除的组织、用户、权限组，以及涉及的应用。
- 权限点始终是只读结果，不被误认为可编辑对象。
- 应用停用、角色停用、无权限、加载失败、空数据都有稳定状态。
- 任意保存失败都不丢失用户草稿。
