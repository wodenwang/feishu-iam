## 结论

`v0.14.0` 可以进入工程评审。把 `#16` 合并进 `v0.14.0` 是合理的：导航补齐是后台壳层质量问题，和应用/角色详情页的页面级体验改造属于同一轮后台可用性收口。

## 必须保留的设计边界

1. 系统管理父级应像树节点一样可展开收起，点击父级整行即可触发；不能只把 chevron 当成唯一展开目标。
2. 应用详情和角色详情都不再默认使用右侧抽屉；它们是独立详情页。
3. 独立详情页必须有明确返回列表按钮，不能只依赖浏览器后退。
4. 返回列表应尽量保留筛选、分页、排序和搜索上下文。
5. Tab 是详情页信息架构的主骨架，不能把所有模块堆在一个长面板中。
6. 权限管理不承载角色元数据新增、编辑或启停；该边界来自 `#9`，不能因为改详情页而回退。
7. 窄屏下详情页可以变成单列 Tab 内容，但不能出现表单按钮遮挡、横向内容撑破或无法返回列表的问题。

## 原型处理

本轮不重新绘制完整 Pencil `.pen`：已有 `v0.11.2` 应用工作区、`v0.11.3` 角色工作区和 `v0.13.1` 导航原型已经覆盖视觉和交互基线；本版本的核心变化是把承载方式从抽屉升级为独立路由详情页。该变化用规格和工程计划即可表达，后续以 Browser 自检和 design-review 实现态截图作为最终视觉证据。

## 风险和设计要求

- 风险：把原抽屉内容直接搬到页面，可能变成低密度长页面。要求按 Tab 拆分。
- 风险：详情路由新增后，面包屑、返回按钮和主导航高亮可能不一致。要求 App 路由测试覆盖。
- 风险：列表上下文丢失会影响高频后台操作。要求列表查询串保留或等价恢复。
- 风险：系统管理父级整行可点击后，父级默认页面入口不明显。要求二级项中的 `系统信息` 继续作为可见入口。
