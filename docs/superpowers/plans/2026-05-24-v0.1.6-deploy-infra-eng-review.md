# feishu-iam v0.1.6 最小部署基础设施工程评审

日期：2026-05-24
分支：`codex/v0.1.6-deploy-infra`

## 结论

本版本目标锁定为 `v0.1.6` 最小部署基础设施闭环。它不新增 IAM 业务能力，只让当前仓库可以通过 Docker Compose 在 `bpmt-120:/home/bpmt/feishu-iam` 运行，并让后续 `/land-and-deploy` 能执行真实部署和健康检查。

## 已确认边界

- 常驻容器：`feishu-iam` 与 `postgres` 两个容器。
- `feishu-iam` 容器同时服务 Vite build 后的前端静态资源和 Fastify API。
- PostgreSQL 数据挂载到宿主机 `./data/postgres`。
- 应用日志使用 `docker compose logs`，不默认挂载 `./logs`。
- 远端部署目录：`/home/bpmt/feishu-iam`。
- 远端 nginx 已占用 `8000`、`8001`，部署时选择下一个空闲应用端口。
- 当前验证下优先使用 `8002`；如果部署时被占用则递增。
- 如果旧的 `/home/bpmt/feishu-iam` 已存在，本切片允许停止、备份或清理旧部署后重建。

## 架构决策

### 1. 单应用容器服务前端与 API

采用 Fastify 在生产模式服务 `dist/` 静态文件。这样 `/api/*` 与前端页面同源，保留 session cookie 简单性，不引入 CORS 和额外 nginx 容器。

### 2. Docker image 在远端本地构建

本切片先不要求 GHCR。原因是目标服务器已经可 SSH 登录，远端本地 build 足以闭环部署。后续需要正式分发镜像时再增加 GHCR push。

### 3. Compose 只管理应用和数据库

远端全局 nginx 已存在于 `/home/bpmt/nginx`，本切片不把 nginx 纳入 `feishu-iam` compose。部署阶段如需域名访问，只修改现有 nginx 反向代理到选定端口。

### 4. 生产 mock 边界保持不变

`NODE_ENV=production` 下继续拒绝 `FEISHU_AUTH_MODE=mock`。远端真实部署必须使用 `FEISHU_AUTH_MODE=real` 和专用自建飞书应用凭证；如果只跑链路验证，必须明确不是生产环境。

## 风险与处理

| 风险 | 处理 |
|---|---|
| 前端 history fallback 误吞 `/api` 404 | 静态服务仅对非 `/api` GET 请求 fallback 到 `index.html`。 |
| Secret 泄露到仓库 | 仓库只提交 `.env.example` 占位符和 compose 变量名；远端 `.env` 不提交。 |
| 端口冲突 | 部署脚本/人工部署前读取远端 nginx 配置和监听端口，选择下一个空闲端口。 |
| 数据丢失 | PostgreSQL 使用 `./data/postgres` 持久化；清理旧部署时保留或显式备份数据目录。 |
| 真实飞书 OAuth 未配置 | 本版本可以完成部署基础设施，但真实登录验收依赖服务器侧 Secret 和飞书回调配置。 |

## 测试策略

- `npm run server:build`
- `npm run server:test`
- `npm test`
- `npm run build`
- Docker image build
- 本地 Compose 启动 `feishu-iam` + `postgres`
- `curl http://127.0.0.1:<port>/api/health`
- 浏览器打开 compose 暴露端口，验证前端页面由应用容器服务
- 远端 `docker compose ps` 与 `/api/health` 检查

## 完成标准

- 仓库包含可复用 `Dockerfile`、`docker-compose.yml`、`.dockerignore`。
- Fastify 生产模式可以服务 `dist/` 前端静态资源。
- CI 至少验证 server/frontend tests、build、Docker image build。
- README/CHANGELOG/VERSION/package metadata 更新到 `v0.1.6`。
- 远端 `bpmt-120:/home/bpmt/feishu-iam` 部署成功。
- 部署报告记录 commit、端口、compose 状态、health check 结果和 nginx 状态。
