## 结论

`v1.0.7` 可以按用户确认的 B 方案在一个版本内完成 S1 和 S2，不需要拆成 `v1.0.8`。但实施顺序必须保持：

1. 先完成 S1 角色工作台减负，去掉重复上下文和错误的信息重心。
2. 再完成 S2 权限矩阵 MVP，把最终权限查询放到独立入口。
3. 权限矩阵不得反向进入角色工作台，也不得让 `应用权限` tab 继续承担跨主体、跨应用分析。

当前 PRD 范围整体可控；真正的风险不是视觉复杂度，而是信息架构边界和后端查询契约。如果 Step 3 设计制品把页面结构、状态和文案锁清楚，后续可以进入工程评审。

## UX Risks

### R1：权限管理二级入口不能用页内 Tab 替代

产品推荐使用左侧二级导航：

- `权限管理 / 角色授权`
- `权限管理 / 权限矩阵`

原因：

- `系统管理` 已经采用一级分组 + 二级入口，权限管理跟随同一 IA 模式最一致。
- 权限矩阵是查询任务，不是角色列表内的一个视图状态；放页内 Tab 会继续混淆“配置角色”和“查询最终权限”。
- 侧边栏二级入口更利于深链、面包屑和后续审计/排障文档引用。

进入 Step 3 时需要设计权限管理左侧二级导航的展开态、收起态、移动端 Sheet 状态和当前页高亮状态。

### R2：角色工作台 header 需要承接 `角色上下文`

截图显示当前有页面标题、独立 `角色上下文` 卡片和总览基础信息三层重复。Step 3 应设计为：

- PageHeader：标题 `角色配置工作台`，副标题说明配置边界。
- Header metadata：角色名称、角色 key、状态、关联应用数量。
- 总览：基础信息 + 授权摘要 + 操作审计入口。

删除独立 `角色上下文` 卡片后，页面首屏会明显减负。不要用新的大卡片替代旧卡片。

### R3：`创建角色` disabled 不应继续作为无解释按钮

当前代码显示 `创建角色` 在未选择具体应用时 disabled，即使平台管理员也不能直接点击。这个设计与“权限管理不管理角色元数据，角色属于应用之下资源”的治理约束存在张力。

建议 Step 3 给出明确交互：

- 平台管理员点击 `创建角色` 时打开创建弹框。
- 如果当前筛选为 `全部应用`，弹框第一项要求选择应用。
- 如果当前已筛选具体应用，弹框默认带入该应用。
- 应用管理员或无权限用户才禁用按钮，并通过 tooltip 或说明文案解释。

这样既保留“角色必须归属应用”的模型，也解决平台管理员看到无解释 disabled 的问题。

### R4：应用选择器弹框需要成为“管理角色关联应用”

线上截图里的 `添加应用` 是 disabled select + button，无法承载移除和变更确认。Step 3 应设计一个 Dialog：

- 标题：`管理角色关联应用`
- 顶部搜索：按应用名称 / app key 查询。
- 已绑定应用区：展示名称、app key、状态、当前应用标记、移除动作。
- 可添加应用区：展示未绑定应用、添加动作。
- 底部变更摘要：新增、移除、保持不变。
- 确认前对移除做强提示：这是软解除或停用绑定，会写入操作审计。

当前应用被移除后的 fallback 必须在弹框确认文案中说明。

### R5：应用权限页右侧只保留当前应用核对，不保留跨应用分析

`应用权限` 页可以保留两栏，但右侧边界应收窄为：

- 保存摘要。
- 当前应用最终权限点精简列表。
- 当前应用 request id / 更新时间等必要排障信息。

`权限点对比` 应移走。权限组内权限点可作为当前权限组的局部展开，不再做跨应用、跨主体比较。

### R6：权限矩阵首版必须突出“只读”和“范围说明”

权限矩阵页面建议结构：

- PageHeader：`权限矩阵`，说明用于查询用户或组织最终权限，不编辑权限。
- 查询区：主体类型 segmented control：`用户` / `组织`。
- 主体选择器：复用用户选择器或组织选择器。
- Scope note：用户查询包含用户直接绑定和所属组织绑定；组织查询只统计直接绑定该组织的角色，不展开组织下用户。
- 结果区：按应用分组的 accordion 或 cards，每个应用展示匹配角色、权限组、最终权限点。
- 空状态：未选择主体、无权限结果、无查询权限三类文案要区分。

不要把权限矩阵做成宽表格。权限点 key、角色 key、权限组 key 都较长，按应用分组的卡片/表格组合更稳。

## Accessibility Risks

- 纯 icon 行操作必须继续保留 `aria-label`、`title` 或 tooltip；新增权限矩阵和应用弹框不能只靠图标表达添加/移除。
- Dialog 打开后焦点进入弹框，关闭后返回触发按钮；移除应用确认后焦点不能丢失。
- `用户 / 组织` 切换必须是可键盘操作的 segmented control 或 Tabs，不用裸 div。
- 长 key 的复制按钮需要可读标签，例如 `复制权限点 key base-portal.demo.embedded`。
- 空、错、加载、权限不足状态不能只靠颜色或小徽标表达，必须有中文说明和可恢复动作。
- 390px 下应用弹框建议纵向堆叠为 `已绑定 / 可添加 / 变更摘要`，不做左右双栏硬压缩。

## Opportunity Areas

1. `权限管理` 二级导航可以与 `系统管理` 的 Collapsible 模式完全复用，降低学习成本。
2. `查看操作审计` 入口应从角色工作台总览直接跳转到 `系统管理 / 操作审计`，带 `resourceType=iam_role` 或至少带角色 key。
3. 权限矩阵可以成为排障文档中的固定入口：管理员输入用户或组织后，复制权限点 key 和 request id 给接入系统排查。
4. 创建角色入口可以从“disabled 按钮”升级为“选择应用后创建”，更符合平台管理员的心理模型。
