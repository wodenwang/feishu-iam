# v0.1 第三方接入 Demo 实施计划

## 目标

提供一个可运行的第三方系统 Demo，验证 feishu-iam 的 OAuth2 Authorization Code、Application API 权限注册、Agent Prompt 环境变量接入和授权结果调用链路。

## 范围

- Demo 只模拟第三方业务系统，不保存 feishu-iam 的真实 secret。
- Demo 登录必须走 feishu-iam OAuth2 Authorization Code。
- Demo 通过 Application API 注册权限组和权限点。
- Agent Prompt 只引用环境变量名，不输出真实 secret。

## 任务拆分

1. 创建 Demo 骨架
   - 使用轻量 Node.js / TypeScript Web 服务。
   - 提供首页、登录入口、回调页、业务菜单页、无权限页。
   - `.env.example` 只包含 `IAM_BASE_URL`、`IAM_APP_KEY`、`IAM_APP_SECRET`、`IAM_API_SECRET`、`OAUTH_REDIRECT_URI` 等变量名。

2. 接入 OAuth2 Authorization Code
   - 首页点击登录后跳转 feishu-iam 授权地址。
   - 回调页使用 code 换取 Demo 侧 session。
   - session 中只保存必要的用户标识、角色摘要和过期时间。
   - 禁止本地 username / password 登录。

3. 接入 Application API
   - 启动命令或管理页调用 feishu-iam 注册 Demo 权限组。
   - 权限点示例：
     - `demo.customer:view`
     - `demo.customer:update`
     - `demo.report:view`
   - 注册请求使用 `IAM_API_SECRET` 环境变量读取，不写死在代码或文档中。

4. 实现授权体验
   - 有权限用户能看到客户列表和报表入口。
   - 无权限用户进入无权限页，显示 requestId 和可联系的应用管理员。
   - 页面展示当前授权摘要，方便端到端验收。

5. 输出 Agent Prompt 示例
   - 文档提供给第三方开发 Agent 的 prompt。
   - prompt 只包含变量名、API 路径和权限命名规则。
   - 明确禁止把 `IAM_APP_SECRET`、`IAM_API_SECRET` 写入源码、日志、截图或提交记录。

## 测试与验证命令

```bash
npm --prefix examples/thirdparty-demo install
npm --prefix examples/thirdparty-demo run build
npm --prefix examples/thirdparty-demo test
npm run e2e -- tests/e2e/thirdparty-demo.spec.ts
grep -R "sec_\\|api_\\|app_secret" examples/thirdparty-demo docs/integration --exclude=.env.example || true
```

## 预期输出

- 一个可本地启动的第三方 Demo。
- Demo 能完成 OAuth 登录、权限注册、授权页面展示和无权限兜底。
- 文档能指导第三方开发者用环境变量完成接入，不暴露真实 secret。

## 完成标准

- Demo README 包含 10 分钟内可跑通的本地步骤。
- OAuth 回调、Application API 注册、授权失败页都有测试覆盖。
- E2E 至少覆盖授权成功和无权限两条路径。
- 仓库没有真实 secret、token、导出用户或同步数据。

## 风险与边界

- Demo 不替代真实第三方 SDK，仅验证接入协议和开发体验。
- v0.1 不提供多语言 SDK，先用 HTTP API 和 TypeScript 示例降低范围。
- Demo 数据必须是样例业务数据，不能混入真实客户或员工数据。
