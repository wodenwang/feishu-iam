# Feishu IAM v1.0.2 前端 UI/UX 迭代方案

## 1. 背景

`v1.0.1` 已修复旧 `/api/auth/feishu/callback` 返回框架默认 404 的问题。2026-06-13 通过线上服务 `https://feishu-iam.riversoft.com.cn/` 重新审计后，本方案不再只针对首页和公开错误页，而是覆盖登录后的完整管理控制台。

本版本按“整体 UI/UX 补丁包，分风险切片实施”的方案推进：设计上统一收口移动端可用性、公开问题页、复杂配置区和后台视觉层级；实施上先修阻塞用户使用和支持排障的 P0，再处理 P1/P2 观感与效率问题。

截图与原始审计记录位于：

- `docs/design-audits/2026-06-13-frontend-uiux/`

## 2. 目标

v1.0.2 的前端目标不是更换前端技术架构，而是在继续使用 shadcn/ui + tweakcn + Tailwind token 的前提下，把控制台关键工作流、移动端可用性、公开问题提示和整体视觉层级提升到正式版标准。

目标用户：

- 平台管理员：快速判断风险、完成应用、角色、同步、审计和管理员授权操作。
- 第三方系统用户：在 SSO 失败时能看懂问题，并只复制 `request id` 给支持人员。
- 运维和 Agent：能通过稳定的 request id、截图证据和文档规则追踪问题。

## 3. 非目标

- 不改变 shadcn/ui + tweakcn + Tailwind 的前端技术架构。
- 不引入 Ant Design、MUI、Chakra、Arco、Element Plus 或 Bootstrap。
- 不新增 DDL、不扩大权限模型、不实现完整 OIDC、SAML、refresh token、ABAC、资源级权限或数据范围权限。
- 不改变第三方应用 `redirect_uri` 精确匹配规则。
- 不保存或展示 secret、token、cookie、authorization、授权码、token hash、state hash 或 raw payload。

## 4. 设计自由度

本版本允许大幅调整 UI 与调色方案，但必须集中在设计系统与 token 层完成：

- 可以重新定义 primary、accent、sidebar、status、chart 和 problem page token。
- 可以弱化或替换现有 Riversoft 蓝绿配色，只要保留企业后台的可读性、对比度和品牌解释。
- 可以调整后台壳层、表格、筛选、详情页、复杂配置工作区、工作台和公开问题页的信息层级。
- 不允许在页面组件中零散硬编码颜色，也不允许用营销式 hero、漂浮装饰、低密度大卡片堆叠替代后台工作区。

## 5. 线上审计结论

### 5.1 已确认优点

- 登录后主导航层级稳定，`系统管理` 二级菜单在桌面和移动 Sheet 中都可理解。
- 桌面端工作台、应用管理、权限管理、飞书同步、管理员授权、操作审计、系统信息、应用详情和角色详情均可进入。
- 登录后控制台主要页面 console 未观察到 error / warn 日志。
- 飞书同步稳定态具备比较完整的运维控制台结构，组织与用户、同步历史、字段诊断和高级操作分层清楚。
- 角色详情移动端已采用 `待选 / 已选 / 摘要` 分步方向，符合复杂授权移动端降级的正确方向。
- 未登录管理后台页已经提供统一 HTML 问题提示，不暴露裸 JSON、堆栈或框架默认错误。

## 6. 下一版本方案

### 6.1 移动端列表与 Tab 收口

范围：

- 应用管理列表。
- 权限管理列表。
- 管理员授权列表。
- 操作审计 Tab。
- 飞书同步、系统信息等系统管理页面中会造成不可读或不可达的移动端布局。
- 应用详情和角色详情 Tab。

方案：

- 390px 下不允许 DataTable 把列自然压缩到不可读；列表应改为移动端行卡片、关键字段列表或明确可横向滚动表格。
- 移动端行卡片至少展示名称、关键 ID、状态、最近更新时间和主操作；长 `app_key`、角色 key、request id 必须换行或截断并可复制。
- 操作列在移动端改为单个 `更多` 菜单或固定尺寸图标组，不能挤压主字段。
- 超过 4 项的 Tab 在移动端必须使用横向滚动、折行分组、Select 或 overflow menu，不允许静态撑破视口。
- 移动端筛选区维持单列，查询和重置按钮在同一行或稳定换行，不压缩按钮文字。

验收：

- `24-mobile-applications-390.png` 同类问题消失：不再出现 `app_key` 逐字竖排、列头错位、更新时间和操作按钮重叠。
- `27-mobile-audit-390.png` 同类问题消失：Tab 不再超出视口。
- 390px 下应用管理、权限管理、管理员授权都能在首屏看清至少一条数据的名称、状态和主操作。
- 390px 下应用详情、角色详情、操作审计和系统管理二级页不因静态宽度导致页面级横向溢出。

### 6.2 复杂配置工作区

范围：

- 角色详情 `组织与用户绑定`。
- 角色详情 `权限组绑定`。
- 应用详情 `开发信息`。

方案：

- 角色组织与用户绑定保持 `待选 / 已选 / 摘要` 模型，但桌面端应明确主选择区和摘要区比例；右侧摘要可折叠或改为底部 sticky summary，避免压迫主工作区。
- 移动端继续使用分步面板，但当前步骤切换、保存按钮、变更摘要和已选主体需要更明确，不应让用户在长页面里寻找保存入口。
- 应用详情开发信息按任务分组：回调地址、OAuth credential、Developer credential、接入提示词。移动端可使用折叠分组或锚点导航，避免所有凭证和提示词一次性铺开。
- 所有复制按钮保留复制成功反馈；凭证查看和轮换继续保持危险确认和一次性 secret 边界。

验收：

- 1440px 下角色绑定主工作区不被摘要区挤压，保存入口清晰但不遮挡。
- 390px 下角色绑定能在不横向滚动的情况下完成待选、已选和摘要切换。
- 390px 下应用开发信息能快速定位回调地址、OAuth、Developer API 和接入提示词。

### 6.3 后台视觉体系升级

范围：

- `AppShell`
- `PageHeader`
- `DataTable`
- `FilterBar`
- `StatusBadge`
- `PageState`
- 工作台和系统管理类页面

方案：

- 允许重做调色方案，但必须从 `apps/admin-web/src/index.css` 的 CSS variables 开始，不在业务页面硬编码。
- 对后台壳层建立更清晰的层级：Sidebar、TopBar、内容背景、panel surface、table surface、selected / hover / focus 状态必须可区分。
- 工作台减少“解释性段落”的视觉重量，强化风险、系统健康、最近同步、审计入口的可扫描性。
- 系统信息页从稀疏分栏调整为更清晰的运维信息面板，版本、运行状态、部署入口和升级说明要能被快速扫描。
- 表格页面统一列密度、空态、错误态、筛选栏和分页节奏，避免每个模块的视觉重心不同。
- 系统管理移动端同样遵守后台自适应规则：不强求桌面完整信息一次性出现在首屏，但必须保持字段可读、Tab 可达、按钮可点、主要任务可继续。

验收：

- 1440px 下工作台、应用管理、权限管理、飞书同步、操作审计的首屏都能在 5 秒内识别主任务。
- 同类筛选区、表格、状态标签、操作列和分页视觉一致。
- 颜色 token 通过对比度检查：正文至少 4.5:1，大字和 UI 组件至少 3:1。

### 6.4 公开问题页统一

范围：

- 管理后台未登录 / session 失效 / 无权限问题页。
- OAuth `/oauth/authorize` 参数错误页。
- 飞书内部回调 `/oauth/feishu/callback` 和旧 `/api/auth/feishu/callback` 错误页。
- 第三方 SSO 回调链路中所有需要展示给用户的 HTML 错误页。

方案：

- 建立统一 `ProblemFeedbackPage` 视觉模型：品牌区、状态图标、用户可读标题、恢复建议、request id、主操作、技术详情。
- 所有公开问题页只复制 `request id`，不复制整段问题信息。
- `问题信息` 改为短错误摘要，不作为复制对象；必要时放入“技术详情”折叠区。
- 移动端内容靠上，顶部留白控制在 32-56px；主标题、说明、request id 和主操作必须出现在第一屏。
- 主操作按场景区分：管理后台是 `飞书登录`，OAuth 错误页是 `返回原系统重新登录` 或只保留复制 request id。

验收：

- `/` 未登录、`/oauth/authorize` 缺参数、`/api/auth/feishu/callback` 缺 code 三条路径在 1440px、768px、390px 下无横向溢出。
- 三条路径均只有 `复制 request id`，不存在 `复制问题信息`。
- 错误摘要不包含 secret、token、cookie、authorization、授权码、token hash、state hash 或 raw payload。

### 6.5 操作审计追踪入口

方案：

- 保持 request id 为唯一主排障凭证，不恢复整段问题信息粘贴。
- 在追踪空态增加“从公开问题页复制 request id”“从应用详情追踪入口进入”“从用户反馈复制 request id”的短提示。
- 如果存在最近查询或从应用详情带入上下文，显示轻量入口，但不保存敏感问题原文。

## 9. 计划设计评审补充决策

本节来自 `2026-06-13` 的 `gstack /plan-design-review`。原始方案方向成立，但进入 Pencil 原型和工程评审前必须补齐以下设计决策。

### 9.1 信息架构和移动端优先级

v1.0.2 的移动端不是桌面后台的缩小版。390px 下每个页面只保留一个主任务，次要信息进入详情、折叠区或横向可滚动容器。

```text
390px 管理控制台
  ├─ 顶部：品牌 / 当前页 / 菜单入口
  ├─ 页面标题：当前任务 + 1 个主操作
  ├─ 筛选区：单列，查询 / 重置清晰可点
  ├─ 主内容：
  │    ├─ 资源列表页：移动端资源卡片
  │    ├─ 多 Tab 页面：可滚动 Tabs 或 Select
  │    ├─ 复杂配置页：待选 / 已选 / 摘要分步
  │    └─ 问题页：标题 / 说明 / request id / 主操作首屏可见
  └─ 次要内容：折叠区、详情页或复制入口
```

首屏优先级：

1. 用户当前是否在正确页面。
2. 当前最重要状态或对象是什么。
3. 下一步可执行动作是什么。

禁止为了保留桌面全部列而牺牲可读性。移动端资源卡片必须比横向滚动表格优先；只有字段强依赖列对比时才允许横向滚动。

### 9.2 共享组件边界

实现阶段不得在每个页面重复写一套移动端补丁。应优先补齐共享能力：

| 组件 | v1.0.2 设计职责 | 不做 |
|---|---|---|
| `DataTable` 或配套 wrapper | 支持桌面表格 + 移动端资源卡片配置，字段包含 label、value、copyable、priority、hideOnMobile、primaryAction。 | 不在 DataTable 内推断业务字段，不让表格自动压缩到逐字竖排。 |
| `TabsList` / 项目级 Tabs wrapper | 提供移动端横向滚动、可见滚动区域、稳定 focus 和 selected 状态。 | 不在每个页面手写不同的 overflow 样式。 |
| `ProblemFeedbackPage` | 统一公开问题页结构，只复制 request id，技术详情弱化或折叠。 | 不保留复制整段问题信息或本地提取入口。 |
| `PageHeader` / `FilterBar` | 移动端单列和主操作稳定换行，避免按钮文字被压缩。 | 不把筛选塞入标题区。 |
| `OrgUserSelector` | 继续承载角色组织/用户绑定的待选、已选、摘要模型。 | 不新增资源级权限、ABAC 或后端 DDL。 |

### 9.3 状态矩阵

| 功能区域 | Loading | Empty | Error | Success | Partial |
|---|---|---|---|---|---|
| 移动端资源卡片 | 卡片骨架屏，保留标题和操作位高度 | 展示当前筛选下无结果、主操作或重置入口 | 中文错误说明 + 重试，不暴露堆栈 | 每条卡片展示名称、关键 ID、状态、更新时间、主操作 | 某些字段缺失时显示 `暂无`，不挤压布局 |
| 移动端 Tabs | 当前 Tab 内容 skeleton | 当前 Tab 无数据时说明该 Tab 语义 | 当前 Tab 错误不阻断其他 Tab | 选中态清晰，URL `tab` 状态保留 | Tab 很多时可滚动或收纳，不能截断 |
| 公开问题页 | 不展示无限转圈；若需要等待，说明正在准备登录 | 不适用 | 标题、说明、request id、主操作首屏可见 | 复制 request id 有反馈 | 技术详情折叠，错误摘要不作为复制对象 |
| 角色组织/用户绑定 | 待选区 skeleton，不清空已选草稿 | 无组织/用户时说明筛选或权限原因 | 错误贴近待选区，已选草稿保留 | 保存后显示成功反馈并保留当前上下文 | 组织、用户、权限组部分加载失败时只禁用受影响区 |
| 飞书同步移动端 | 总览 skeleton，历史列表可延迟 | 无历史时给出同步入口或权限说明 | 同步状态错误与历史错误分区展示 | 最新同步、组织用户、历史、诊断均可达 | 诊断失败不阻断查看历史 |

### 9.4 用户旅程

| 时间层级 | 用户场景 | 设计要求 |
|---|---|---|
| 5 秒 | 管理员打开 390px 后台页面 | 立刻知道当前模块、主对象和下一步动作；不能先看到被压碎的表格列。 |
| 5 分钟 | 管理员完成一次应用/角色/审计排障操作 | 筛选、查看详情、复制 request id、切换 Tab 和保存配置路径明确，返回后不丢上下文。 |
| 5 年 | 平台长期维护身份权限中台 | 问题页、安全边界、审计入口和系统同步状态稳定可信，不因移动端或错误页显得临时拼凑。 |

### 9.5 Product Design 和 Pencil 决策

本版本值得使用 Product Design，但只能作为 Pencil 前的可选视觉目标分支。

使用条件：

- `gstack /plan-design-review` 完成后仍缺少明确视觉方向。
- 需要比较公开问题页、移动端资源卡片、系统管理移动端和复杂配置区的视觉结构。
- Product Design 输出必须保存或引用到 `design/`，并写明选中方向。

推荐 Product Design 探索 3 个方向：

1. **克制后台强化版**：保持当前 Riversoft 蓝绿，重点提升密度、状态层级和移动端卡片。
2. **高对比运维版**：更强调系统状态、风险、审计和同步排障，适合平台管理员。
3. **问题页可信版**：优先统一公开错误页和 request id 支持沟通，再回推后台 token。

Pencil 仍是正式设计治理产物。Product Design 不能替代 `.pen` 源文件、导出截图、设计说明和后续原型评审。

### 9.6 Pencil 原型最低范围

v1.0.2 Pencil 原型至少覆盖：

- 公开问题页：1440px、768px、390px。
- 应用管理移动端资源卡片。
- 操作审计移动端 Tab 和追踪空态。
- 管理员授权移动端资源卡片。
- 应用详情开发信息移动端分组。
- 角色详情组织与用户绑定移动端分步。
- 系统信息或飞书同步移动端代表页。

若时间不足，Pencil 第一稿必须先覆盖 P0：应用管理移动端、操作审计移动端、公开问题页。

### 9.7 无障碍和响应式门禁

- 所有移动端可点击目标不小于 44px。
- 图标按钮必须有 `aria-label` 和 `title`。
- Tabs 可键盘切换，横向滚动不隐藏当前选中态。
- 复制 request id 成功/失败必须有视觉反馈，并能被屏幕阅读器感知。
- 200% zoom 下长 `app_key`、role key、request id、URL 不得逐字竖排或遮挡操作。
- 正文对比度至少 4.5:1，UI 组件和大字至少 3:1。

### 9.8 设计评审结论

当前计划从 7/10 提升到 8/10，可以进入 Product Design/Pencil 原型阶段。剩余不到 10/10 的原因是还没有实际视觉目标和 `.pen` 原型；这些必须在进入 `plan-eng-review` 前补齐或明确豁免。
