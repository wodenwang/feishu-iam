# 2026-06-20 权限管理 UX 审视

## 会话目标

审视 Feishu IAM 权限管理模块中角色详情的 `组织与用户绑定` 和 `权限组绑定` 两个界面，梳理当前 UX 问题，并策划更清晰的管理配置流程和配套体验。

## 用户原始关键要求摘要

- 使用 `product-design` 插件审视权限管理模块 UX。
- 重点关注两个线上页面：
  - `tab=subjects`：组织人员绑定。
  - `tab=groups`：权限点 / 权限组绑定。
- 当前页面操作繁琐、混乱，且 UI 存在间距太宽等问题。
- 需要输出问题梳理和一套清晰的权限管理配置流程。
- 用户后续明确：移动端不重要，能兜底即可；主要重新策划 PC 端权限管理设计。
- 用户进一步明确：角色是独立资源，不再附属于应用；权限管理首屏必须是角色列表；角色列表支持查询、启停和批量操作；应用管理移除角色管理。
- 用户补充确认：本版本需要移除原 `应用管理` 中的角色管理部分，包括应用详情中的角色管理 Tab、入口和操作按钮。
- 用户最终纠偏：绑定组织 / 用户和绑定应用 / 权限组 / 权限应该拆开，避免单个界面太重。
- 用户进一步确认：`组织与用户` 必须使用同一个组织树选择器；权限点对比必须保留；角色配置页必须是独立页面，不能做成抽屉。
- 用户最终要求放弃 Pencil：Pencil 生成内容质量不可采纳，所有本轮 Pencil 设计稿和截图移除，后续只用 Product Design 策划设计。

## 使用或形成的关键约束

- 使用 Product Design 的 `get-context` 和 `audit` 流程进行 UX 审视。
- 遵守仓库 `DESIGN.md` 中关于权限管理的边界：
  - 权限管理围绕角色授权绑定展开。
  - 权限点主要由开发者 API 和应用管理相关能力维护，权限管理页以绑定和只读解释为主。
- 用户最新方向已覆盖并改变旧边界：角色成为独立资源，应用管理不再承载角色管理；后续实现前需要同步更新 `DESIGN.md` / `AGENTS.md` 中相冲突的边界。
- 不修改产品代码。
- 不记录明文密钥、token、cookie 或密码。

## 重要设计决策和原因

- 建议把角色授权配置从多个割裂 Tab 清洗为 `组织与用户`、`应用权限`、`变更记录` 三个清晰工作区。
- 建议将组织与用户绑定的区域语义稳定为：待选区、已选区、变更摘要区。
- 建议将权限组绑定从“可选权限组 + 结果预览 + 最终权限点”混排，调整为以权限组选择为主、最终权限点只读解释为辅。
- 重新确认 PC 端为主设计对象：角色配置页采用分工作区结构，每个工作区内部使用左侧配置区 + 右侧摘要保存栏；移动端只做不溢出、不遮挡、能完成基础链路的兜底。
- 后续又撤回“锁定当前工作台稿”的判断：新增角色-应用多对多和角色独立化要求后，原单应用角色详情原型只能作为旧草案。
- 新的权限管理 IA：`权限管理` 首屏为角色列表；角色列表支持按应用、名称、编号、状态查询，支持启用 / 停用和批量操作；行内入口为 `配置`。
- 新的角色配置页：`组织与用户` 工作区处理角色级主体绑定；`应用权限` 工作区处理角色-应用绑定、应用切换、权限组绑定、权限点只读预览和比对；`变更记录` 工作区承载授权审计。
- `组织与用户` 工作区必须使用统一组织树选择器，在同一棵树中勾选组织节点和用户节点。
- 权限点对比是 `应用权限` 核心能力，必须保留。
- 角色配置页必须是独立页面，并提供 `返回角色列表`，不能做成抽屉、Sheet 或弹窗。
- 本轮 Pencil 生成的 `.pen`、导出 PNG、prompt 和旧原型说明已弃用并移除，不能作为后续实现或评审依据。
- 应用管理重新定位：移除角色管理，聚焦密钥刷新 / 设置、回调设置、Codex 提示词生成，以及应用权限组 / 权限查看、查询和比对。
- `应用管理` 中原角色管理部分必须在本版本移除；角色新增、编辑、删除、启停和配置统一归入 `权限管理`。
