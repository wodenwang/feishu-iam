# v0.1 交付与部署实施计划

## 目标

把 v0.1 访问闭环变成可安装、可配置、可验证的最小交付包，覆盖本地开发、环境变量、部署、初始化和验收文档。

## 范围

- 一个部署实例绑定一个飞书租户和一个专用自建飞书应用。
- 配置模板只包含变量名和占位说明。
- 不把飞书凭证、tokens、同步快照、导出用户列表打入镜像或提交到仓库。
- v0.1 优先保证可复现交付，不追求复杂高可用架构。

- `README.md`

## 任务拆分

1. 本地开发路径
   - README 顶部保留最短启动路径。
   - `docs/deployment/local-dev.md` 说明前端、后端、Demo 的本地启动顺序。
   - 增加环境检查脚本，验证必要变量是否存在且没有使用真实示例值提交。

2. 配置模板
   - `.env.example` 覆盖飞书应用、数据库、会话密钥、IAM 应用密钥、回调地址。
   - 所有敏感变量用 `<replace-me>` 占位。
   - 文档说明哪些变量来自飞书开放平台，哪些由 feishu-iam 生成。

3. 容器化交付
   - 增加前端/后端构建 Dockerfile。
   - `docker-compose.yml` 支持本地验证数据库、后端、前端、第三方 Demo。
   - compose 文件不包含真实 secret，只引用 `.env`。

4. CI 与发布检查
   - CI 运行 build、unit test、E2E smoke。
   - `scripts/release-verify.sh` 串联环境检查、测试、构建和 secret grep。
   - 失败时输出清晰的修复提示。

5. 初始化与验收文档
   - `docs/deployment/production.md` 描述部署后第一位飞书用户绑定平台管理员。
   - `docs/deployment/release-checklist.md` 包含飞书应用权限、回调 URL、同步、审计、第三方 Demo 验收项。
   - 文档明确禁止绕过飞书登录创建本地管理员。

## 测试与验证命令

```bash
npm run build
npm test
npm run e2e
node scripts/check-env.ts --example .env.example
bash scripts/release-verify.sh
docker compose config
grep -R "tenant_access_token\\|app_secret\\|sec_\\|api_" . --exclude-dir=node_modules --exclude=.env.example || true
```

## 预期输出

- 新开发者可以按 README 和 local-dev 文档启动完整 v0.1。
- 部署人员可以按 production 文档配置飞书应用和环境变量。
- CI 和 release verify 能阻止测试失败、构建失败和明显 secret 泄漏。

## 完成标准

- 本地和容器路径至少各跑通一次。
- release checklist 每一项都有命令、页面或日志证据。
- `.env.example` 完整但不含真实值。
- CI 能在干净环境中完成 build、unit test 和 E2E smoke。

## 风险与边界

- v0.1 不承诺生产级高可用、备份恢复和多区域部署。
- 容器交付优先服务验收和小规模部署，后续再拆 Helm 或 Terraform。
- 飞书开放平台配置仍需人工在管理后台完成，文档必须把步骤写清楚。
