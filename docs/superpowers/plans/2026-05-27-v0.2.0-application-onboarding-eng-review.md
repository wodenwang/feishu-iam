## 当前实现基线

- `application_oauth_redirect_uris` 只有 `application_id`、`redirect_uri`、`created_at`，主键为 `(application_id, redirect_uri)`。
- 新建应用会自动插入 `http://127.0.0.1:4200/oauth/callback`。
- `/api/oauth/authorize` 只校验 `app_key + redirect_uri` 精确匹配，没有 URI 状态。
- `application_secrets` 和 `application_api_credentials` 只保存单个 hash，HTTP API 没有轮换接口。
- `application_admins` 已支持多行绑定，但应用详情只展示最早的 owner，没有增删接口和最后管理员保护。
- `/api/audit-logs` 已支持 `action`、`result`、`keyword`、`targetId`、`targetType` 筛选，应用管理员必须带应用 target。
- 前端 HTTP 模式下 `rotateApplicationSecret` 仍是 unsupported，应用详情页结构远低于 v0.2 Pencil 原型。

## 工程决策

### D1：redirect URI 使用现有复合主键，不引入独立 id

保留 `(application_id, redirect_uri)` 主键，避免对已有授权匹配和迁移带来不必要重写。管理接口通过 request body 传 `redirectUri` 定位记录。

新增迁移字段：

- `status text not null default 'active' check (status in ('active', 'disabled'))`
- `environment text not null default 'local' check (environment in ('production', 'staging', 'local'))`
- `note text not null default ''`
- `created_by_feishu_user_id text references feishu_users(feishu_user_id)`
- `updated_at timestamptz not null default now()`
- `disabled_at timestamptz`

`last_matched_at` 不在本切片实现，避免在 authorize 热路径增加写入；概览中的“最近命中”第一版可显示 `-`。

### D2：redirect URI 不做硬删除

只提供新增、停用、恢复。停用 URI 不再允许 OAuth authorize 命中，已有 bearer token/session 不受影响。所有配置变更写审计。

接口：

- `GET /api/applications/:id/redirect-uris`
- `POST /api/applications/:id/redirect-uris`
- `PATCH /api/applications/:id/redirect-uris/status`

校验：

- URI 必须可被 `URL` 解析。
- `production` 和 `staging` 默认要求 `https:`。
- `local` 允许 `http://127.0.0.1` 和 `http://localhost`。
- 重复 URI 返回 `409 OAUTH_REDIRECT_URI_EXISTS`。
- 无权限返回 `403 FORBIDDEN`。

OAuth authorize 必须追加 `r.status = 'active'` 条件。

### D3：secret 轮换立即替换当前 hash，新 secret 只返回一次

接口：

- `POST /api/applications/:id/secrets/rotate`

请求：

```json
{ "kind": "app_secret" }
```

或：

```json
{ "kind": "api_secret" }
```

响应只在本次返回明文：

```json
{
  "kind": "app_secret",
  "secret": "sec_...",
  "rotatedAt": "2026-05-27T..."
}
```

实现要求：

- `app_secret` 更新 `application_secrets.secret_hash`。
- `api_secret` 更新 `application_api_credentials.api_secret_hash`。
- 迁移为两个 secret 表补 `updated_at timestamptz not null default now()`。
- 审计动作使用 `secret.rotate`，metadata 只记录 `appKey` 和 `kind`，不得记录 secret 明文、hash 或 HMAC。
- 旧 secret 立即失效；OAuth token exchange 和 Application API HMAC 都使用新 hash 校验。

### D4：应用管理员 v0.2 只做增删，不做编辑

Pencil 原型已关闭 `编辑` 操作。v0.2.0 只实现：

- `GET /api/applications/:id/admins`
- `POST /api/applications/:id/admins`
- `DELETE /api/applications/:id/admins/:feishuUserId`

数据表暂不新增角色字段；`主要管理员` 由 `created_at asc` 第一位派生，其他人为 `应用管理员`。后续如需角色编辑，再作为独立切片。

删除保护：

- 删除前统计该应用管理员数量。
- 如果只剩 1 人，返回 `409 LAST_APPLICATION_ADMIN`。
- 不做软删除；审计日志保留追溯。

### D5：权限边界采用“平台管理员可维护，应用管理员可查看”

为避免应用管理员自我移除、扩大自身权限或轮换生产 secret，v0.2.0 的写操作只允许平台管理员：

- 新增 / 停用 / 恢复 redirect URI
- 轮换 appSecret / API Secret
- 新增 / 移除应用管理员

应用管理员仍可在自己 scope 内查看：

- 应用详情
- redirect URI 列表
- secret 状态和配置端点
- 应用管理员列表
- 本应用审计日志

后端所有写接口必须同时执行：

1. 已登录
2. `actor.isPlatformAdmin === true`
3. 应用存在

读接口使用 `requireAdminActor` + `requireApplicationScope`。

### D6：审计动作与筛选

新增或使用以下动作：

- `oauth.redirect_uri.create`
- `oauth.redirect_uri.disable`
- `oauth.redirect_uri.enable`
- `secret.rotate`
- `secret.copy`
- `application.admin.add`
- `application.admin.remove`

`GET /api/audit-logs` 现有筛选可复用。前端审计 Tab 默认传 `targetType=application` 和当前 `targetId`，并增加 action/result/keyword 筛选。

### D7：前端以 Pencil 为蓝图改造应用详情页

范围只覆盖 `/applications/:id`：

- 概览第一屏：配置完整性、最近风险、最近审计。
- 接入配置 Tab：OAuth 配置、Application API 配置、redirect URI Table、Drawer、停用/恢复确认、secret 轮换确认和一次性结果 Modal。
- 应用管理员 Tab：Table、Add Drawer、最后管理员保护错误。
- 审计记录 Tab：配置变更筛选。
- 768px 响应式：折叠左侧导航由现有 Layout 能力承接，详情页内部降低列数、表格横向滚动。

前端类型和 query 必须扩展，不在页面 JSX 中硬编码 mock 数据。

## 实施切片

第一 vertical slice 即 v0.2.0 的可运行闭环：后端 runtime 能力、应用详情页 UI、验收脚本和文档一起交付。原因是 redirect URI、secret、管理员和审计互相构成“应用接入生产化”的最小闭环，拆掉任一项都会让原型无法进入工程验收。

## 测试策略

后端测试：

- redirect URI CRUD、重复校验、停用后 authorize 拒绝、恢复后 authorize 成功。
- secret 轮换后旧 secret 失败、新 secret 成功，响应和审计不泄漏明文。
- API Secret 轮换后旧 HMAC 失败、新 HMAC 成功。
- 应用管理员新增、重复、移除、最后管理员保护。
- 平台管理员写权限、应用管理员只读且不能写其他应用。
- 审计动作存在且 metadata 不含 secret。

前端测试：

- `ApplicationDetailPage` 展示 v0.2 Tabs、redirect URI 表格、管理员表格和审计筛选。
- 新增 redirect URI Drawer 校验与提交。
- secret 轮换确认、二次输入、一次性结果 Modal。
- 最后管理员保护错误展示。
- HTTP adapter、DTO mapper、query invalidation。

验收脚本：

- 新增 `scripts/verify-v0.2-application-onboarding.sh`。
- 通过 HTTP API 创建应用、配置 URI、跑 OAuth authorize/token、轮换 secret、验证旧新 secret、维护管理员、查询审计。
- 输出不得打印 appSecret、apiSecret、bearer token、authorization code、cookie 或 HMAC signature。

手动/视觉验证：

- Playwright 检查 1440、1280、768 三个视口。
- 覆盖正常、空、错误、加载、无权限或禁用操作状态。
- 后续 `gstack /design-review`、`gstack /qa`、`gstack /review` 发现问题时，修复后重新运行对应技能直到无问题。

## 风险与处理

| 风险 | 处理 |
|---|---|
| 同时改后端、前端和脚本，范围偏大 | 使用一个闭环 vertical slice，但按后端契约、前端接入、脚本文档分阶段提交验证证据。 |
| secret 明文泄漏到日志、截图或 Git | 测试断言审计 metadata 不含明文；脚本只输出 mask；原型已替换非真实占位值。 |
| 应用管理员写权限过宽 | v0.2 写操作限定平台管理员；应用管理员只读本应用。 |
| redirect URI 复合主键给前端操作带来编码问题 | 写接口用 body 传 URI，不把 URI 放路径参数。 |
| 旧 API 客户端在 secret 轮换后立即失败 | 符合“轮换立即失效”语义，Modal 和文档必须明确提示。 |
