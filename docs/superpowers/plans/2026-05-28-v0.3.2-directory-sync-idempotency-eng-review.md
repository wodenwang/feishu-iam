# v0.3.2 组织同步幂等修复工程评审

日期：2026-05-28

## 结论

通过，按补丁版本执行。`v0.3.2` 只关闭 GitHub issue #27：飞书组织同步后出现重复“it部”，并同步修正版本元数据。该版本不重开目录治理，不新增 `/directory` 编辑，不实现 event-driven worker、告警、OIDC 或 SDK/CLI。

## 架构决策

- D1：`directory_departments.id` 继续作为目录部门主键。真实飞书 adapter 继续优先使用 `open_department_id`，不按名称合并部门。
- D2：一次 full sync 成功后，以本次飞书 snapshot 作为默认目录投影来源。本次 snapshot 外仍为 `active` 的历史部门标记为 `disabled`，不直接删除。
- D3：默认目录 API 只返回 active 部门。这样历史 seed/mock 部门不污染组织树，同时保留旧记录以避免破坏 `role_department_bindings` 等外键引用。
- D4：用户部门归属只接受本次 snapshot 内的部门 ID。用户指向 snapshot 外部门时写为 `null`，避免旧部门继续影响用户列表路径。
- D5：合法同名部门通过不同飞书部门 ID 和 parent_id 保留，不做名称级去重。

## 数据流

```text
Feishu departments/users snapshot
  -> RealFeishuDirectorySyncAdapter maps stable department ids
  -> applyDirectorySnapshot()
       -> upsert snapshot departments as active/current status
       -> mark active departments outside snapshot as disabled
       -> upsert users only when department id exists in snapshot
       -> mark missing users resigned
  -> Directory API returns only active department tree and active-derived paths
```

## 主要风险

| 风险 | 处理 |
|---|---|
| 历史角色部门绑定仍引用 disabled 部门 | 不删除部门记录；默认目录隐藏 disabled。后续如需清理，另开目录治理版本。 |
| 合法同名部门被误合并 | 不按名称去重；测试覆盖不同父级、不同 ID 的同名“it部”。 |
| 第二次 full sync 继续增加部门数量 | 测试覆盖连续两次同步 active 部门数量稳定。 |
| 历史 seed `dept_it` 与真实 `open_department_id` 并存 | full sync 后 seed 标记 disabled；目录 API 只返回真实 active 节点。 |
| VERSION 仍停留在 `0.3.0` | 本版本把 `VERSION`、`package.json`、README 和 CHANGELOG 一并更新。 |

## 测试策略

- `server/tests/sync.test.ts`
  - 回归 #27：已有 `dept_it` seed 后同步真实 `od_it`，默认部门 API 只返回真实 active 节点。
  - 幂等：同一 snapshot 连续同步两次，active 部门数量稳定，第二次不产生 created department。
  - 合法同名：不同父级、不同 ID 的两个“it部”都保留，并通过 path 区分。
- `server/tests` 既有同步、预检、事件重试、权限边界测试必须继续通过。
- 前端测试只在 active-only 查询影响断言时调整，不新增 UI 能力。
