## 2. 总体结论

`v0.13.1` 适合作为小版本推进。问题足够明确，且不需要扩大到应用管理、权限管理或飞书同步业务逻辑。

设计方向应从“已有子路由但视觉仍像扁平列表”修到“一级分组 + 可展开二级菜单 + 收缩态可理解 + 移动端层级完整”。这属于后台壳层基础体验，优先级高于继续扩展新功能。

## 4. 关键设计判断

### 4.1 不能继续做扁平导航

系统管理已经承载飞书同步、管理员授权、操作审计和系统信息。若继续把二级项直接平铺在一级菜单下面，即使有缩进，用户仍难以理解模块归属。`#12` 的核心不是“再加一点样式”，而是导航组件结构必须显式表达父子关系。

### 4.2 桌面展开态应常驻显示二级关系

桌面后台用户需要快速扫读和切换，系统管理二级菜单不应只藏进临时下拉。展开态建议默认显示当前父级的二级项，并允许用户折叠父级；当前二级页必须强制展开对应父级。

### 4.3 收缩态不能丢失分组语义

左侧菜单收缩后只剩图标。当前位于系统管理二级页时，系统管理图标必须 active；tooltip 或弹出内容应说明“系统管理”以及二级入口，避免用户无法返回其他系统管理页面。

### 4.4 移动端不是桌面收缩态

移动端 Sheet 有足够宽度展示完整层级，不应退化成只有一级图标。移动端应保留父级标题、二级缩进和当前项高亮。

## 5. 需要在原型中验证的问题

- 展开态：`系统管理` 父级是否像分组，而不是普通链接。
- 展开态：父级 active 和二级 active 是否同时可理解但不过度抢色。
- 收缩态：用户是否能发现系统管理下还有二级入口。
- 移动端：Sheet 中二级菜单是否清楚、触控目标是否足够。
- 键盘：父级展开按钮、二级菜单和收缩按钮能否顺序 Tab 到达。

## 6. 进入下一步的条件

可以进入 Pencil 原型，但原型必须覆盖至少四个画面：

1. 桌面展开态，当前位于 `系统管理 / 操作审计`。
2. 桌面展开态，`系统管理` 折叠。
3. 桌面收缩态，当前位于系统管理二级页。
4. 移动端 Sheet 展开态。

原型通过后再进入工程评审，工程评审必须明确：

- 是否新增 `Collapsible` UI primitive。
- `AppShell` 是否拆出 `SidebarNav` / `SidebarNavItem`。
- 如何保持当前二级页强制展开父级。
- 如何测试 `aria-expanded`、`aria-current`、tooltip、移动端 Sheet 和旧路由兼容。

## 7. 评审结论

本轮没有发现需要扩大产品范围的设计阻塞。`v0.13.1` 应继续作为 GitLab `#12` 的小版本修复推进。
