# Feishu IAM v1.0.7 权限管理信息架构收敛 PRD

## 版本定位

`v1.0.7` 是 `v1.0.5` 权限管理角色配置工作台和 `v1.0.6` 权限管理 UI/UX 小版本之后的产品收敛版本。

本版本目标不是扩大权限模型，而是把“角色配置”和“权限结果查询”分开：

- 角色工作台只承担角色授权配置。
- 权限矩阵承担按主体查询最终权限和来源解释。
- 应用权限页不再混入跨主体、跨应用的权限对比分析。

本版本一次性覆盖 S1 角色工作台减负和 S2 权限矩阵 MVP 两个 vertical slice；切片只用于实施顺序、验证边界和风险控制，不代表拆分成两个独立发布版本。

## 目标用户

- 平台管理员：配置角色、绑定组织与用户、绑定应用权限、查看最终权限。
- 接入系统管理员：核查某个用户或组织在某些应用下最终拥有哪些权限。
- 排障人员：回答“为什么这个用户有/没有某个菜单、按钮或接口权限”。

## 核心问题

当前权限管理把两类任务混在一起：

1. 配置任务：给某个角色绑定组织、用户、应用和权限组。
2. 查询任务：按用户或组织查看最终生效权限和来源。

结果是：

- 角色工作台信息过重。
- `应用权限` 页右侧承载过多解释性内容。
- `权限点对比` 出现在角色配置流程中，容易被误解为当前角色配置的一部分。
- 用户真正想按主体查询完整权限时，没有独立入口。

## 版本目标

### S1：角色工作台减负

角色工作台必须更像配置工作台，而不是权限分析报表。

范围：

1. 移除独立 `角色上下文` 区域。
2. 将角色 key、状态和必要识别信息合并进页面 header 或总览。
3. 将 `基础信息` Tab 的必要字段合并到 `总览`。
4. 移除 `基础信息` Tab。
5. 移除 `变更记录` Tab。
6. 在总览或保存摘要附近提供跳转 `操作审计` 的辅助入口，支持按角色 key 查询或复制角色 key。
7. `应用权限` 的 `添加应用` 改为弹框选择器。
8. 弹框选择器支持搜索、添加和移除当前角色工作区中的应用。
9. 移除应用时采用软解除或停用角色-应用绑定，并写入操作审计。
10. `应用权限` 只保留当前应用的权限组选择和最终权限点精简列表。
11. 从角色工作台移除 `权限点对比`。
12. 调查并修复平台管理员登录下 `创建角色` 按钮 disabled 的问题。

### S2：权限矩阵 MVP

新增 `权限矩阵` 二级功能，承担主体最终权限查询和来源解释。

范围：

1. 在 `权限管理` 下新增二级入口 `权限矩阵`。
2. 支持按用户查询最终权限。
3. 支持按组织查询最终权限，首版只计算直接绑定该组织的角色。
4. 查询结果按应用分组。
5. 每个应用下展示匹配角色、权限组、最终权限点。
6. 权限点必须展示 key、名称、来源角色、来源权限组和状态。
7. 长 app key、角色 key、权限组 key、权限点 key 必须可读，不撑破布局。
8. 提供加载、空状态、错误状态和权限不足状态。
9. 不在权限矩阵中编辑权限。

## 信息架构

### 权限管理导航

`权限管理` 从单入口升级为带二级能力的模块：

1. `角色授权`
   - 对应当前 `/admin/permissions` 角色列表。
   - 负责角色配置入口、角色创建、启停、批量操作。

2. `权限矩阵`
   - 新增二级入口。
   - 负责用户或组织最终权限查询。

具体路由可以在工程评审中确定，建议：

- `/admin/permissions` 保持兼容，继续指向角色授权列表。
- `/admin/permissions/matrix` 承载权限矩阵。

如果当前侧边栏暂不支持 `权限管理` 二级目录，应在工程评审中评估：

- 方案 A：将 `权限管理` 改为一级分组，二级为 `角色授权` / `权限矩阵`。
- 方案 B：保持左侧单入口，进入权限管理页后使用页内二级 Tabs。

产品推荐方案 A，因为它与 `系统管理` 的二级结构一致。

## 关键用户流程

### 流程 1：配置角色授权

1. 管理员进入 `权限管理 / 角色授权`。
2. 使用筛选条件定位角色。
3. 点击 `配置` 进入角色工作台。
4. 在 `总览` 查看角色基础信息和授权摘要。
5. 在 `组织与用户` 维护主体绑定。
6. 在 `应用权限` 选择当前应用，维护权限组绑定。
7. 右侧只核对当前应用的最终权限点精简列表和保存摘要。
8. 保存变更并写入操作审计。

### 流程 2：维护角色关联应用

1. 管理员进入角色工作台 `应用权限`。
2. 点击 `管理应用` 或 `添加应用`。
3. 弹框展示可绑定应用和已绑定应用。
4. 管理员搜索应用。
5. 勾选新增应用或移除已绑定应用。
6. 确认前展示变更摘要。
7. 确认后软解除或停用角色-应用绑定，并写入操作审计。
8. 当前应用被移除时，自动切换到剩余第一个可用应用；没有剩余应用时展示空状态。

### 流程 3：按用户查询权限矩阵

1. 管理员进入 `权限管理 / 权限矩阵`。
2. 选择查询类型：`用户`。
3. 通过用户选择器选择飞书用户。
4. 点击查询。
5. 页面展示该用户在各应用下的最终权限。
6. 每个应用展示匹配角色、权限组、权限点。
7. 管理员可复制权限点 key、角色 key 或 request id。

### 流程 4：按组织查询权限矩阵

1. 管理员进入 `权限管理 / 权限矩阵`。
2. 选择查询类型：`组织`。
3. 通过组织选择器选择飞书组织。
4. 点击查询。
5. 页面只展示直接绑定该组织的角色带来的权限。
6. 页面明确提示：首版不展开组织下全部用户。

## 功能需求

### R1：角色工作台总览收敛

- 页面保留 `角色配置工作台` 标题。
- 角色 key、状态、关联应用数量应在 header 或总览中直接可见。
- 总览展示角色基础信息、授权摘要和操作审计入口。
- 不再展示独立 `角色上下文` 卡片。
- 不再保留 `基础信息` Tab。
- 不再保留 `变更记录` Tab。

### R2：操作审计入口

- 在总览或保存摘要旁提供 `查看操作审计` 入口。
- 入口应跳转 `系统管理 / 操作审计`。
- 如果当前审计页支持 query 参数，应带上角色 key 或 resource id。
- 如果暂不支持 query 参数，应至少提供复制角色 key 能力和说明。

### R3：应用选择器弹框

- `添加应用` 改为弹框。
- 弹框标题明确，例如 `管理角色关联应用`。
- 弹框包含搜索框、已绑定应用列表、可添加应用列表。
- 每个应用展示名称、app key、状态和绑定状态。
- 支持添加未绑定应用。
- 支持移除已绑定应用。
- 所有变更提交前展示确认摘要。
- 当前角色没有任何已绑定应用时，`应用权限` 展示明确空状态和添加入口。

### R4：角色-应用绑定移除

- 移除应用不是硬删除。
- 首选实现为设置绑定状态为 disabled 或等价软解除。
- 操作必须写入审计日志。
- 审计至少记录操作者、目标角色、目标应用、before/after、结果、IP、User-Agent、request id。
- 移除当前应用后必须有稳定 fallback：
  - 优先切换到剩余第一个 active 绑定应用。
  - 如果没有 active 绑定应用，展示空状态。

### R5：应用权限工作区减负

- `应用权限` 只解决“当前角色在当前应用下绑定哪些权限组”。
- 左侧保留当前应用切换和权限组选择。
- 右侧保留保存摘要和最终权限点精简列表。
- 最终权限点列表只展示 key、名称、来源类型、来源权限组。
- 不再展示 `权限点对比`。
- 权限组内权限点仍可通过 `查看权限点` 展开，作为局部辅助信息。

### R6：权限矩阵入口

- 新增 `权限矩阵` 二级入口。
- 权限矩阵页面有明确标题、说明、查询区和结果区。
- 查询区支持主体类型切换：`用户` / `组织`。
- 用户查询使用现有飞书用户选择能力。
- 组织查询使用现有组织选择能力。
- 查询按钮有 loading 状态。
- 未选择主体时不能查询，并展示字段级提示。

### R7：权限矩阵结果

- 结果按应用分组。
- 每个应用展示：
  - 应用名称和 app key。
  - 匹配角色。
  - 权限组。
  - 最终权限点。
  - 计算时间。
- 权限点行展示：
  - 权限点名称。
  - 权限点 key。
  - 来源角色。
  - 来源权限组。
  - 状态。
- 用户查询展示该用户直接绑定角色和通过所属组织匹配的角色。
- 组织查询只展示直接绑定该组织的角色，并提示不展开组织下用户。

### R8：创建角色 disabled 修复

- 调查平台管理员登录下 `创建角色` disabled 的原因。
- 如果是前端 `canManageGlobalRoles` 或应用上下文判断错误，修复前端逻辑。
- 如果是后端权限策略导致，给出明确只读说明，不允许无提示 disabled。
- 验收时平台管理员应能理解为什么能或不能创建角色。

## 非目标

- 不新增 DDL，除非工程评审证明无法用现有绑定状态表达软解除。
- 不实现 ABAC。
- 不实现资源级权限。
- 不实现 deny 规则。
- 不实现数据范围权限。
- 不实现 SAML、完整 OIDC 或 refresh token。
- 不改变管理员 session 机制。
- 不把飞书组织下所有用户展开成组织权限矩阵首版默认行为。
- 不在权限矩阵中编辑角色、权限组或权限点。
- 不恢复应用管理中的角色管理。
- 不做全站 UI 主题重构。

## 后端需求

### 管理后台角色-应用绑定接口

当前前端已有绑定应用接口，但本版本需要支持软解除或停用绑定。

建议新增管理后台接口，最终路径由工程评审确定：

```text
POST /api/v1/admin/applications/{appKey}/iam-roles/{roleId}/application-binding/disable
```

或提供更通用的状态接口：

```text
PATCH /api/v1/admin/applications/{appKey}/iam-roles/{roleId}/application-binding
```

请求体示例：

```json
{
  "status": "disabled"
}
```

要求：

- 只能平台管理员执行。
- 不硬删除绑定。
- 写操作审计。
- 返回更新后的角色绑定摘要。

### 管理后台权限矩阵查询接口

建议新增只读接口，最终路径由工程评审确定：

```text
GET /api/v1/admin/permission-matrix?subjectType=user&subjectId={user_id}
GET /api/v1/admin/permission-matrix?subjectType=department&subjectId={department_id}
```

响应需要支持前端按应用渲染：

```json
{
  "subject": {
    "type": "user",
    "id": "f23b315a",
    "name": "王文哲"
  },
  "scope_note": "用户查询包含直接用户绑定和用户所属组织绑定。",
  "applications": [
    {
      "app_key": "base-portal",
      "name": "基础门户",
      "matched_roles": [
        {
          "key": "base-portal.admin",
          "name": "管理员",
          "match_type": "direct_user"
        }
      ],
      "permission_groups": [
        {
          "key": "base-portal.demo.users",
          "name": "Base Portal 示例接入用户"
        }
      ],
      "permission_points": [
        {
          "key": "base-portal.demo.embedded",
          "name": "SSO Demo",
          "source_roles": ["base-portal.admin"],
          "source_groups": ["base-portal.demo.users"]
        }
      ]
    }
  ],
  "computed_at": "2026-06-21T00:00:00.000Z"
}
```

要求：

- 不返回 secret、token、cookie、authorization、授权码、token hash、state hash 或 raw payload。
- 用户查询可复用现有有效权限计算逻辑，但需要补齐来源解释。
- 组织查询首版只匹配直接绑定该组织的角色。
- 空结果返回正常空数组，不返回框架默认错误。
- 权限不足返回统一后台错误结构和 request id。

## 前端需求

### 页面和路由

- 保持 `/admin/permissions` 兼容角色列表。
- 新增权限矩阵入口，推荐路径 `/admin/permissions/matrix`。
- 如改造左侧导航为二级结构，当前 `/admin/permissions` 应映射到 `角色授权`。
- 旧角色详情深链必须继续可用：
  - `/admin/permissions/roles/{roleId}`
  - `tab=subjects`
  - `tab=permissions`

### 组件复用

优先复用：

- `PageHeader`
- `DataTable`
- `FilterBar`
- `FormDialog`
- `ConfirmDialog`
- `StatusBadge`
- `OrgUserSelector`
- `OrgBrowser`
- `ResponsiveTabsList`

不得为单页重新造一套基础按钮、表格、弹窗或状态组件。

### 响应式

- 桌面端优先信息密度和扫描效率。
- 390px 下不得出现页面级横向溢出。
- 权限矩阵在窄屏下可以按应用卡片展示，不强行压缩成宽表格。
- 长 key 必须使用换行、截断、tooltip 或复制能力。

## 权限与安全

- 所有写操作必须经过管理员 session 校验。
- 角色-应用绑定移除必须写审计。
- 权限矩阵是只读查询，不写审计日志也可以；如工程评审认为查询敏感，可写安全事件或查询审计，但不得记录敏感凭证。
- 前端不保存权限矩阵查询原始大 payload 到本地存储。
- 错误提示只展示中文安全说明和 request id，不暴露堆栈。

## 验收标准

### S1 验收

- 角色工作台不再展示独立 `角色上下文` 区域。
- `总览` 能看到角色名称、角色 key、状态、关联应用数量、组织数量、用户数量、权限组数量。
- 角色工作台 tab 只保留必要配置入口，至少为 `总览`、`组织与用户`、`应用权限`。
- `基础信息` 和 `变更记录` tab 被移除。
- 变更追溯入口能引导到 `操作审计`。
- `添加应用` 打开弹框选择器。
- 弹框支持搜索、添加应用、移除应用、确认和取消。
- 移除应用后写入审计，并有稳定 fallback。
- `应用权限` 不再展示 `权限点对比`。
- 最终权限点列表精简且长 key 不撑破布局。
- 平台管理员登录下 `创建角色` 不再无解释 disabled。

- 可以按用户查询权限矩阵。
- 可以按组织查询权限矩阵。
- 组织查询页面明确说明首版只统计直接绑定该组织的角色。
- 查询结果按应用展示匹配角色、权限组和最终权限点。
- 空结果、加载、错误、权限不足状态可读。
- 1440px、768px、390px 下无明显布局错位、横向撑破或按钮不可点。
- Browser 自检覆盖用户查询、组织查询、空状态和窄屏布局。
