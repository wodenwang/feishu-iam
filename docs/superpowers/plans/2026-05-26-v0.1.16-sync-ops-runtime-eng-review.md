# v0.1.16 同步运营 Runtime 工程评审

## 结论

通过，但按 vertical slice 执行。`v0.1.16` 合并承接原先拆散的同步健康、权限预检、scheduled full sync 和同步策略预留；实现时第一切片只交付同步健康与权限预检 Runtime，后续 scheduled full sync 必须在第一切片验证后再推进。

本版本仍不重开 `/directory` 编辑，不实现飞书事件订阅 Runtime，不实现真正 incremental sync，不扩大应用管理员权限，不触碰 OAuth/OIDC。

## 范围锁定

In scope:

- `GET /api/sync/status`：平台管理员查看最近同步、运行中状态、目录投影数量、健康状态和原因。
- `POST /api/sync/preflight`：平台管理员执行飞书通讯录权限预检，检查 token、部门读取、用户读取。
- preflight 不写 `feishu_users`、`directory_departments`、`directory_users`，不创建 full sync run。
- preflight 成功和失败写 `sync.preflight` 审计，并保留 requestId。
- scheduled full sync 作为本版本后续切片：默认关闭，使用 system actor，不伪造飞书用户。
- 为后续 incremental/event sync 预留 adapter/service 边界说明，但不实现 Runtime。
- `/sync` 页面后续接入健康摘要、预检入口和 scheduled 配置状态。
- README、CHANGELOG、VERSION 和验证截图随最终版本收口更新。

Out of scope:

- `/directory` 编辑、删除、导入、导出或目录治理 UI。
- 飞书 webhook、事件订阅验签、事件重放、真正增量同步。
- 告警系统、飞书机器人通知、Slack 通知或多环境监控面板。
- 应用管理员触发或查看同步运营能力。
- OAuth/OIDC、redirect URI、secret rotation、第三方 Demo 改动。
- 持久化 tenant token、App Secret、原始通讯录 payload、导出用户列表或同步快照。

## 架构决策

### 1. 第一切片先做 status + preflight

合并版 `v0.1.16` 的完整目标比单个小 patch 大。为了不把 scheduler、system actor migration、前端状态和文档同时混在一个不可验证 diff 中，第一切片只交付后端 status + preflight，并通过后端测试证明安全边界。

### 2. Preflight 是诊断动作，不是同步 run

`POST /api/sync/preflight` 不创建 `sync_runs`，也不写目录投影。它只验证飞书应用凭证和通讯录读取权限是否可用。审计动作用 `sync.preflight`，目标类型用 `sync_preflight`。

### 3. Adapter 暴露轻量预检能力

`DirectorySyncAdapter` 增加 `preflight()`。真实飞书 adapter 复用 tenant token 获取、部门读取和用户读取的最小请求；测试 fake adapter 可注入成功或失败结果。业务层只消费标准化 `DirectorySyncPreflightResult`。

### 4. 健康状态从现有数据计算

`GET /api/sync/status` 不新增表。它从 `sync_runs`、`directory_users`、`directory_departments` 计算：

- `latestRun`
- `latestSuccessfulRun`
- `latestFailedRun`
- `isRunning`
- `directoryUserCount`
- `directoryDepartmentCount`
- `healthStatus`
- `healthReasons`

第一切片不需要 scheduled metadata。后续 scheduler 切片再增加 scheduled 配置投影。

### 5. Scheduled sync 使用 system actor，不伪造飞书用户

后续切片需要 migration 调整 `sync_runs.operator_feishu_user_id` 或引入 `operator_type`。不能插入 `ou_system_sync` 之类假飞书用户，因为项目规则要求飞书是唯一身份源。

## 第一 Vertical Slice

1. 平台管理员调用 `GET /api/sync/status`。
2. 后端返回当前同步健康状态、最近同步和目录投影数量。
3. 平台管理员调用 `POST /api/sync/preflight`。
4. 后端通过 adapter 完成 token、部门、用户三个阶段预检。
5. 成功或失败都写 `sync.preflight` 审计。
6. 验证 preflight 不修改目录投影，不创建 `sync_runs`。
7. 普通用户和应用管理员无法访问 status/preflight。

## 风险与控制

| 风险 | 控制 |
|---|---|
| preflight 意外写入目录投影 | 测试记录调用前后目录和 sync_runs 数量，要求不变。 |
| 飞书错误泄露 secret/token/raw payload | adapter/service 只返回 stage、code、message、requestId；不返回 headers 或 token。 |
| 应用管理员越权查看同步运营状态 | status 和 preflight 都复用 platform admin 校验。 |
| 合并版范围继续膨胀 | 第一切片只做 backend status + preflight；scheduler 和 frontend 是后续切片。 |
| 健康状态误导 | 第一版用保守规则：无成功为 unknown，最近失败晚于最近成功为 failed，运行中为 warning。 |

## 测试策略

必须覆盖：

- 平台管理员可读取 sync status。
- 无同步记录时 health 为 `unknown`。
- 最近失败晚于最近成功时 health 为 `failed`。
- running sync 存在时 health 为 `warning`。
- status 返回目录用户和部门数量。
- 平台管理员可执行 preflight success，并写 success audit。
- preflight failure 写 failure audit，返回脱敏错误。
- preflight 不创建 `sync_runs`，不写目录投影。
- 普通用户和应用管理员不能访问 status/preflight。
