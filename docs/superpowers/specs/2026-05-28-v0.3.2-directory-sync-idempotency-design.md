# v0.3.2 组织同步幂等修复设计

日期：2026-05-28

## 背景

`v0.3.2` 是 issue-backed bugfix release，用于关闭 GitHub issue #27：飞书组织同步后出现重复“it部”。当前 `v0.3.0` 已进入同步可靠性主线，`v0.3.1` 只处理 Admin Shell 补丁；本版本继续保持小补丁边界，修复目录投影幂等问题，并统一版本元数据。

项目硬边界保持不变：

- 飞书是组织结构和用户的唯一来源。
- 不新增本地组织来源。
- 不新增 username/password、本地 root 或独立账号体系。
- 不提交飞书凭证、tokens、导出用户列表或同步快照。

## 目标

- 同一个飞书部门在系统中只有一个有效目录投影。
- 连续执行 full sync 不产生重复部门节点。
- 初始化 seed/mock 部门不再污染真实飞书同步后的默认目录树。
- 同名但不同父级的合法飞书部门仍保留，不按名称合并。
- `package.json`、`VERSION`、`README.md`、`CHANGELOG.md` 的版本信息统一到 `0.3.2` / `v0.3.2`。

## 非目标

- 不做 `/directory` 编辑、导入、导出。
- 不做目录治理工作流。
- 不做完整 event-driven sync worker、事件积压调度器或告警平台。
- 不做 OIDC/JWKS/PKCE/refresh token。
- 不做 SDK、CLI、Helm、Terraform。
- 不改变飞书认证和用户身份绑定边界。

## 方案

采用“full sync 快照替换默认目录投影”的语义，但不破坏历史引用。

### 部门唯一性

`directory_departments.id` 继续作为目录部门主键。真实飞书 adapter 继续优先使用 `open_department_id`，必要时兼容 `department_id`。同步 upsert 只按部门 ID 冲突更新，不按名称合并。

这意味着：

- 同一个飞书部门重复同步时更新同一条记录。
- 飞书侧允许不同父级下存在同名部门时，系统保留多条不同 ID 的记录。
- 名称只用于展示和路径，不作为幂等键。

### Full Sync 快照应用

一次 full sync 成功获取 snapshot 后：

1. 本次 snapshot 内的部门全部 upsert 到 `directory_departments`，状态为 snapshot 状态，默认 `active`。
2. 本次 snapshot 外、当前仍为 `active` 的历史部门标记为 `disabled`，不直接删除。
3. 用户同步仍按本次 snapshot 的部门 ID 校验；用户指向 snapshot 外部门时，部门归属写为 `null`。
4. 用户离职标记逻辑保持现有语义：本次 snapshot 外的既有目录用户标记为 `resigned`。

选择标记 `disabled` 而不是删除，是为了避免破坏历史 `role_department_bindings`、审计记录或其他外键引用。默认目录查询只展示 active 部门，因此历史 seed 的 `dept_it` 在真实飞书同步后不会继续出现在组织树中。

### Directory API

不新增 API。

`GET /api/directory/departments`：

- 只返回 `status = 'active'` 的部门。
- 递归树只从 active root 开始，并只连接 active child。
- `total` 只统计 active 部门。
- 不按名称去重；合法同名部门通过 `path` 区分。

`GET /api/directory/users`：

- 部门筛选只基于 active 部门树。
- 用户列表中的 `department_name` / `department_path` 只从 active department tree 解析。
- 用户历史上指向 disabled 部门时，默认展示为 `-`。
- 不改变用户列表响应字段和分页契约。

### 前端影响

`/directory` 不新增页面能力。前端继续消费现有部门树和用户列表接口。必要时只更新测试断言，确保路径展示可以区分合法同名部门。

## 实施范围

- `server/src/modules/sync/syncService.ts`
  - 在 full sync 应用 snapshot 后，将 snapshot 外 active 部门标记为 `disabled`。
  - 保持部门 upsert 按 ID 冲突更新。
  - 保持用户部门归属只接受本次 snapshot 内部门。
- `server/src/modules/directory/directoryRoutes.ts`
  - 部门递归查询和用户路径解析只使用 active 部门。
  - 部门 total 只统计 active 部门。

## 验收标准

- 使用包含“it部”的飞书组织结构同步后，默认目录树不再显示同一飞书部门的重复节点。
- 连续执行至少两次 full sync 后，active 部门数量稳定。
- 同名但不同父级的合法部门不被错误合并。
- 历史 seed/mock 部门在真实同步后被标记为 disabled，不污染默认目录树。
- 部门同步仍优先基于飞书部门唯一标识和父子关系处理 upsert。
- 版本元数据统一到 `v0.3.2`。

## 风险与约束

- 历史 role department 绑定可能仍引用 disabled 部门。v0.3.2 不删除这些记录，也不新增治理 UI；后续若需要清理，另开目录治理版本。
- 如果真实飞书返回的 `open_department_id` 与历史 `department_id` 表示同一个部门但值不同，系统会把旧 ID 标记为 disabled，把新 ID 作为有效投影。这符合“飞书当前快照为准”的语义。
- 如果飞书侧确实存在合法同名部门，系统不合并；用户需通过路径区分。
