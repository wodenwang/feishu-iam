# Feishu IAM v0.15.2 组织用户选择器重构规格

日期：2026-05-28
状态：已确认，可进入实施计划

## 1. 背景

`v0.15.0` 已完成组织树与组织用户选择器，`v0.15.1` 已完成权限可解释性。当前 GitLab 唯一打开的 issue 是 `#26 组织和用户选择组件重构`，反馈集中在角色组织与用户绑定选择器：

- 组织架构树不应从某个局部部门开始，而应从真实最顶层组织开始。
- 左侧更换路径时，右侧已选列表必须保留。
- 用户和组织应放在同一个待选列表，通过头像或稳定视觉标识区分。

本版本定位为 `v0.15.2`，属于 `v0.15.x` 组织选择体验的后续修复小版本，不升到 `v0.16.0`。

## 2. 版本目标

`v0.15.2` 只收口 GitLab issue `#26`：把角色详情页中的组织与用户绑定选择器重构为更接近飞书官方组织成员选择体验的单一候选列表。

## 3. 范围

### 纳入范围

- 角色详情页 `组织与用户绑定` Tab 中的 `OrgUserSelector`。
- 复用的 `OrgBrowser` 组件需要支持：
  - 明确请求顶层组织。
  - 按路径下钻组织。
  - 组织和用户同列表展示。
  - 搜索组织或用户。
  - 选择状态不因路径切换丢失。
- 飞书同步页面继续复用 `OrgBrowser`，但保持只读浏览和排障主旅程不回退。
- 应用级飞书部门候选接口和后台飞书部门查询接口支持明确查询顶层组织。

### 不纳入范围

- 不新增数据库 DDL。
- 不修改 `iam_role_subjects` 保存结构。
- 不实现资源级权限、ABAC、deny 规则或飞书角色同步。
- 不修改 SSO/OIDC 协议面。
- 不恢复权限管理中的角色元数据新增、编辑或启停。
- 不重构整个后台信息架构。
- 不把飞书通讯录实时树作为默认目标，继续读取本地飞书镜像。

## 4. 交互要求

- 顶层进入时只展示真实顶层组织和直属根级用户；不展示全量部门混排导致用户误以为从某个中间部门开始。
- 组织和用户在同一个待选列表中展示：
  - 组织使用建筑或组织图标。
  - 用户使用用户头像式图标。
  - 每行显示名称、ID 和主体类型。
- 组织行提供下钻入口和选择入口。
- 用户行提供选择入口；禁用或删除用户不可选。
- 已选组织与用户列表在右侧或移动端 `已选` 页签中保留，不因待选列表路径切换、返回上级、搜索或加载更多而清空。
- 保存 payload 仍区分 `feishu_department` 与 `feishu_user`，组织主体不等于自动展开全部用户。
- 桌面宽度下保持左待选、右已选和摘要的工作区结构；窄屏下继续使用待选、已选、摘要分步面板。

## 5. 数据和 API 契约

- 前端使用特殊查询值 `parent_department_id=__root__` 表示顶层组织。
- 后端把 `__root__` 解析为 `parentDepartmentId: null`。
- 不传 `parent_department_id` 仍表示不按父组织过滤，用于关键词搜索和兼容旧调用。
- 应用级候选接口继续只返回当前管理员有权管理应用范围内可见的本地飞书镜像数据。

## 6. 验收标准

- GitLab issue `#26` 的三条要求均满足。
- 角色详情页可从顶层组织进入，能下钻、返回、搜索、选择组织、选择用户并保存。
- 路径切换后已选组织与用户不丢失。
- 用户和组织在同一待选列表中可区分。
- 飞书同步页面组织用户浏览仍可打开并查看组织/用户。
- 桌面和 390px 窄屏下无明显遮挡、溢出、按钮变形或异常空白。
- 相关前端测试、类型检查、lint 和 Browser 自检通过。
