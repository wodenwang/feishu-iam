## 架构决策

### 1. 验收脚本采用 API black-box，不写数据库

脚本只依赖运行中的 IAM runtime，通过 HTTP API 完成验收。这样可以复用真实 migration、auth、request context、audit、OAuth 和 Application API 签名路径，避免把验收变成测试专用后门。

例外：脚本允许读取 `RUNTIME_API_BASE_URL` 和环境变量，不直接访问 `DATABASE_URL`。

### 2. 自动验收固定使用 mock Feishu

真实飞书登录和通讯录预检需要外部应用配置、回调地址、权限和可能的 IP 白名单，不适合作为本地自动验收门槛。`v0.1.17` 的自动脚本目标是证明 IAM runtime 链路本身完整；真实飞书配置作为 README 手动验收说明。

### 3. OAuth 必须进入脚本主路径

旧 `verify-v0.1.2-access-loop.sh` 只用 IAM session cookie 查询权限。当前版本必须通过：

```text
/api/oauth/authorize -> /api/oauth/token -> Bearer token -> /api/application/me/permissions
```

这证明第三方 Demo 的真实接入模型，而不是只证明 Admin Console session 内部权限。

### 4. 敏感输出边界

脚本可以打印：

- runtime base URL
- appKey
- 有权限/无权限 permissionCodes
- 审计动作摘要

脚本不得打印：

- `appSecret`
- `apiSecret`
- bearer token
- cookie
- HMAC signature
- Feishu App Secret / tenant token

### 5. 同步纳入健康检查，不扩大同步平台

脚本检查 `/api/sync/status` 和 mock mode `POST /api/sync/preflight`。这证明同步运营入口仍可用，并产生可审计预检行为。脚本不触发真实 full sync，不要求真实飞书通讯录权限，不实现增量或事件同步。

## 第一 Vertical Slice

1. 新增 `scripts/verify-v0.1-access-loop.sh`。
2. 复用旧脚本的 HMAC helper、mock login helper 和 request helper。
3. 增加 OAuth helper：
   - 登录飞书 mock 用户取得 IAM session cookie。
   - 调用 `/api/oauth/authorize`，从 302 Location 提取 code。
   - 调用 `/api/oauth/token` 取得 bearer token。
   - 不打印 code/token/secret。
4. 增加 sync status/preflight 检查。
5. 增加审计动作断言。
6. 更新 README 和 Demo docs 的验收入口。
7. 写入 QA/review 占位报告，后续验证命令补齐结果。

## 风险与控制

| 风险 | 控制 |
|---|---|
| 脚本泄露一次性 secret 或 token | 只在进程内变量使用，输出前统一只打印 appKey 和摘要。 |
| OAuth redirect 自动跟随导致拿不到 code | fetch 对 authorize 使用 `redirect: 'manual'`，显式读取 Location。 |
| 审计分页漏掉动作 | 使用本次脚本唯一 suffix 过滤 keyword，并针对关键动作做至少一次列表断言。 |
| 自动脚本误要求真实飞书凭证 | README 明确脚本要求 `FEISHU_AUTH_MODE=mock`；真实飞书作为手动配置验收。 |
| 本版膨胀到 v0.2 | 工程计划明确 out of scope，不触碰 OAuth 配置 UI、secret rotation、OIDC 和 sync 增量。 |
