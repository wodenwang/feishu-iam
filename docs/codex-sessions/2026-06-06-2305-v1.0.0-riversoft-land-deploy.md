# v1.0.0 Riversoft 正式版远端落地与部署

## 会话目标

使用 `gstack /land-and-deploy` 对 Feishu IAM v1.0.0 Riversoft 正式版第一个 vertical slice 做最终远端落地和部署。

## 用户关键要求

- push 当前分支、创建合并入口、完成最终落地。
- 打 `v1.0.0` tag、构建并上传 Docker 镜像、创建 release。
- 使用远端上下文 `bpmt@120.24.236.92`、`/home/bpmt/feishu-iam`、`https://feishu-iam.riversoft.com.cn/` 完成部署和线上健康检查。
- 线上验证覆盖 `/ready`、`/version`、后台入口、Riversoft Logo、应用详情不暴露敏感凭证、王文哲平台管理员初始化结果。
- 不提交或输出真实 secret、token、cookie、飞书 `app_secret`、数据库密码或生产导出数据。
- 用户随后明确授权：远端服务器上的同名旧服务可以直接停掉、删掉。

## 关键过程

1. 本地分支 `codex/v1.0.0-riversoft-ui-init` 已 push 到 GitHub。
2. 创建 GitHub PR `#1`：`https://github.com/wodenwang/feishu-iam/pull/1`。
3. PR 已 squash merge 到 `main`，merge commit 为 `a8aca2e156656af581eb1bb1dfb3a05989a526bf`。
4. 标准多架构 buildx 构建受本机 Docker BuildKit 到 Debian apt 源网络影响失败；改用临时 HTTP Debian 镜像源后完成构建，但向内网 HTTP Registry `dockerhub.it.tangtring.com:80` push 时仍按 HTTPS 探测，返回 `EOF`。
5. 远端服务器无法直接从 GitHub 私有仓库拉取代码，也无法解析 `dockerhub.it.tangtring.com`；未能作为 registry 发布构建机使用。
6. 改走项目历史使用过的 amd64 离线 tar 路径：本机构建 `linux/amd64` 镜像并上传到远端 `docker load`。
7. 远端旧 `v0.4.0` 部署目录不是当前仓库形态，旧数据库表字段与当前迁移链不兼容。根据用户授权，停止并移除旧 compose 服务，旧目录整体备份为 `/home/bpmt/feishu-iam-v0.4.0-backup-20260606-225509`。
8. 在 `/home/bpmt/feishu-iam` 创建新的 v1.0.0 标准部署目录，使用新的数据库数据目录。
9. 从旧库只读解析“王文哲”的飞书用户 ID，作为 `INITIAL_PLATFORM_ADMIN_FEISHU_USER_ID` 写入新 `.env`；不输出任何敏感值。
10. 远端 `postgres:16-alpine` 直接拉取 Docker Hub 失败，部署 compose 的 db 镜像临时改为服务器已有的华为云 Postgres 镜像。
11. 按受控顺序手动应用迁移：先执行 `V0_1_0`、`V0_2_0`，插入“王文哲”这一条平台管理员初始化前置 Feishu 用户，再继续执行到 `V1_0_0`。
12. 启动 v1.0.0 Web 服务，公网 `/ready` 和 `/version` 验证通过。
