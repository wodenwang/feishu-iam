# 权限管理 UX 审视与清洗方案

## 审视范围与设计重心

- 组织与用户绑定：`/admin/permissions/base-portal/roles/a5dac1d8-7e54-4571-870f-4b191908b7cd?tab=subjects`
- 权限组绑定：`/admin/permissions/base-portal/roles/a5dac1d8-7e54-4571-870f-4b191908b7cd?tab=groups`
- 目标角色：`base-portal.admin`
- 审视时间：2026-06-20
- 设计重心：PC 端后台配置体验。移动端只做兜底可用，不作为主要设计驱动。

## 最新产品定向

本 UX 方案已经被新的权限管理方向更新，旧截图仍作为问题证据，但旧“单应用角色详情页”不再是目标模型。

用户确认的新流程：

1. 点击 `权限管理` 后首先进入角色列表。
2. 角色列表支持按应用、名称、编号等条件查询。
3. 角色列表支持启用 / 停用和批量操作。
4. 点击角色行 `配置` 进入角色配置页。
5. 角色配置页拆成 `组织与用户`、`应用权限`、`变更记录` 三个工作区。
6. `组织与用户` 只处理组织和用户绑定，回答“谁拥有这个角色”。
7. `应用权限` 处理角色绑定应用、按应用绑定权限组，并查看权限点，回答“这个角色在各应用中能做什么”。
8. `组织与用户` 必须使用同一个组织树选择器，组织节点和用户节点在同一棵树中勾选，不拆成两个独立选择器。
9. 权限点对比是 `应用权限` 的核心能力，必须保留在主工作区。
10. 角色配置页必须是独立页面，不能做成抽屉、Sheet 或弹窗；通过 `返回角色列表` 回到列表页。
11. 应用管理不再提供 `角色管理`；角色是独立存在，不再附属于应用。
12. 应用管理聚焦密钥刷新 / 设置、回调设置、Codex 提示词生成，以及应用权限组 / 权限查看、查询、比对。

因此，新版 PC 端方案不应只是在现有角色详情页中合并 `tab=subjects` 和 `tab=groups`，也不应把两类高频绑定强行塞进一个沉重界面；目标主流程应是“角色列表 -> 角色配置页 -> 分工作区完成主体绑定和应用权限绑定”。

## 当前主要问题

### 1. 任务模型不够清晰

当前角色详情同时暴露 `总览 / 组织与用户绑定 / 权限组绑定 / 基础信息 / 操作说明`。用户真正要做的任务是“给这个角色配置谁拥有它，以及它带来什么权限”，但页面将任务拆成多个说明性 Tab，用户需要自行理解两个绑定之间的关系。PC 端应该把这个页面重设为清晰的角色配置页，只保留高频任务工作区：`组织与用户`、`应用权限`、`变更记录`。

建议把权限管理的配置主线改成：

1. 先确认角色和应用上下文。
2. 在 `组织与用户` 工作区通过同一个组织树选择器配置授权对象。
3. 在 `应用权限` 工作区选择应用、绑定权限组并查看最终权限点。
4. 在当前工作区检查变更摘要。
5. 分别保存当前工作区变更，统一写入角色授权审计。

### 2. 组织与用户绑定的左右区含义重复

桌面视图中，左侧是待选，右侧包含已选组织、已选用户和变更摘要。右侧信息本应承担“结果确认”，但现在既解释概念，又列已选，又显示保存摘要，造成认知负担。

建议改为固定三栏语义，但选择入口必须统一：

- 左：组织/用户混合树，只负责查找和选择；组织节点和用户节点都能在同一棵树中勾选。
- 中：已选主体清单，只负责呈现已选结果和移除。
- 右或底部 sticky：变更摘要和保存，只负责风险确认。

说明文案要减少，只保留关键规则：选择组织不会自动展开成该组织下全部用户；用户可单独选择。

### 3. 权限组绑定主次倒置

权限组页左侧“可选权限组”上方存在明显空白，右侧“绑定结果预览”和“最终权限点”堆在一起。实际用户最关心的是：

- 我勾了哪些权限组。
- 这些权限组最终给出哪些权限点。
- 保存会新增或移除什么。

当前页面把最终权限点放在右侧附属区域，且搜索权限组和搜索最终权限点并列出现，容易让用户误以为权限点也可直接编辑。

建议将权限组页改成上下或主从结构：

- 上方：权限组筛选和可选权限组表格。
- 中部：选中权限组的权限点预览，可按权限组展开。
- 右侧或底部：最终权限点汇总和变更摘要。
- 明确标注“权限点只读，由权限组带出；本页不维护权限点 CRUD”。

### 4. 信息密度不是单纯“太少”，而是密度分布不均

桌面页有大块空白、重复边框和多层卡片；移动端则把大块卡片、外层 Tab、内层 Tab 和说明文案纵向堆叠，首屏大量空间被标题、返回按钮、角色摘要和说明占用，真正配置区位置靠后。

建议：

- 角色顶部摘要压缩成一行 sticky context bar：角色名、key、状态、返回。
- 删除重复的第二个“角色详情”大卡片，或合并到页面标题区。
- Tab 区改为更短的任务导航：`授权对象`、`授权内容`、`变更确认`。
- 卡片内 padding 收敛到 16px；复杂列表使用表格/紧凑 list，不再卡片套卡片。

### 5. 保存动作割裂

当前主体绑定和权限组绑定各自保存，变更摘要也分散。新版不强行把两类绑定合成一个大表单，但需要让它们共享角色上下文、顶部状态和审计语义。

建议保留后端现有两个 PUT 接口，前端体验改成分工作区草稿：

- 用户在 `组织与用户` 工作区修改主体，在 `应用权限` 工作区修改角色-应用权限组。
- 顶部展示跨工作区的未保存状态，例如 `组织与用户有变更`、`应用权限有变更`。
- 当前工作区右侧展示当前 diff：主体 diff 或权限组 / 最终权限点 diff。
- 点击 `保存组织与用户` 或 `保存应用权限` 后只提交当前工作区；失败时显示可恢复错误，不清空草稿。

## PC 端重新策划方案

### 1. 入口：权限管理角色列表

- 页面首先展示角色列表，而不是先进入某个应用下的角色详情。
- 筛选支持应用、名称、编号、状态、授权状态。
- 列表展示基础信息、启用 / 停用开关、关联应用数、已绑定组织数、已绑定用户数、按应用汇总的权限组数。
- 支持勾选角色后批量启用 / 停用。
- 行内主操作命名为 `配置`，不要只叫 `详情`。

### 2. 角色配置页

进入角色后，默认看到“角色配置页”，而不是默认停在总览。PC 端首屏要让管理员清楚知道当前角色、角色状态、关联应用数量，以及当前正在处理哪类授权任务。

顶部 context bar，高度控制在一行到两行内：

- `管理员 / base-portal.admin / 启用`
- 角色编号
- 关联应用数
- 返回角色列表
- 未保存变更状态

主工作区建议采用一级 Tabs：

- `组织与用户`：承载组织 / 用户绑定、待选浏览、已选清单和主体变更保存。
- `应用权限`：承载应用切换、角色-应用绑定、权限组绑定、权限点预览、查询和比对。
- `变更记录`：承载角色授权变更审计、操作者、时间、对象和结果。

每个工作区内部保持 `左侧主配置区 + 右侧检查栏`：

1. `组织与用户`
   - 组织树和用户搜索放在同一工作区。
   - 待选列表采用紧凑行：图标、名称、类型、路径 / ID、操作。
   - 已选主体不要另开远距离卡片，放在同一段右侧或下方的已选清单中。
   - 操作只保留 `选择`、`下钻`、`移除`，减少解释性按钮。

2. `应用权限`
   - 先切换当前配置应用。
- 权限组列表使用紧凑表格或可勾选列表。
- 每行展示权限组名称、key、状态、权限点数量、是否已选。
- 权限点只作为展开预览或右侧结果，不允许被编辑。
- 增加 `仅看已选`、`仅看有变更`、权限组 / 权限点比对作为 PC 端高效筛选。
- 权限点比对必须保留为可见主区域，不能降级为次级链接或隐藏在弹窗中。

右侧检查栏随工作区变化：

1. `当前授权对象`
   - 组织数量、用户数量。
   - 展示前 3-5 个已选主体，超出显示 `查看更多`。
   - 只在 `组织与用户` 工作区作为主要保存摘要展示。

2. `最终权限`
   - 权限组数量、最终权限点数量。
   - 展示关键权限点或最近变更权限点。
   - 只在 `应用权限` 工作区作为主要保存摘要展示。

3. `变更与保存`
   - 在 `组织与用户` 中展示新增 / 移除组织和用户，主按钮为 `保存组织与用户`。
   - 在 `应用权限` 中展示新增 / 移除权限组、涉及应用和最终权限点变化，主按钮为 `保存应用权限`。

### 3. 推荐 PC 操作流

1. `授权对象`
   - 左侧为统一组织树 + 搜索，组织节点和用户节点混合展示并可勾选。
   - 中间已选主体，同表展示组织和用户，可按类型筛选但不拆成两套选择器。
   - 主体卡片展示名称、类型、路径、ID、失效状态。
   - 操作只保留 `选择`、`下钻`、`移除`。

2. `授权内容`
   - 权限组列表使用紧凑表格或可勾选列表。
   - 每个权限组展示名称、key、状态、权限点数量。
   - 展开后查看权限点，权限点只读。
   - 右侧或下方展示最终权限点汇总。

3. `变更记录`
   - 展示历史组织、用户、应用、权限组变更。
   - 展示操作者、操作时间、结果和 request id。
   - 支持按应用、操作类型和时间过滤。

角色基础信息保留在顶部 context bar 或只读详情抽屉，不作为主流程 Tab。操作说明合并为帮助抽屉或折叠说明。

## PC 端关键布局规则

- 页面顶部不要重复出现两个“角色详情”大块。角色名、key、状态、返回入口应合并进同一个 compact header。
- 角色配置必须使用独立页面路由，不允许做成抽屉、Sheet、弹窗或从列表右侧滑出的详情面板。
- 页面必须提供稳定的 `返回角色列表`，返回后保留列表筛选、分页和排序。
- 主配置区不再用大面积空白卡片承载说明。说明文字控制在每段标题下 1 行以内。
- 卡片只用于清单项或侧栏摘要，不要出现卡片套卡片。
- 权限组列表首屏必须直接露出至少 4-6 条有效行；当前只有 1 条数据时，也不能产生大块无意义空白。
- 当前工作区保存按钮应始终位于右侧检查栏底部或页面底部 sticky 区，且只在当前工作区存在变更时启用。
- 权限点的视觉语义必须是“结果预览”，不能和权限组选择控件同级竞争。

## 移动端兜底策略

移动端不是主要考虑范畴，只需要保证关键链路可用、不溢出、不遮挡。390px 下可以沿用简化分步导航：

1. `对象`
2. `权限`
3. `确认`

每一步只显示当前任务，底部固定显示：

- 已选组织数。
- 已选用户数。
- 已选权限组数。
- 下一步 / 保存按钮。

移动端不要求达到 PC 端同等效率，但不应出现横向溢出、按钮不可点、保存区被遮挡或两层 Tab 互相干扰。

## 优先级建议

P0：

- 权限管理入口改为角色列表，支持应用、名称、编号查询。
- 角色列表增加启用 / 停用和批量操作。
- 角色配置页改为独立页面，并提供 `返回角色列表`。
- `组织与用户` 改为统一组织树选择器，同一选择器中勾选组织和用户。
- 压缩角色顶部重复摘要和大块空白。
- 权限组页去掉无意义大空白，改紧凑列表。
- 权限点对比保留为 `应用权限` 的核心区域。
- 将保存摘要固定为 PC 端右侧检查栏或统一底部 sticky 区域。
- 重设 PC 端角色配置页：`组织与用户` 与 `应用权限` 分工作区承载，右侧摘要和保存跟随当前工作区。
- 应用管理移除角色管理入口。

P1：

- 角色列表操作从 `查看详情` 改为 `配置`。
- 顶部统一展示跨工作区未保存状态，当前工作区独立保存。
- 当前工作区展示主体、权限组和最终权限点 diff。
- PC 端增加 `仅看已选`、`仅看有变更`、权限组展开预览、权限组 / 权限点比对等效率控件。
- 应用管理保留应用权限组 / 权限查看、查询、比对。

P2：

- 为权限组和最终权限点增加更强的搜索反馈、数量统计和空状态。
- 增加“仅看已选 / 仅看有变更”过滤。
- 后续如果接口支持，再补最终权限点 diff。
- 移动端兜底整理，保证可用但不作为主线体验优化。

## 验收标准

- 桌面 1440px 下，当前工作区首屏能同时看见主要配置区和保存/变更摘要。
- 权限管理首屏是角色列表，而不是应用内角色管理。
- 角色列表支持查询、启停、批量操作和 `配置` 入口。
- PC 端角色授权页不再重复展示两个“角色详情”大块，配置区没有无意义大空白。
- PC 端权限组列表首屏能直接看到可选项、已选状态、权限点数量和保存摘要。
- `应用权限` 工作区能切换应用，并只绑定当前应用下的权限组和权限。
- 应用管理不再出现角色管理能力。
- 390px 下可完成 `对象 -> 权限 -> 确认` 基础链路即可，不作为主要体验评价标准。
- 未保存变更在两个工作区之间不丢失，保存前能看清当前工作区 diff，顶部能提示其他工作区仍有未保存变更。
- 权限点在 UI 中始终表现为只读结果，不让用户误以为可以在权限管理中直接维护权限点。
- Console 无错误，关键接口无非预期失败请求。
