# v0.1.12 Third-party OAuth Demo Runtime 工程评审

## 结论

通过，但带一个明确限制：`v0.1.12` 只交付第三方 Demo 的最小 OAuth Authorization Code runtime，不交付完整 OAuth/OIDC 产品体验。

本切片允许的完成状态是：

1. 浏览器已有 IAM 飞书登录态。
2. 第三方 Demo 发起 `/api/oauth/authorize`。
3. IAM 校验应用、redirect URI 和当前 Feishu user。
4. IAM 颁发短期一次性 authorization code。
5. Demo 使用 `appSecret` 换取短期 bearer token。
6. Demo 使用 bearer token + Application API HMAC 查询当前用户权限。
7. Demo 页面按 permission code 显示允许/拒绝。

本切片不要求未登录用户从 Demo 自动跳转飞书登录后恢复原始 authorize 请求。当前未登录时可以先进入 IAM 登录页，用户登录后重新从 Demo 发起。自动恢复 authorize request 需要额外保存 pending OAuth request，适合后续 UX/security 小版本。

## 范围锁定

In scope:

- `GET /api/oauth/authorize`
- `POST /api/oauth/token`
- `application_oauth_redirect_uris`
- `application_oauth_authorization_codes`
- `application_oauth_sessions`
- Application API permission query 支持第三方 bearer token。
- Bearer token 必须绑定 application，跨应用调用必须拒绝。
- `examples/thirdparty-demo` 默认 OAuth mode。
- `server/tests/oauth.test.ts` 覆盖核心安全边界。
- Demo 文档、`.env.example`、README、CHANGELOG、版本元数据。

Out of scope:

- OIDC discovery。
- JWKS。
- PKCE。
- refresh token。
- 复杂 consent 页面。
- 自动恢复未登录用户的 pending authorize request。
- Sync runtime / tenant token / 通讯录全量同步。
- `/directory` 能力扩展。
- 本地 username/password 登录。

## 架构决策

### 1. `appSecret` 和 `apiSecret` 分工

- `appSecret` 用于 OAuth token exchange。
- `apiSecret` 用于 Application API HMAC。

这保持 v0.1.11 接入材料的语义清晰：一个 secret 证明应用 OAuth client 身份，另一个 secret 证明 Application API 调用身份。

### 2. authorization code 存储

authorization code 只存 SHA-256 hash，不存明文。

字段必须包含：

- application id
- redirect URI
- Feishu user id
- state
- expires_at
- consumed_at

code 有效期 5 分钟，只能消费一次。`token` endpoint 必须在事务里 `for update` 锁定 code row，防止并发重复消费。

### 3. bearer session 存储

third-party bearer token 只存 SHA-256 hash，不存明文。

字段必须包含：

- application id
- Feishu user id
- expires_at

当前 bearer token 有效期 8 小时，不提供 refresh token。过期后重新走 OAuth authorize。

### 4. redirect URI

第一版使用 `application_oauth_redirect_uris` 表，并为现有和新增应用插入本地 Demo callback：

```text
http://127.0.0.1:4200/oauth/callback
```

这是 demo-first 的保守选择。后续若要开放真实第三方应用配置，必须增加 Admin Console 配置入口、URI 校验、变更审计和危险操作确认。

### 5. permission query actor

`requestContext` 优先解析 IAM session cookie；没有 cookie 时解析 `Authorization: Bearer ...`。

Application API permission query 继续要求 HMAC。若 bearer token 的 `application_id` 与 HMAC credential 的 application 不一致，必须返回 403。

这避免 A 应用拿到的 token 被 B 应用的 API secret 拿去查询权限。

## 数据流

```text
Demo /login
  -> IAM GET /api/oauth/authorize?client_id&redirect_uri&state
  -> IAM validates current Feishu actor and redirect URI
  -> IAM stores hashed authorization code
  -> Demo /oauth/callback?code&state
  -> Demo POST /api/oauth/token with appSecret
  -> IAM consumes code and stores hashed bearer token
  -> Demo GET /api/application/me/permissions with bearer token + HMAC
  -> IAM returns permission point codes for the Feishu user in that application
```

## 风险与控制

| 风险 | 控制 |
|---|---|
| authorization code 重放 | code 只存 hash，`token` 事务内 `for update`，成功后写 `consumed_at`。 |
| redirect URI 劫持 | authorize 和 token exchange 都校验完全一致的 redirect URI。 |
| client secret 泄露 | secret 只从创建响应一次性获得，代码和文档只写环境变量名。 |
| 跨应用 token 混用 | bearer actor 带 `oauthApplicationId`，permission query 对比 Application API credential 的 app id。 |
| 未登录用户体验不完整 | 本版记录为 out of scope；后续可加 pending authorize request cookie/table。 |
| refresh token 复杂度过早进入 | 本版只发短期 bearer token，不实现 refresh token。 |
| 本地 Demo callback 被当成生产能力 | README 和 Demo 文档标记为最小 Demo；真实第三方配置后续单独建模。 |

## 测试策略

必须覆盖：

- authorize 成功颁发 code。
- token exchange 成功返回 bearer token。
- code 二次消费失败。
- redirect URI 不匹配失败。
- client secret 错误失败。
- bearer token + Application API HMAC 可以查询所属应用权限。
- bearer token 不能跨应用使用。
- 有权限用户返回 `demo.customer:view`。
- 无权限用户返回空权限列表。

## 性能与运维

- authorization code 和 bearer session 表需要 expiry index，便于后续清理任务。
- 当前没有后台清理 job，短期本地 demo 可接受；生产化前需要增加 expired token/code cleanup。
- `requestContext` 增加 bearer token 查询，只在没有 IAM session cookie 时触发，避免影响 Admin Console 常规路径。

## NOT in scope

- 未登录用户自动恢复 authorize request：需要 pending request 存储和登录后 redirect 策略，本版先不做。
- 应用 redirect URI 管理 UI：需要表单、校验、审计和危险操作确认，本版先只支持本地 Demo callback。
- OIDC discovery/JWKS/refresh token/PKCE：属于更完整 OAuth/OIDC 产品化版本。
- Sync runtime：与第三方 OAuth Demo 的最小闭环无关，继续延后。
