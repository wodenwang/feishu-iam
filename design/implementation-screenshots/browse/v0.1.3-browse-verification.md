# v0.1.3 HTTP mode `/browse` 验证记录

验证日期：2026-05-24

验证目标：仅覆盖 v0.1.3 第一个 vertical slice：HTTP mode session、initialization、applications list/create、one-time secret modal、audit visible。未展开 Roles/Directory、Dashboard/Sync、Application Detail/Onboarding 后续切片。

- 前端：Vite HTTP mode，`http://127.0.0.1:5173`
- 后端：Fastify local runtime，`http://127.0.0.1:4100`
- 数据库：本地 PostgreSQL `feishu_iam_manual`
- 浏览器工具：gstack `/browse`
- 视口：`1440x960`、`1280x900`、`768x1024`

## 状态覆盖

| 状态 | 验证方式 | 结果 |
|---|---|---|
| Loading | 访问 `/applications` 时捕获 session loading | 通过，展示“加载飞书会话” |
| Empty | 重置数据库后进入 `/applications` | 通过，列表空态可见并可创建 |
| API error | 临时 drop `applications` 表触发 500，随后恢复数据库 | 通过，展示“加载应用列表失败”、重试按钮和 requestId |
| 401 | 设置无效 `iam_session` 后访问 `/applications` | 通过，展示“会话已过期”和重新登录入口 |
| 403 | 使用非平台管理员 mock 飞书用户访问 `/applications` | 通过，展示“403 无权限访问当前页面” |
| 409/requestId | 重复创建同名应用 | 通过，表单展示“应用名称已存在”；后端返回 409 |
| requestId 详情 | 打开 audit detail drawer | 通过，详情页展示 requestId |

负向测试期间 console/network 中出现的 `409` 和 `500` 为刻意触发：`409` 用于重复应用名校验，`500` 用于 API error/requestId 展示。数据库已在 API error 验证后恢复。
