## Step 2：权限管理角色列表

截图：02-permissions-role-list.png

观察：

- 当前生产数据展示 3 个 IAM 角色。
- 角色列表支持应用筛选、角色查询、角色编号、状态、授权状态和排序。
- 行内操作为图标按钮，当前能进入角色配置、编辑和停用。
- 当前登录人显示为平台管理员，但 `创建角色` 按钮处于 disabled 状态。
- 页面没有明显横向溢出，console error 数量为 0。

PRD 含义：

- 需要确认 `创建角色` disabled 是否为预期。如果平台管理员应能创建角色，本次版本应把它纳入修复或单独登记。
- 列表页本身不是本轮最大问题，下一版重点应进入角色工作台信息架构和应用权限交互。

## Step 3：角色工作台总览

截图：03-role-workbench-overview.png

观察：

- 工作台顶部存在页面级 `角色配置工作台`，下方还有独立 `角色上下文` 卡片。
- tab 仍包含 `总览 / 组织与用户 / 应用权限 / 基础信息 / 变更记录`。
- `基础信息概览` 已在总览中展示角色名称、角色 key、状态、创建时间和更新时间。
- `基础信息` tab 与总览信息边界重复。

PRD 含义：

- 支持 GitHub issue `#6`：移除独立 `角色上下文` 区域，将角色 key、状态和必要上下文并入页面 header 或总览。
- `基础信息` tab 可以移除，必要字段保留在总览。

## Step 5：应用权限

截图：

- 05-role-workbench-application-permissions.png
- 06-application-permissions-switched-app.png

观察：

- 当前应用已使用纵向 tab，跨应用角色能在 `飞书IAM的SSO DEMO` 与 `基础门户` 之间切换。
- `添加应用` 仍是 disabled select + button，不是弹框选择器；没有移除应用入口。
- 切换应用会更新 URL `appKey` 并重载当前应用上下文，存在明显上下文切换感。
- 右侧同时承载 `绑定结果预览`、`最终权限点` 和 `权限点对比`。
- `权限点对比` 仍在角色配置页内，和用户提出的“应拆到权限矩阵”一致。

PRD 含义：

- 支持 GitHub issue `#7`：`添加应用` 应改为弹框选择器，并把添加/移除应用做成同一个应用集合管理流程。
- 支持 GitHub issue `#8`：`应用权限` tab 聚焦角色下当前应用的权限组配置和最终权限点核对；不要继续承载跨主体、跨应用的权限对比。
- 支持 GitHub issue `#9`：`权限点对比` 更适合迁到 `权限矩阵` 二级功能。

## Step 6：基础信息与变更记录冗余 tab

截图：

- 07-role-workbench-base-tab.png
- 08-role-workbench-audit-tab.png

观察：

- `变更记录` tab 当前只是说明文字，不是真正的角色变更日志列表。
- 说明文字要求用户到系统审计日志按角色 key 或 request id 查询。
- 这类内容更适合在总览或保存摘要旁提供“查看操作审计”的辅助入口。

PRD 含义：

- 支持 GitHub issue `#6`：移除 `变更记录` tab。
- 如果需要保留追溯能力，提供跳转到 `系统管理 / 操作审计` 的链接，并自动带上角色 key 查询参数或给出复制角色 key。

## 初步版本范围建议

建议版本：`v1.0.7 权限管理信息架构收敛`。

建议拆成两个 vertical slice：

1. `S1 角色工作台减负`：覆盖 GitHub issue `#6/#7/#8`。
2. `S2 权限矩阵 MVP`：覆盖 GitHub issue `#9`。

关键边界：

- `#6/#8` 主要是前端信息架构和工作台交互。
- `#7` 如果包含“移除应用”，需要新增角色-应用绑定解除或停用接口，并写操作审计；不需要新增 DDL。
- `#9` 需要新增管理后台只读查询 API。现有 `PermissionCalculationService.calculate(appKey, userId)` 只支持按用户 + 单应用计算，组织查询和跨应用汇总需要额外设计。

用户后续确认选择 B：S1 和 S2 在同一个 `v1.0.7` 版本内一次性完成，不拆成独立的 `v1.0.8`。
