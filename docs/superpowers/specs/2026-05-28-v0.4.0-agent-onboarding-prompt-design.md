# v0.4.0 Agent 接入初始化提示词设计

日期：2026-05-28

## 背景

`v0.4.0` 从 `v0.3.x` 同步可靠性主线切换到第三方应用接入体验主线。当前系统已经具备应用创建、OAuth redirect URI 管理、`appSecret` / `apiSecret` 轮换、Application API 权限注册、权限查询、应用诊断和基础 Agent Prompt 复制能力。

本版本要解决的问题是：第三方内部项目也通常由 Codex 或 Claude Code 开发，接入方需要一段可以直接交给 Agent 的初始化提示词，让 Agent 在第三方项目中创建或维护 `AGENTS.md`、`CLAUDE.md`，并按清晰 SOP 完成 OAuth、Application API、权限注册、权限查询和验收。

项目硬边界保持不变：

- 飞书是组织结构和用户的唯一来源。
- 不新增 username/password、本地 root 或独立账号体系。
- 不改变现有 OAuth 和 Application API runtime 协议。
- 不提交飞书应用凭证、tokens、导出的用户列表或同步快照。

## 目标

- 应用详情页可以生成 Agent-first 第三方项目初始化提示词。
- 提示词足够完整，第三方项目里的 Codex / Claude Code 可以据此初始化 `AGENTS.md` 和 `CLAUDE.md`。
- 提示词包含接口说明、HMAC 签名规则、OAuth SOP、Application API SOP、权限命名规范、`.env` 配置清单、验收清单和排障指引。
- 创建应用或轮换 secret 的一次性结果可以生成包含当次明文 secret 的完整提示词。
- 普通应用详情页提示词只包含 secret 占位符，不提供历史 secret 再次查看能力。
- 审计记录可以区分普通占位符提示词复制和一次性明文提示词复制，但不写入明文 secret。

## 非目标

- 不新增 SDK、CLI、npm package、Helm、Terraform 或下载包。
- 不新增可解密 secret 存储。
- 不提供历史 secret 再次查看。
- 不做 OIDC Discovery、JWKS、PKCE 或 refresh token。
- 不改 OAuth authorize/token 或 Application API 路径。
- 不扩展同步、目录治理、事件 worker 或告警平台。
- 不新建文档站。

## 用户与场景

主要用户是第三方内部业务系统的开发 Agent 和负责接入的应用管理员。

典型流程：

1. 平台管理员在 `feishu-iam` 创建应用或轮换 secret。
2. 管理员复制一次性完整初始化提示词。
3. 管理员把提示词交给第三方项目中的 Codex / Claude Code。
4. 第三方 Agent 创建或更新 `AGENTS.md`、`CLAUDE.md`，并写入 IAM 接入规则、环境变量、接口说明和验收命令。
5. 第三方 Agent 实现 OAuth 登录、Application API 权限注册和权限查询。
6. 管理员在 `feishu-iam` 完成角色授权，并用诊断和审计排查问题。

如果不是创建或轮换后的当次上下文，管理员仍可在应用详情页复制占位符版提示词，用于文档初始化或交给接入方自行填充 secret。

## 提示词结构

提示词必须是可直接粘贴给 Agent 的任务指令，而不是普通文档摘要。

内容分为 8 段：

1. 角色与目标：说明当前第三方项目需要接入 `feishu-iam`，要求 Agent 创建或维护 `AGENTS.md`、`CLAUDE.md`。
2. 硬性约束：只使用飞书 / `feishu-iam` SSO，不新增本地账号体系，不绕过 IAM 权限判断，不提交 secret。
3. 当前应用配置：`IAM_BASE_URL`、`IAM_APP_KEY`、OAuth authorize/token endpoint、Application API endpoint、active redirect URI。
4. Secret 配置：一次性完整提示词填入 `IAM_APP_SECRET` / `IAM_API_SECRET` 明文；普通提示词使用 `<FEISHU_IAM_APP_SECRET>` 和 `<FEISHU_IAM_API_SECRET>` 占位符。
5. OAuth SOP：登录跳转、callback、state 校验、code exchange、token 保存、失败恢复和重新登录入口。
6. Application API SOP：权限组注册、权限点注册、当前用户权限查询、HMAC header 和 canonical string。
7. 第三方治理文档要求：`AGENTS.md` / `CLAUDE.md` 必须包含 IAM 接入边界、环境变量、接口说明、权限命名、验证命令和排障步骤。
8. 验收清单：`.env` 完整、redirect URI 启用、权限注册成功、角色授权完成、OAuth 登录成功、权限查询返回预期、无权限路径可恢复。

## Secret 明文边界

本版本允许一次性提示词包含明文 secret，原因是当前目标偏内部项目接入效率。

边界如下：

- 创建应用返回的一次性 `appSecret` / `apiSecret` 可用于生成完整提示词。
- 轮换 `appSecret` 时，一次性提示词只填入新的 `IAM_APP_SECRET`，`IAM_API_SECRET` 仍用占位符。
- 轮换 `apiSecret` 时，一次性提示词只填入新的 `IAM_API_SECRET`，`IAM_APP_SECRET` 仍用占位符。
- 普通应用详情页提示词永远只显示占位符。
- 不保存可解密 secret，不新增 secret 明文字段。
- 审计 metadata 只记录 prompt kind 和是否 one-time，不记录 secret 明文。
- README、测试输出和截图不得包含真实 secret。

## 页面与交互

复用现有应用详情页 `接入配置` Tab，不新增独立页面。

- `第三方应用接入提示词` 区块展示普通占位符版 prompt，并提供复制按钮。
- 轮换 secret 成功后的结果区域新增“复制一次性 Agent 初始化提示词”按钮。
- 一次性提示词复制前展示明确 warning：内容包含明文 secret，只能写入第三方项目运行时环境或 secret manager，不得提交 Git。
- 复制失败时保留手动复制兜底提示。

创建应用结果当前在应用列表页已有一次性 secret 展示，本版本可在该结果 Modal 中补充同样的“一次性 Agent 初始化提示词”能力；如果实现成本过高，至少保证轮换路径完整，创建路径在后续小版本补齐。但最终 `v0.4.0` 的验收目标优先覆盖创建和轮换两种来源。

## 接口说明

提示词内必须使用当前真实 runtime 路径：

- `GET /api/oauth/authorize`
- `POST /api/oauth/token`
- `PUT /api/application/permission-groups`
- `PUT /api/application/permission-points`
- `GET /api/application/me/permissions`

Application API HMAC header：

- `x-fiam-app-key`
- `x-fiam-timestamp`
- `x-fiam-nonce`
- `x-fiam-body-sha256`
- `x-fiam-signature`

canonical string：

```text
METHOD
PATH
NORMALIZED_QUERY
TIMESTAMP
NONCE
BODY_SHA256_HEX
```

签名使用 `IAM_API_SECRET` 对 canonical string 做 `HMAC-SHA256 hex`。body hash 使用原始请求体的 `SHA-256 hex`。timestamp 容忍窗口为 5 分钟，nonce 不可重放。

## 验收标准

- 应用详情普通提示词包含 `AGENTS.md`、`CLAUDE.md`、接口说明、OAuth SOP、Application API SOP、HMAC 规则和 secret 占位符。
- 普通提示词不包含 `appSecretPreview`、`apiSecretPreview` 或任何明文 secret。
- 轮换 `appSecret` 后的一次性提示词包含新的 `IAM_APP_SECRET` 明文，`IAM_API_SECRET` 为占位符。
- 轮换 `apiSecret` 后的一次性提示词包含新的 `IAM_API_SECRET` 明文，`IAM_APP_SECRET` 为占位符。
- 创建应用的一次性结果可生成同时包含 `appSecret` 和 `apiSecret` 的完整提示词。
- `secret.copy` 审计可区分普通 prompt 和 one-time prompt，但 metadata 不包含 secret 明文。
- README 和第三方接入文档说明 Agent 初始化 SOP。
- 版本元数据统一到 `0.4.0` / `v0.4.0`。

## 设计评审结论

该版本复用现有应用详情页和创建/轮换结果，不新建信息架构或页面布局，因此不需要新的 Pencil 原型。需要重点控制的是文案结构、一次性 secret 暴露边界、复制反馈和审计不泄密。
